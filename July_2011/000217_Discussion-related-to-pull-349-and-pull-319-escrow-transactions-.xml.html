    Summary:

    The pull request 349 on the GitHub repository for Bitcoin implements a multiple signature escrowed transaction. The feature has been tested on a private network and appears to work well. However, it needs more testing and discussion before being added to the client. The implementation requires a new transaction type, which won't be deployable until an updated isStandard is widely used in the network. Therefore, it is recommended to separate the IsStandard part of the patch and get it out in version 0.4. The patch also exposes an issue with multisig validation due to redundancy in script length coding opcodes that allow coding a script multiple ways. One candidate fix is to exclude the length opcodes from the comparison, but it carries a risk of creating differences in how nodes validate transactions leading to lasting forks. Another fix would be to define that OP_PUSHDATA* cannot be used to push smaller lengths that could be more efficiently encoded with direct length opcodes. This would have the benefit of being consistent with current behavior and carry no severe split risk. Another way of fixing this would be to change how the templating code works to make sure it codes the template the same way the original was coded.
