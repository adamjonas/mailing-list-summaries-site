    Summary:

    In an email conversation dated August 3, 2011, it was discussed that with Eligius mining IsStandard transactions and other pools open to the idea, it could be possible to quickly get 30% or more of mining power to upgrade. However, it was also noted that propagation is a crucial factor in addition to mining power. The focus on maturity/bugfixes was predicted to delay the full patch, but the IsStandard part was seen as uncontroversial and could go in quickly. The suggestion was made to split pull request 319 into two separate ones: one for the IsStandard change and the other for the full escrow feature set. Pull request 349 was also mentioned as an implementation of multiple signature escrowed transactions that needs further testing and discussion before being added to the client. It was noted that the patch exposes an issue with multisig validation due to redundancy in the script length coding opcodes, and one candidate fix for this is to exclude the length opcodes from the comparison. However, this fix carries a risk of creating differences in how nodes validate transactions leading to lasting forks. Another way of fixing it would be to define that OP_PUSHDATA* cannot be used to push the smaller lengths which could be more efficiently encoded with the direct length opcodes, but this may not solve the problem entirely. Lastly, changing how the templating code works to ensure it codes the template the same way the original was coded was suggested as another potential solution.
