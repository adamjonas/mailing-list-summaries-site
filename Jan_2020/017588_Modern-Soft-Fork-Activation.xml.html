Modern Soft Fork Activation

    Summary:

    The discussion revolves around the proposal of implementing a soft-fork activation path through BIP 9. The goal is to have an "out" and observe the response of the ecosystem once the change is visible, before actually activating it. The concern raised is that there might not be enough time for observation and discussion. The proposed solution is to release the first point with the soft-fork activation parameters without updating getblocktemplate to signal the version bit by default. This way, miners can only enable signalling if they need to rush things, and there is more time for observation. The downside of this approach is that it might lead to forking the chain with no gain. The discussion also touches on how version bits are set on the mining software rather than the node software, which means that there is no strong link between signalling and having updated your software to enforce new rules. People have suggested moving signalling into the coinbase or similar to make that link tighter.
