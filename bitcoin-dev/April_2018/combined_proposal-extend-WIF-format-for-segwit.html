Summary:

The Bitcoin Improvement Proposal (BIP) XXX proposes extending the private key format to specify the type of public key it corresponds to. This is due to the presence of several types of public keys which can be associated with a given private key such as P2PKH, P2SH-P2WPKH, P2WPKH, and more. By extending the suffix, the proposal aims to specify what kind of public key was associated with the given private key. The proposal includes a table with values, types, compression, and clarification.Currently, the Wallet Import Format (WIF) appends a 0x01 byte after the raw private key when used in conjunction with a compressed public key. This allows wallets to associate a single Bitcoin address to a WIF key. However, it would be useful to extend the semantics of that byte to signal for segwit scripts, as these scripts result in different addresses. There are proposals to add new bytes for segwit script embedded in P2SH and native segwit script. It is also possible to use a bech32 format for the private key if it is going to be used with a bech32 address.Pieter Wuille, a Bitcoin Core developer, has expressed concerns over the ambiguity of the Wallet Import Format (WIF) when used with Segregated Witness (SegWit) scripts. He believes that the current WIF format is not acceptable to use with SegWit scripts and could cause "technological debt" by forcing wallets to try all possible scripts. Wuille argues that defining his own format makes sense for the xpub/xprv format because Electrum users need to share master public keys across wallets. However, it makes less sense for WIF because it's mostly used to import/sweep keys from other wallets. There is also a discussion on whether wallet software should attempt to error-correct private keys.
