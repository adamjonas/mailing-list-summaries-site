Summary:

In various email exchanges and discussions, the concept of Simplified Payment Verification (SPV) proofs is explored. SPV nodes trust that the most-work chain is valid based on economic arguments about the opportunity cost of mining invalid blocks. Linear scanning of block headers can be avoided by using back-links, which reduces space usage. Sergio Lerner's definition of SPV proof is a sequence of bits that establishes a block as part of the best-chain, while Mark Friedenbach's definition is a sequence of bits that convincingly establishes for a client without access to the blockchain that a block has an ancestor at some specified height and work distance back.The conversation between Mark Friedenbach and Sergio Lerner centers on the security assumptions of SPV nodes, and the proposed SmartSPV protocol does not meet these requirements, relying instead on blind trust in one's view of the network from their peers. Additionally, Peter Todd suggests that back-link block-history commitments with an interactive protocol would allow a binary search to find common ancestors and trust the intermediate links actually exist, without relying on honest peers. The proposed solution involves using NPPs and additional back-link commitments to construct an SPV proof with an average of log(N) headers. The email also proposes a use case for an SPV client that has just come online to ask peers what the last block height/time is. Furthermore, Peter Todd counters the assumption that economic arguments based on the opportunity cost of mining invalid blocks are necessary to trust the most-work chain as valid, without invoking moon math or interactive protocols requiring honest peers over jamming-free networks.A proposed use case for an SPV client involves asking peers what the last block height/time is and retrieving compact non-interactive proofs-of-inclusion for transactions without downloading every intervening block header. The recipient of an argument regarding SPV proofs believes that they have broader applications, including 2-way pegs. These pegs use proof-of-work to reach a consensus over the most-work side-chain header. While an honest peer is necessary to learn about the best chain, trust in that peer is not required because a non-51% attack against the user is probabilistically detectable with existing tools.The SmartSPV protocol may be useful, but the recipient argues that back-link commitments and compact SPV proofs can accomplish the same thing more efficiently. The recipient suggests that these methods can achieve the same results with fewer messages, less bandwidth, and equal security. However, they note that the two proposals are not in conflict with each other.
