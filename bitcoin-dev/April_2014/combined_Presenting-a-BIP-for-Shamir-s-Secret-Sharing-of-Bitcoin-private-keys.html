Summary:

The Bitcoin community has been actively discussing and exchanging emails about ways to improve the security and usability of Bitcoin wallets. One proposal that was debated was the use of Shamir's Secret Sharing to decompose a seed for a BIP32 master key, allowing for reconstitution of the key without revealing information about it. However, there were debates over whether to use GF(2^8) or big integer arithmetic in implementing SSS, as well as changing the key fingerprint algorithm. Some suggested using multisig transactions instead of SSS.Discussions also revolved around testnet encodings, with some suggesting that extra encoding was unnecessary given the many alternative chains available. Tamas Blummer suggested that BIPs should be chain agnostic and questioned the usefulness of testnet in BIPs. There were proposals made for adding prefixes for testnet, encoding N-of-M shares, and distinguishing serialization of keys when using test chains.Another topic of discussion was the compatibility between private key secret sharing schemes with threshold ECDSA, the challenges involved in identifying compatible fragments in wallet security, and the need for a decoding scheme that supported list decoding without increasing the complexity of basic implementation. The choice of fields for an implementation of this BIP was also discussed, with any implementation needing big integer arithmetic to do base-58 anyway.Despite these issues, most agreed that if people are using SSS in the field, it is better to write a BIP than not. Armory has implemented "Fragmented Backups" which are appreciated by advanced users, while Matt Whitlock is working on extending his draft BIP to apply to BIP32 master seeds of various sizes. Gregory Maxwell and Watson Ladd discussed the possibility of computing a signature from shares without reconstructing the private key using MPC techniques for Bitcoin. Matt Whitlock expressed his intentional omission of the parameter M from shares to prevent adversaries from having crucial information. Alan Reiner believed that omitting parameters from shares was problematic because users may make backups of wallets without correctly remembering the M-parameter. Armory is a Bitcoin wallet that defines the "Fragment ID" as the first few bytes of the hash of the root pubKey + M-parameter, converted to base58.In one email exchange, Jan Møller suggested using different prefixes for testnet keys/seeds to avoid mixing them up with mainnet keys. Tamas Blummer proposed using Shamir's Secret Sharing to decompose a seed for a BIP32 master key, while acknowledging that multi-signature transactions offer more fine-grained control of security at the transaction level. The conversation among Bitcoin developers and open-source evangelists acknowledged the value of an informational BIP on Shamir's Secret Sharing Scheme, suggesting usability features can be added while keeping the underlying secret well-protected.
