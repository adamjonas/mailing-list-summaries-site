Summary:

The conversation discusses the issue of network performance and how it can be improved. Matt Whitlock suggests that the problem lies with people not knowing how to configure their routers correctly. He recommends moving the packet queue from the cable modem into the router where intelligent decisions about packet priority and reordering can be enacted to improve network performance. The author questions whether high traffic on a Bitcoin node could cause problems for home routers and modems, such as crashing or slowing down due to overflow of NAT pin-hole tables, UDP traffic, and send buffers being filled up with outgoing data causing delays on interactive traffic. They suggest that implementing uTP could greatly improve the performance of the Bitcoin network by using a congestion window algorithm to use as much TCP bandwidth as possible while throttling down in case of cross traffic and considering other features like optimum MTUs and clock drift phenomena. Wladimir van der Laan, the lead developer of Bitcoin Core, addressed concerns about the potential impact of high traffic on Bitcoin nodes. He stated that the default maximum amount of connections is 125, which should not be a problem for home routers even if they are cheaply made. Additionally, Bitcoin does not use UDP, so there should be no impact from UDP traffic. However, filling up a home DSL or cable modem's send buffer with outgoing data is possible and could cause delays in interactive traffic. Throttling to prevent this is planned but postponed until parallel block download is implemented. When asked about using uTP to improve network performance, Wladimir stated that there were enough other areas for improvement to focus on first, and that benchmarking and analysis should be done to identify these areas.The conversation also highlights the issues that the bittorrent community faced and solved with uTP. One of the problems was a home router crashing or slowing down when its NAT pin-hole table overflows, triggered by many TCP connections. Another issue was a home DSL or cable modem having its send buffer filled up by outgoing data, causing delays on interactive traffic. The bittorrent community solved these issues with uTP, which uses a congestion window algorithm that allows for better performance. However, there are concerns about adding an own UDP network stack involving a ton of exposed code, which could lead to inadvertent bugs or backdoors. It is suggested that there doesn't have to be just one network transport for Bitcoin, and it would be great for someone to go build an alternative transport protocol to gateway to and see what useful things they can accomplish.Finally, Gregory raises a question about the possibility of having multiple network transports for Bitcoin. There is no formal abstraction for a Transport layer but it can be created when required. Angel asks if Bitcoin nodes could cause routers to crash or slow down due to high traffic, but Wladimir explains that the default maximum amount of connections is 125 and there should be no problem even for cheap routers. Filling up the send buffer is possible, but throttling will be postponed until parallel block download implementation. Finally, Wladimir suggests starting with benchmarking and analysis to find out where the performance issues lie.
