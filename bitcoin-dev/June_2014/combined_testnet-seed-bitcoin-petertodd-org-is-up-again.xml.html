Summary:

In the context of DNS settings for Bitcoin seeders, various individuals are discussing and exchanging emails regarding the problems they are facing. Alex Kotenko has set up DNS seeds but is having trouble with his DNS settings, and seeks help from Robert McKay to debug it. McKay suggests that the 'aa' flag is missing in the reply, causing the recursor to go into a loop chasing the authority server, resulting in a non-authoritative answer with itself as the authority again.To solve the problem, the use of bind forwarding is suggested, but it causes issues with the authoritative answer bit being stripped off in the reply, leading to a loop in the recursor. The solution of setting up a custom SOA record is considered but raises concerns about its effectiveness and appropriateness. Suggestions are also made to add simple logging to the seeder and to use DNS over HTTPS for protocol redundancy.The importance of having fallbacks such as hardcoded node lists and manual peer entry is emphasized, especially during initial startup, and running a centralized high-availability/low-latency node is recommended for instant startup. However, this method relies on a centralized resource. Overall, the discussions aim to find solutions to ensure the smooth functioning of DNS settings for Bitcoin seeders.Another topic discussed is the inability of the seeder or bitcoind to restart at home due to Todd's travel locations. It is suggested that teaching the seeder or bitcoind how to write a zone file and use matured DNS servers to serve this zone would be beneficial for everyone. Pieter Wuille suggests adding support for an option to the seeder, "for non-matching requests, forward to other DNS server at IP:PORT" so that they could cascade them. However, Robert McKay opposes the idea, stating that the 'aa' flag is missing, and it might not work for some resolvers.Andreas Schildbach suggests that the seeder or bitcoind should be taught how to write a zone file and use DNS servers to serve this zone, but he was uncertain if this approach would work since he had never run his own DNS before. Matt's implementation is already following this approach, thus Andreas does not need to run his own implementation. Pieter chose not to do so as he wanted to serve different responses to each query, but more diversity in implementation is still beneficial.The conversation also touches upon issues with ISP's buggy DNS servers and running centralized high-availability/low-latency nodes. The suggestion was made that if an application is insecure because all the DNS seeds are down, then the application needs to be fixed, and fallbacks such as hardcoded node lists and manual peer entry should be implemented. Running a centralized high-availability/low-latency node can ensure instant startup, but this method relies on a centralized resource.In conclusion, the discussions aim to find solutions to ensure the smooth functioning of DNS settings for Bitcoin seeders, which include suggestions such as adding simple logging, using DNS over HTTPS for protocol redundancy, and having fallbacks such as hardcoded node lists and manual peer entry. The importance of avoiding reliance on centralized resources such as high-availability/low-latency nodes is also emphasized.
