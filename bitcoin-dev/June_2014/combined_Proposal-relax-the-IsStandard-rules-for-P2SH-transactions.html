Summary:

The discussion is about opening up the "IsStandard" transaction rules on the main Bitcoin network to allow the use of multi-signature and other more complicated transaction forms in applications. Gavin Andresen, Chief Scientist of Bitcoin Foundation, suggests turning a GitHub gist into an informative BIP. The proposal is to allow any script containing 15 or fewer signature operations as a pay-to-script-hash (P2SH) script to be relayed and mined by the reference implementation.However, there are two risks to doing this: the risk of a not-yet-discovered chain-forking bug and the risk of a denial-of-service attack via an expensive-to-store-or-verify transaction. Despite this, Andresen believes that the risk of such bugs is low, and there are protections in place to mitigate the potential for DoS attacks. Additionally, a soft-fork-safe patch is being worked on which defines an opcode whitelist of soft-fork-safe opcodes and considers scripts that leave extra items on the stack after execution to be non-standard.In 2014, Andresen proposed keeping the current limit of 15 ECDSA signature checking operations per transaction output in place to prevent attackers from crafting a single standard transaction that requires an order of magnitude more validation. Wladimir agreed with the proposal, stressing the importance of lifting restrictions and increasing flexibility while maintaining DoS limits. However, the proposal would require specialized wallets or applications to use new transaction forms.In an email conversation between Andresen and Peter Todd dated June 18, 2014, the two discussed proposed changes to Bitcoin's code. Andresen suggested that all changes should be separate pull requests, and Todd disagreed, noting that the malleability protection change had already been heavily reviewed and would be beneficial. The two also discussed whether future soft-forks would always require a transaction nVersion bump or if a whitelist was necessary to protect against invalid up-version transactions.Andresen believes that comments made by Todd should be separate pull requests to avoid bike-shedding and make it easier to review, and there is rough consensus regarding the lack of immediate negative feedback on the issue. Detecting scripts that leave extra items on the stack will be done in a different part of the code than the one addressed by the pull request. The idea behind IsStandard is to get a quick yes/no before running a potentially expensive Script execution.Overall, the conversations focused on proposed changes to Bitcoin's code, including the use of soft-forks bumping version numbers and allowing for more complicated transaction forms. There were debates around potential issues such as invalid up-version transactions and "do-nothing" behavior with certain opcodes. However, there was general consensus around the need to maintain DoS limits while increasing flexibility.
