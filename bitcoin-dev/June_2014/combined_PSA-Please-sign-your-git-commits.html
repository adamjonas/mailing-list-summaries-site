Summary:

The context is divided into two main discussions. The first conversation revolves around the security of multi-sig wallets and PGP keyrings in Bitcoin Core development. Jeff Garzik believes that current multisig wallet technology with 2FA and other features is more secure than PGP keyring, which lacks 2FA and has associated risks such as theft, data destruction, and backup. However, some contributors note that despite the compromise process of PGP keyring being rarely exercised and offline signing working well, implementing multisig for git commits could reduce damage when a developer's computer is compromised. The second discussion focuses on the benefits and drawbacks of commit signing in Git. Contributors explore the usefulness of signed commits in identifying compromised accounts or stolen commits, but there are also concerns about its efficacy. Linus Torvalds' negative opinion on commit signing is cited, and various solutions are suggested, such as using a script to add a check for signed commits and rejecting pull requests without signed commits. The email conversation between David A. Harding and Chris Beams provides useful tips for ensuring proper signature of commit messages in Git projects, including adding a script to .git/hooks/post-commit and post-merge to ensure proper signature. They also suggest a pre-push hook instead to defer the check for signatures until the last possible moment, or a git config option to always sign commits. Overall, these discussions provide valuable insights into the challenges and potential solutions for using commit signing in Git and improving workflow designs to reduce damage when a developer's computer is compromised. There is also recognition of the historical context of PGP and its limitations in terms of security compared to current multisig wallet technology.
