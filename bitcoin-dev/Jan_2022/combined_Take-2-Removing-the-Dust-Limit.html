Summary:

The email exchange on the Bitcoin-dev mailing list revolves around the use of 0 value outputs in Bitcoin transactions and their potential impact on the mempool. The discussion includes proposals for creating a UTXO with drop semantics, which would expire and become unspendable if not spent in the block it was created, and using 0 value outputs solely for attaching fees in the mempool. Additionally, there are suggestions for allowing 0 value outputs to be used as Intermediate Outputs and requiring the parent to have a higher feerate after CPFP'ing the parent than the parent alone.Jeremy Rubin proposes a policy change that would allow 0 value outputs for immediately spendable outputs, given certain conditions. This proposal takes into account previous concerns around 0 value outputs and suggests using them solely for attaching fees in the mempool. A utxo with drop semantics is suggested to cover OP_TRUE anchors efficiently and ones that require a signature as well. Ultimately, the goal is to make 0 value outputs inconvenient enough to deter their usage, rather than making them impossible. The author of the message inquired about distinguishing between two dust UTXOs used in lightning anchor outputs and wondered if the protocol could force the user to spend the first of the two outputs. It was noted that values equal to or less than $1 are rarely spent unless forced to do so and suggested storing them separately. The current system uses 330 sats amount, with two commitment transactions having such outputs. One output remains untouched, leading to bloat in the UTXO set, while the other is spent to confirm transactions. Bastien notes that N-party protocols would need N such outputs to avoid pinning, but he's not sure. Anyone can spend these after a CSV of 16 blocks, but this trick wouldn't work for 0-value outputs. The email exchange also discusses Decker Channels, SIGHASH_ALL / APO, and congestion-control trees as possible solutions to certain issues. The proposal for allowing 0 value outputs as Intermediate Outputs has several benefits, including allowing 0 value outputs for things like Anchor Outputs and being helpful for CTV-based contracts and spacechains. Two cases could be added: one bare segwit version and one equivalent to taproot.
