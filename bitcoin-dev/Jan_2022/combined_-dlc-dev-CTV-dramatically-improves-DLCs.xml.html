Summary:

The email thread on the Bitcoin-dev mailing list discusses the application of OP_CTV and its alternative covenant constructions. ANYPREVOUT-based covenants are suggested as a viable option to enable similar benefits, with the script of Taproot leaves being set to CHECKSIGVERIFY CHECKSIGVERIFY with ANYPREVOUTANYSCRIPT signature of the CET for public key P=G. However, this method has an additional overhead of 64 witness bytes compared to CTV.Thibaut Le Guilly responds to a message about using CTV in the Discreet Log Contract (DLC) protocol, suggesting that CHECKSIGFROMSTACK might be a better opcode for DLC because it would allow requiring an oracle signature over the outcome without any special tricks or needing the oracle to release a nonce in advance. However, he admits that he hasn't studied covenant opcodes in detail yet. Jeremy responds to Thibaut's message, stating that CSFS might have independent benefits but in this case, CTV is only being used in the user-generated mapping of Oracle result to Transaction Outcome. He adds that it would only be complimentary if Thibaut came up with something CSFS based for the Oracles. Overall, the email thread provides insights into different options for enabling similar benefits to OP_CTV and the potential use of CHECKSIGFROMSTACK in DLC.
