Summary:

The discussion revolves around the issue of sharing alt-chain merkle tree. The proposal is to use UUID-as-path for this purpose with a full 256 bits for better security. To prove extra data validity, a whole Merkle path is required. It is advised to limit actual path lengths and choose something like 32 levels.One suggestion involves using H(uuid | nonce) as allowed paths, while another suggests using the last txout so that midstate compression can be applied in the future. However, storing the whole Merkle branch is not feasible, and it is suggested that the data stored in the coinbase output can always be the 256-bit root hash truncated to less.Mike Hearn suggests defining a short hash in the coinbase to commit to extra data that is relayed along with block data. This will not be part of the chain but can be extended as required and can be stored temporarily in the block db and erased after some time. Mark Friedenbach proposed using per-purpose UUIDs to validate data committed to a jagged tree structure. The UUID bits are interpreted as allowed paths (0 = left, 1 = right) from the top of the tree, and when building the tree, everything that is going to be committed must use its allowed path. If everyone picks their random per-purpose UUIDs, the paths won't collide for very many levels on average, and path lengths will remain short. Validating that some given data was committed properly is simple and easy: just check the path and ensure that the directions from the top of the tree followed the spec. Peter Todd suggested using a hash256-to-UUID mechanism explicitly for this purpose.Mark also proposes an authenticated prefix tree, where composable/commutative properties are not needed, so the path validation, traversal, and proof size can be smaller by using level compression. He offers to make finishing this a higher priority if there is general interest.The proposals aim to solve the problem of double-counting the bitcoin PoW for more than one aux block on the same chain, potentially facilitating aux chain attacks. Overall, the conversation revolves around various solutions to improve the security and validation of transiently relayed data, such as IP addresses and extra data in alt-chains, by utilizing per-purpose UUIDs and jagged tree structures.
