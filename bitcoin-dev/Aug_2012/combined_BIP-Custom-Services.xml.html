Summary:

During a discussion about service advertisement, an individual expressed their lack of enthusiasm for the topic and raised a valid concern. They noted that capabilities that are not advertised with CAddress cannot be effectively utilized. In response, Jeff suggested that if individuals will inevitably implement their own P2P commands for their client subset, some basic rules should be established regarding how to extend the P2P command set.While there may be pushback against adding extra functionality to the core Bitcoin network, people may choose to reuse the code with different peer discovery seeds and bootstrap similar but unrelated P2P networks for new applications. BitcoinJS is implementing a Distributed Hash Table (DHT) that will store only a user-defined percentage of the blockchain index locally, giving users the option to run anywhere from a large BitcoinJS node to a lightweight one.A second BIP was created to discover and make use of these capabilities by prefixing the standard addr message with a service-specific message to mark off service support. The DHT network is extensible, allowing the decentralized pool protocol to live within it, and custom services can be added to the standard protocol. Stefan Thomas's solution to announce custom protocol extensions without using up NODE_* flags was refined with feedback from Amir Taaki, Mike Hearn, and Pieter Wuille.Earlier drafts contained a description of a mechanism for peer exchange for these custom services, which was split off into a separate BIP. Overall, more elaboration of the use cases for additional functionality could be helpful in determining whether or not it should be added to the core Bitcoin network.
