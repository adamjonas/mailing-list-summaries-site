Summary:

The context revolves around the PaymentRequest system and ways to improve its efficiency and security measures. One of the main topics of discussion is the use of hash functions in generating QR codes for payment requests. It is suggested that the length of the hash should be determined by the person generating the QR code, with implementers experimenting with different lengths to find the optimum balance between security and convenience.SHA256 is considered a viable option for hashing, but it's noted that the first x bits of a SHA256 sum are just as good as an equally secure hash function of that length. Adding more bytes to QR codes impacts scannability, and concerns are raised about making QR codes bloated and harder to scan. BitPay's testing shows that adding another field to PaymentRequest containing an ECC signature calculated using the public key adds complications, raises concerns about re-purposing BIP21 address, and excludes stateless devices.There are discussions around the efficiency of adding another field to PaymentRequest containing an ECC signature calculated using the public key, which adds complications and raises concerns about re-purposing BIP21 address and stateless devices. Mike Hearn suggests that 80 bits of entropy would be sufficient for payment requests, evaluates different hash functions, and suggests signing with the key in the first part of the URI as an alternative in the Bluetooth context.Another proposal involves adding an "h" parameter containing a hash of the PaymentRequest message fetched via the "r" parameter to link the trust anchor to the payment request message securely. This amendment to BIP72 prevents certain apps from comparing address+amount fields and is considered a secure way to link the trust anchor to the payment request message.The discussions also touch on the use of certificates in verifying website authenticity, along with concerns about X.509's ability to replace trust anchors established by scanning QR codes or tapping devices together. It is suggested that existing address hashes could be used as both a backward compatibility measure and an authentication mechanism.Overall, the focus is on finding efficient and secure ways to improve the PaymentRequest system. Different proposals are discussed, evaluated, and tested to find the best way forward. Leaving the implementation of different lengths open for experimentation could determine the optimum length, and the use of existing address hashes is considered as a viable solution.
