Summary:

The Bitcoin Core implementation of BIP159 proposes to allow pruned nodes to serve a limited number of historical blocks. However, there is concern that this may be wasteful for nodes with 10-100 GB of storage as they would only be able to share the most recent 288 blocks. To allow for more flexibility and less waste for these nodes, a future extension of the BIP could limit choices to e.g. 288 + 1000 * 2^n. There is also concern about a counter-measure for peer fingerprinting that would limit pruned nodes to only serve the last 288 blocks. This may require a new message type to communicate the chosen prune depth to avoid fingerprinting. Waiting for BIP150 may be appropriate in this case.Suggestions for improving the BIP include adding links to mailing list discussions in the reference section and explaining that 288 is not just the minimum limit for Bitcoin Core but also the bulk of traffic. Defining additional pruning levels has been delayed due to insufficient experience from practice and the likelihood of poor interaction with other future proposals. Currently, the network has an abundance of nodes that serve anything, making rare handles for additional blocks moot. It is worth noting that Gregory Maxwell shared his experience that virtually no one sets any parameter other than the minimum while using pruning in Bitcoin, but Peter Todd runs all his pruned nodes with the prune parameter set to about a month's worth of blocks, which is a few GB.
