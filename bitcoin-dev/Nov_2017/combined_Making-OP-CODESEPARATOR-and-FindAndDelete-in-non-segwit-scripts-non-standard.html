Summary:

The Bitcoin community has been discussing the proposal to remove OP_CODESEPARATOR and FindAndDelete() from non-SegWit scripts. The aim is to eventually soft-fork these features out, although some have raised concerns about destroying value and leaving pre-signed transactions vulnerable.Making them non-standard is seen as a good first step in discouraging their use and better informing discussions about the possibility of anyone still using them. It has been suggested that it may be necessary to set clear best-practices. Soft-forking behavior into OP_NOPs or nSequence could be useful for making the code more secure, and it is believed that we should be willing to soft-fork out things which clearly fall outside of best-practices.Johnson Lau proposes disabling both functions to guarantee that scriptCode serialized inside SignatureHash() must be constant. However, Mark Friedenbach objects to this proposal as it could potentially destroy value. Nicolas Dorier pointed out that he was using CODESEPARATOR in TumbleBit, but only in places like SegWit where the potential validation complexity blowup is massively reduced.If a softfork is used to remove FindAndDelete() and OP_CODESEPARATOR from non-SegWit scripts, all blocks before the softfork block can be whitelisted to completely remove FindAndDelete() from the consensus code later. Matt Corallo believes that making things non-standard is a good change on its own. He disagrees that only fundamentally insecure transactions should be soft-forked out, pointing out that they have generally been willing to soft-fork out things which clearly fall outside of best-practices.In conclusion, while there are concerns about destroying value and pre-signed transactions, making OP_CODESEPARATOR and FindAndDelete non-standard is seen as a necessary step in discouraging their use and setting clear best-practices. It has been pointed out that making things non-standard is often the first step towards soft-forking out behavior.
