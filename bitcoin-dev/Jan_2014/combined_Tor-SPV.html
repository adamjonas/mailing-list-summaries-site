Summary:

The conversation in this context revolves around the need to improve the privacy and security of Simplified Payment Verification (SPV) wallets. Various suggestions have been made, including connecting without using Tor by identifying nodes on the clearnet via their addr announcements+service flag. However, some suggest that connecting through Tor would preserve the privacy of the SPV node and reduce the ability of an attacker to MITM. The use of hidden services is also discussed, with the main goal being to provide authentication and encryption for secure connections.There has been a proposal to introduce a new P2P protocol command "tor?" to add a service flag and bump the protocol version. This would allow SPV wallets to bypass Tor entirely if the mapping is recorded on disk. Anti-Sybil techniques are required, and various suggestions have been made, such as using proof-of-stake as an anti-Sybil scheme on Tor. Using hidden services as a means of authentication and encryption raises concerns about malicious nodes generating multiple hidden service keys that map to their own node and waiting for traffic to arrive. To avoid this issue, clearnet nodes can inform about their own hidden service IDs.Mike Hearn proposed three ways in which SPV wallets could identify nodes for measuring pending transactions, such as querying a set of seed nodes or identifying nodes on the clearnet via their addr announcements+service flag. It was suggested that SPV nodes could connect to the IP using Tor, which would preserve the privacy of the SPV node and make it hard to see that it was running Bitcoin.The proposals aim to provide better privacy and security for SPV wallets and improve the overall transactions per second on the Bitcoin network. However, there are concerns that the use of hidden services may result in network slowdowns and reduce overall transactions per second. In a forum post, Mike Hearn discusses using IPv4-based anti-Sybil heuristics that could improve the system in the future.A proposal has been made to use hidden services as a more complicated, out of process/sandboxable SSL for protecting end-user-to-network communication. The main problem with this approach is that SPV wallets cannot verify unconfirmed transactions and rely on counting peers, which can increase the probability of being MITMd. Therefore, a proposal has been made to upgrade the P2P protocol by introducing a new service flag and tor? message to identify nodes on the clearnet via their addr announcements+service flag, allowing SPV wallets to bypass Tor entirely. Additionally, it is suggested to gossip the mapping of (IP -> onion address) to allow detection of a future MITM. The suggestion is to add the service flag so that the remote peer can add IPv6-ified hidden service addresses to "addr" messages instead of a separate message type that implies binding a clearnet IP to a hidden service ID. Brooks Boyd proposed utilizing hidden services for their encryption capabilities rather than to hide P2P nodes. It was suggested that three-hop hidden service circuits would be sufficient for most nodes that weren't Tor-exclusive, as six-hop circuits were deemed overkill. However, reducing the number of hops below three on both sides (client/server) would require a change to CircuitPathChooser for Orchid or custom-building a circuit using the control port for other Tor implementations. The adoption of such measures may result in network slowdowns and a decrease in overall transactions per second on the Bitcoin network.
