Summary:

A recent discussion on the bitcoin-dev mailing list proposed changing the fee requirement for transactions to support more use cases and make it more incentive compatible. This change would allow for "retroactive transaction merging" and would be resistant to attacks, but there is concern about the incentive compatibility of BIP125 since it does not seem rational to accept a replacement transaction with a lower fee rate in a competitive mempool. Another topic discussed is the feasibility of merging multiple unconfirmed transactions while stripping extraneous inputs and change as they go. However, the bip125 rule currently prohibits this by requiring that the replacement transaction pays an absolute fee of at least the sum paid by the original transactions.Some users have proposed modifying the rules of BIP125 to allow retroactive transaction merging, although this could introduce DoS vectors. The discussion also explores possible solutions for tracking replaced transactions, such as only considering a transaction replaceable if it has change, but this opens up complexities. One solution proposed is to limit abuse by requiring the fee rate to increase, which is similar to the spirit of RBF.In addition, the context discusses the challenges of determining whether it is safe to send payment to a user without waiting for n transactions after a transaction has been confirmed. One possible solution is to consider a transaction "replaceable" only if it has change so that payments can immediately be made that source the change and provide safety in a reorg. However, this adds complexity and only works for transactions with change.The conversation also revolves around whether it is better for miners to relay a replacement transaction over the original transactions, which may not always result in higher fees. It is argued that it is better for miners to know about the replacement transaction as it costs them nothing, but not knowing about it may result in less earnings. For those creating the replacement transaction, it is better for their transactions to be mined instead of the more expensive original transactions.Overall, the discussions on Bitcoin-dev highlight various challenges related to Bitcoin transactions, including incentive compatibility, DoS vectors, and misconceptions around change. The different proposals and suggestions reflect the ongoing efforts to improve the efficiency and security of Bitcoin transactions.Lastly, the context talks about the difficulty of getting no change with an intelligent algorithm selecting coins when paying for outputs. Peter Todd challenged the assumption that most transactions don't have change, stating that almost all transactions have change because it is rare for inputs to exactly match requested payment. However, Ryan disagreed with this statement, saying it was a common misconception and claimed that good coin selection could avoid change about 40-50% of the time, although he admitted that this rate would be lower in practice. The discussion then moved on to tricks for avoiding change, such as "miner sacrifice" and "output selection". Achow101 and Murch have created a code to implement an efficient algorithm for finding solutions for Bitcoin core.
