The context discusses various proposals and discussions related to Bloom filtering in Bitcoin. One proposal by Matt Corallo and Peter Todd suggests extending BIP 37 with a new service bit called NODE_BLOOM that would allow nodes to explicitly advertise their support for bloom filters. This is intended to address the lack of privacy and DoS risks associated with connection bloom filtering in BIP 37. In response, the need for changes that would begin the process of dropping support for SPV clients was discussed. Additionally, there is a discussion regarding the NODE_BLOOM service bit, its implementation, and the potential issues related to scalability and DoS attacks. The context discusses various Bitcoin Improvement Proposals (BIPs) related to bloom filters and connection filtering algorithms, as well as the risks associated with relying on seed nodes for starting nodes. Peter Todd raises concerns about scalability and privacy issues with Bloom filters and suggests the need for a feature that allows node operators to disable connection bloom filtering. The NODE_BLOOM service bit is proposed, which extends BIP 37 and adds a service bit to allow nodes to advertise that they support bloom filters explicitly. Additionally, Todd notes the decrease in the number of Simplified Payment Verification (SPV) clients on his high-speed nodes and discusses the popularity of wallets such as Mycelium that use SPV verification of block headers with lookup servers instead of bloom filters. Todd also suggests handling future upgrades to more scalable designs by adding a bit. Overall, the context covers discussions related to improving the security and scalability of Bitcoin through various proposed changes and features.A proposal to extend BIP 37, Connection Bloom filtering, has been made by Matt Corallo and Peter Todd. The proposal defines a new service bit called NODE_BLOOM, which enables peers to explicitly advertise their support for bloom filters. In addition, the protocol version is increased from 70002 to 70011 in the reference implementation. This allows nodes to identify old nodes lacking the new service bit but still allow bloom filtering of the connection. The proposal aims to address the lack of privacy and DoS risks associated with connection bloom filtering in BIP 37.It also suggests that DNS seeds and other peer discovery mechanisms should support the ability to specify required services. The proposal is currently a draft and available on GitHub for review. In response to concerns about the impact of widely deploying and enabling bloom filters on current wallets in use, it was suggested that the Core implementation accept and respond to bloom filter requests from non-upgraded clients until some fixed upgrade deadline in the future, regardless of whether or not NODE_BLOOM was set. Bloom filter usage has declined significantly as lite-SPV clients are moving towards using centralized, trusted servers run by wallet authors such as Mycelium, GreenBits, AirBitz, and Electrum. Additionally, Bloom filters have been found to have severe privacy issues offering essentially no privacy at all. Furthermore, Bloom filters still have unsolved DoS attack issues that will get significantly worse under upcoming blocksize increase proposals.The proposed NODE_BLOOM service bit is intended to allow peers to advertise that they support bloom filters explicitly. It also bumps the protocol version to allow peers to identify old nodes which allow bloom filtering of the connection despite lacking the new service bit. BIP 37 did not specify a service bit for the bloom filter service, thus implicitly assuming that all nodes that serve peers data support it. However, the connection filtering algorithm proposed in BIP 37 has been shown to provide little to no privacy, as well as being a large DoS risk on some nodes. Thus, allowing node operators to disable connection bloom filtering is a much-needed feature.The proposal suggests that DNS seeds and other peer discovery mechanisms should support the ability to specify required services. Current implementations simply check only that NODE_NETWORK is set. The increase in protocol version is for backward compatibility. In initial implementations, old nodes which are not yet aware of NODE_BLOOM and use a protocol version below 70011 will function with a node without NODE_BLOOM. This feature may be removed after there are sufficient NODE_BLOOM nodes available and SPV clients have upgraded, allowing node operators to fully close the bloom-related DoS vectors. Overall, there is a need to improve filtering options within Bitcoin connections while considering scalability and privacy concerns.
