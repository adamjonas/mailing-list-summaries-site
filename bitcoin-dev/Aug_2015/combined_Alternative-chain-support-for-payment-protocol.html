Summary:

Discussions among developers have focused on how to support side or alt-chains in BIP70, a payment protocol for bitcoin. One solution suggested is the bip44 coin type registry, which allows an optional coin type field to be added to PaymentRequest and serializing incompatible PaymentDetails to the serialized_payment_details field. However, issues related to human-meaningful identifiers and name collisions have been raised. The genesis hash has also been suggested as an alternative to short names.Mike Hearn notes that BIP70 does not define new strings for alt coins because they can vary in arbitrary ways from Bitcoin, but if an alt coin is similar enough to BIP70 that it can be reused "as is," then a new network string for the alt could be defined. Another option suggested by Hearn is to use the genesis hash as the network name, although this may involve lookups to determine what the request is for. Discussions on supporting side or alt-chains in BIP70 also include the limitations of BIP 70 in supporting different testnets and alternative coins. Ross Nicoll proposes adding a new optional "genesis" field containing the SHA-256 hash of the genesis block of the network the request belongs to, which will uniquely identify chains without a central registry requirement. For backward compatibility, the "network" field would contain "main" for Bitcoin main net, "test" for Bitcoin testnet3, and "other" for other networks apart from those two.The proposal to create a new network string for altcoins has been put forth by Ross Nicoll via bitcoin-dev. This is because altcoins can vary in arbitrary ways from Bitcoin and BIP70 may not work for them. The proposal suggests adding a new optional field named "genesis" in BIP70, which would contain a 16-byte sequence with the SHA-256 hash of the genesis block of the network to which the payment request belongs. This will enable unique identification of chains without any central registry. Furthermore, the context mentions related commits from #6382, including a pull request closed for now until #6235 is merged. The author wants to add the chainID field and support regtest in bip70 too. There are three different strings for testnet3: "testnet3" for the default data directory, "testnet" for the GUI style and strings showed to the user, and "test" for bip70. The author wants to simplify this by unifying everything to always use "testnet3", although that would require modifying bip70.In a discussion on August 10, 2015, Mike Hearn stated that BIP 70 is immutable and can only be extended. However, there was a point made about making exceptions for regtest mode, which would make the code more complex rather than simpler. It was argued that all supported chains should be treated in the same way in the code. The dedicated chain ID for regtest mode is not necessary as it should not be found outside of a developer machine where the ID does not matter.In an email exchange between Ross Nicoll and Luke Dashjr, they discussed the limitations of BIP 70 in supporting different testnets and alternative coins. Ross proposed a new optional "genesis" field to identify chains without any central registry, but Luke pointed out that genesis blocks are not necessarily unique. They also discussed using the most recent block hash for fork detection, but it could advertise if a merchant is on the wrong fork. Ross emphasized that avoiding future problems is the main benefit of the proposal. However, Luke reminded him that BIPs are for Bitcoin improvements specifically, and suggested that the proposal could address the need for payments made of bitcoins on a sidechain by allowing payment directly on the sidechain at the customer's choice.Overall, the discussions center around how to ensure the uniqueness of the chain ID while still allowing for multiple potential networks, as well as how to support side or alt-chains in BIP70. Suggestions include using the bip44 coin type registry, the genesis hash as a network ID instead of short names, and adding a new optional "genesis" field containing the SHA-256 hash of the genesis block of the network the request belongs to.
