Summary:

The Bitcoin community has been discussing the possibility of increasing the block size limit for some time now. One proposal, BitcoinXT, aimed to increase the limit to 8 megabytes but faced opposition from some members who saw it as a Schism hardfork. The success of BitcoinXT relied heavily on the consensus of major exchanges, merchants, and investors. Critics argue that it poses risks as a non-peer reviewed alt with problematic features, including privacy issues. Others suggest that well-tested hardfork proposals already exist that do not require a change in the maximum block size. Some believe that softforks are safer than hardforks.In addition, there has been discussion about the timing of the hardfork and the length of the grace period necessary for it. There have also been proposals to deploy experimental code onto the public global testnet blockchain to collect data for future hardforks. One proposal suggested increasing the block size limit to 1.5MB, citing technological advancements since the 1MB cap was set five years ago. While there is a consensus among core developers that the block size should be raised, questions remain about how, when, how much, and how fast.To test the feasibility of a successful hardfork, collect data for future hardforks, and relieve pressure on full blocks without adversely affecting network performance, an experimental hardfork has been proposed. However, deploying such experimental code on the live blockchain was considered unnecessarily risky. Instead, the proposal suggests deploying a block size limit experiment using the global testnet blockchain to collect data for future hardforks. The proposal includes parameters such as a hardfork date of June 1, 2016, adoption of a backup plan if no other consensus is reached before February 1, 2016, and an increase in block size to 1.5MB after 80% miner approval.If the backup plan was adopted, a better solution was expected to be found by December 31, 2017. The proposal acknowledges that the backup plan is not a final solution and aims to find a better solution with 1.5 years of data and further debate. If agreed upon, the plan will be announced on bitcoin.org, and coding of the patch will begin immediately.In an email dated August 19, 2015, odinn criticised Bitcoin XT, suggesting it poses major risks and has not followed any semblance of process, which is dangerous since it involves hard forking bitcoin while being well aware of miner voting on viable solutions which have followed the process. He suggests users consider BIP 100, but emphasises that it's users, not miners who decide the consensus rules. While odinn believes that Schism hardforks may be necessary in some situations, he doesn't consider this one reasonable, questioning the basis for further promoting XT by suggesting that it should even be tested.The message thread discusses the adoption of BitcoinXT by miners. Chinese miners, who control 60% of the network, have already stated they would not adopt XT, so cannot lead the revolution. Exchanges could encourage miners to switch to Bitcoin-XT by creating a taint system in which all non-XT coinbase outputs are marked as tainted and require a surcharge to be untainted. Another suggestion is checkpointing: creating one big block and agreeing to checkpoint it.Overall, the discussion revolves around the controversial nature of BitcoinXT and the proposed increase in block size limit. The community is exploring various options to test the proposed implementation of the blocksize limit, including deploying experimental code onto the global testnet blockchain. The aim is to evaluate real-world hazards such as network lag and traffic bottlenecks while reducing political angst about the proposals. A backup plan has been proposed if no other consensus is reached before February 1, 2016, which includes increasing the block size to 1.5MB after 80% miner approval. A better solution is expected to be found by December 31, 2017.
