Summary:

There have been ongoing discussions in the Bitcoin development community about the maximum block size limit. One proposal suggests replacing the fixed one-megabyte maximum block size with a dynamically controlled maximum block size that may increase or decrease based on various network factors. Two proposals were presented, each with its own rationale and considerations for implementation. Other solutions were also proposed, such as Making Decentralized Economic Policy, Elastic block cap with rollover penalties, Increase maximum block size, Block size following technological growth, and The Bitcoin Lightning Network: Scalable Off-Chain Instant Payments.There have also been discussions about different voting mechanisms to determine the maximum block size limit, including using XORing votes with SHA256 of the entire blockchain or locking coins (votes) through CLTV. However, concerns were raised about the feasibility and accuracy of these approaches, as they would require technical knowledge and may result in low participation rates.In response, Peter Todd proposed a voting mechanism that allows only users with access to validating nodes to vote, which is considered a fundamental requirement for users to fully participate in Bitcoin's governance. But some concerns were raised regarding this proposal, including how to prevent a user from running multiple nodes and how to verify that a node is a fully validating node with its copy of the blockchain.Ultimately, Bitcoin is not a democracy, and people will use whatever they want. While stakeholder vote is considered important to check on miners' power, it is likely to result in very low-quality results and drastically low participation, similar to rational ignorance. The default mechanism for change would be forking off altcoins and allowing the market to decide which coin is most valuable.The bitcoin-dev mailing list had a discussion about a proposed draft of DynamicMaxBlockSize, which uses a combination of proof-of-work and proof-of-stake models. Voting was not the focus of the discussion, but rather the proposal itself. Hector Chu suggested using CLTV to lock coins as a better alternative to counting votes from UTXO set. However, if a soft-fork to implement CLTV is not feasible, counting votes from the UTXO set can still be used as evidence of commitment. Peter Todd also proposed a toy example of XORing a vote with SHA256 of the entire blockchain, but it was deemed ineffective by Hector Chu.
