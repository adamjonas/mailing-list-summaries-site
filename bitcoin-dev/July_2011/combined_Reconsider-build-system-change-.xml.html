In various email exchanges and forum discussions, developers discuss the advantages and disadvantages of different build systems for software development. Autotools and CMake are two popular options that are compared and contrasted. While some argue that autotools is more standard and follows a traditional build procedure, others suggest that CMake is simpler, more elegant, and can generate native build scripts for multiple platforms. The issue of dependencies and familiarity with the systems is also discussed, with some noting that CMake requires a specific dependency while others point out that it is used by many open-source applications. The importance of choosing a reliable build system is emphasized throughout the conversations.In 2011, a discussion was raised regarding the future build system for Bitcoin. Pieter Wuille expressed his lack of preference for any particular project build system but mentioned specific criteria that should be met, such as ease of setup and maintenance, cross-compilation to Windows, support for OS X, and adaptability to other GUIs. Autotools have been questioned as a future build system choice over CMake, with many people discouraging the use of autotools for new projects. CMake generates native build scripts for Make on all platforms, XCode, and Visual Studio whereas Autoconf only does Make. Furthermore, CMake is supported natively in Qt Creator in the QT SDK and is used by many other KDE/QT as well as other open-source applications. Pieter invited other opinions and asked if anyone is interested in writing a CMake configuration for Bitcoin. John Smith argued in favor of CMake citing its ability to generate native build scripts for all platforms including Make, XCode, and Visual Studio, clearer syntax, faster speed, and native support in Qt Creator in the QT SDK. In various email exchanges and forum discussions, developers discussed the limitations and advantages of different build systems such as autotools and CMake, with some arguing that CMake is simpler, clearer, and more elegant compared to the convoluted m4/autotools syntax. They also discussed the importance of choosing a reliable build system in software development and the availability of options such as autotools and CMake.Some suggested using a shell script named 'configure' to start 'cmake .' and passing through command line arguments such as --prefix, --datadir, --mandir, etc. Overall, the discussion showed the significance of choosing the right build system in software development that meets certain criteria such as ease of setup and maintenance, cross-compilation to different platforms, and adaptability to other GUIs. The use of CMake was preferred by some due to its ability to generate native build scripts for various platforms, clearer syntax, and faster speed. In an email conversation between Luke-Jr and John Smith on July 2, 2011, John questioned the decision to choose autotools over cmake as the future build system. Luke-Jr expressed that he did not have much preference but pointed out that cmake does not follow the standard build procedure of './configure && make && make install'. However, he suggested that a shell script named 'configure' that starts 'cmake .' could be created and it would need to pass through some command line args, such as --prefix. The new sequence for building would then be 'cmake . && make && make install'.John Smith shared a link to a CMake build system for Bitcoin already integrated into rpcminer. In a follow-up email, John Smith highlighted that CMake supports cross-compiling with a link to the CMake Cross Compiling wiki page.
