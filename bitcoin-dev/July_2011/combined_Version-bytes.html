Summary:

Pieter Wuille's proposal to standardize the version bytes used by Bitcoin for multiple applications includes three meaningful components: network, data class, and version. The proposal aims to avoid collisions between different networks and data classes and suggests using bit 16 in the version byte for "alternate chain" and bit 1 for testnet. They have six more bits to play with, namely 128, 64, 32, and 8, 4, 2, which Wuille proposes using (128, 64, 32) for data classes and (8, 4, 2) for versions.Stefan Thomas suggested defining a format for master keys from which to derive deterministic wallet keys, using data class 192 for that purpose, leaving the lower numbers for basic data like public keys. However, the discussion also involved considerations for maintaining backward compatibility, building an image of stability, and visually differentiating testnet codes from realnet ones.On July 8, 2011, Stefan Thomas and JS discussed the possibility of resetting testnet's addresses or using the XOR hack to differentiate between odd (testnet) and even (realnet) network addresses. While Stefan suggested using the lsb as a flag instead of the XOR hack to maintain compatibility with older addresses, JS argued that backward compatibility was important for building an image of stability. Ultimately, JS voted in favor of using the XOR hack for the time being.In this email exchange, Pieter suggests resetting the testnet addresses to differentiate them from realnet ones. Stefan proposes using XOR hack for now but removing it when they reset the testnet. John agrees that XOR hack is cleaner, but breaking compatibility with older addresses is not a good idea at the moment. He argues that it is important to maintain backward compatibility and build an image of stability. Pieter points out that the 111 hack makes testnet codes visually different from realnet ones, which is probably why it was chosen. However, he also mentions that a modification to nVersion of less than 5 in absolute value can cause the first base58 character to remain equal. Additionally, Pieter notes that +/- 111 is also possible instead of XOR 111.
