In July 2011, the Bitcoin community engaged in a discussion about various aspects of the Bitcoin client codebase. Jeff Garzik suggested using "libbitcoin" and starting with a small module to implement in a branch. Matt Corallo discussed an alternative to using Berkeley db 4 by storing file offsets in an STL map. The group debated creating a chain description file for adjustable parameters and the importance of having clean, well-documented code. Luke-Jr suggested beginning a complete rewrite of the codebase, while Gavin Andresen proposed stopping Bitcoin usage for six months to implement a better client. The author argues that having 50-80 code files in Bitcoin may not be bad since it provides an easier-to-read history and fewer merge conflicts. Boost is an unnecessary dependency that should be replaced with lightweight alternatives. The Bitcoin codebase is in need of significant cleanup and reorganization to improve readability, reduce bugs, and make it a reference project for other clients. Improving the code quality is vital for the long-term success of the project.Michael Offel expressed concerns about the long build times for Bitcoin due to the use of boost, a lack of module isolation, and implementations in header files. Douglas Huff agreed with Offel's points and suggested they be addressed. Alan Grimes discussed how the one class-per .h/.cpp rule is being ignored in main.h/cpp and net.h/cpp. Jeff Garzik objected to refactoring all classes into separate files and suggested finding a balance between strict rules and putting logically similar code together.BitCoinJ has a useful NetworkParameters abstraction that groups constants together, making unit testing easier. In an email conversation, Michael Offel listed several areas that needed improvement in the Bitcoin client, including a lack of code documentation, poor use of threads, and hardcoded values. Jeff Garzik defends putting logically similar code together instead of splitting it into many tiny files.
