Summary:

The Bitcoin development mailing list contains several discussions about proposed changes to fix issues with web services being ignorant of generated transactions and the categorization/formatting system for wallets. Gavin Andresen suggests that fixing listtransactions and listreceivedby/etc should be done to credit coin generation transactions to particular addresses/accounts. However, he thinks that changing listtransactions output would not be on the priority list for any web services operators, and instead, they should focus on fixing getbalance and listtransactions performance problems and push-notification of coins received to accounts. Meanwhile, Stéphane proposes that all additions to a wallet should be categorized as "receive" regardless of the transaction type. He also suggests eliminating categories and relying on a "type" field with values of "credit" or "debit". This would allow for additional categories to be added in the future. Stéphane recommends the creation of an official API guide to provide more comprehensive information. In terms of specific changes, the proposed change aims to fix webservices being ignorant of generated transactions by keeping a standard category/format for all similar types of entries. Changes include additions to a wallet, which will always be categorized as "receive," regardless of transaction type, and adding an extra key from "generation" for generated coins. All transactions will have a new key, status, with different values depending on the confirmation stage of the transaction. Finally, it is recommended that people should not be using accounts for receiving funds on merchant sites. Instead, the recommended and safest way to receive funds on a merchant site ATM is to use listreceivedsince called every so often by a cron job. The output is parsed and placed into another table; then another job will go and create a deposit request from that table. Polling listtransactions is less convenient since you always repeat old transactions that you may not need, get generated transactions, and have to pick a good number that goes back far enough but not too far. The {list,get}receivedby* feature should also include generated amounts as there is currently no way to do so, although this issue has not yet been addressed by Stéphane's patch.
