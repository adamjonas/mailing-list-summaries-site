Summary:

The conversation revolves around Bitcoin's infrastructure, with a particular focus on the peer and wallet boxes in the diagram. The current code is considered trustworthy, even though it may lack comments. However, there are concerns about the status of the infrastructure, and it is not known whether it is fully implemented, a working or approved proposal. Regarding the peer, it is taken for granted that it will have a listening component that accepts connections from other peers, but it is unclear whether it has an active component and what autonomous operations it performs. Additionally, there are questions about the exact relationship between the wallet and miner.Most operating systems are multi-user systems, and each user would keep their wallet in an encrypted store in their home directory. However, there are currently no standards for users to set up their personal wallet daemon that connects to a system-wide peer node and provides an interface for only their own GUI.In response to the discussion, it is suggested that existing client libraries be helped with rather than creating new ones. The current code can be segmented into necessary pieces, although comments are being added in this regard. A suggestion is made to implement p2p on a SOHO router as the end goal. It is also possible to run a system-wide bitcoind (or equivalent) to handle p2p connectivity and have each user run a dedicated wallet bitcoind with a unique JSON-RPC port or wx GUI. Finally, the user suggests using an inetd implementation to minimize the amount of code at a marginal cost of overhead.
