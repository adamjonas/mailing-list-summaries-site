Summary:

The email thread discusses suggestions for securing the wallet.dat file. The first suggestion is to overwrite the old, unencrypted wallet.dat a few times with pseudo-random data to secure-delete it. However, Arthur Britto disagrees and suggests that writing zeros just once should be sufficient. The second suggestion is to mark all keys imported from an unencrypted file as potentially compromised and not use them for new addresses. Additionally, the thread mentions that moving the keypool addresses aside so they won't be used sounds like a good idea. It also highlights that there is no way for a wallet to be born-encrypted, so the only way to prevent a leak is to build the wallet initially on a ramdisk or the like, then move it over after encrypting it. Finally, the thread concludes that luke-jr's (2) would make the key leak on a new wallet inconsequential since all keys in it are keypool keys at that point.
