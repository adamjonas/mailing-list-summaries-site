In various email exchanges and forum threads, discussions have been held regarding the use of DNSSEC and HTTPS for Bitcoin transactions. One solution proposed was to use DNSSEC to incorporate Bitcoin into DNS, but others have suggested that HTTPS is a better solution because it can provide different addresses to each client based on their IP address. Some participants have also proposed using email-like addresses as identity labels instead of Bitcoin addresses, which would resolve through a SRV lookup for _bitcoinaddressresolution._tcp.foo.bar.net, expecting a certificate for bar.net.However, there are concerns surrounding the potential illegality of DNSSEC in the US and the difficulty in remembering unique Bitcoin addresses per transaction. Additionally, implementing a full DNS resolver with root trust anchors and knowledge of root servers in Bitcoin is not considered a good idea due to security concerns. Instead, it is suggested to use a simple HTTPS server with addresses and client support, which is more secure and simpler to implement. The thread also discusses the limitations and potential attacks on DNSSEC and HTTPS, and proposes solutions such as a standardized HTTPS-based protocol where a Bitcoin URI could contain a domain name and challenge the domain to sign a nonce with the key corresponding to the address.
