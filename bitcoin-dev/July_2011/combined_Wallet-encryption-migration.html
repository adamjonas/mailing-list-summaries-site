Summary:

When it comes to securing Bitcoin wallets, there are a few measures that can be taken. One suggestion is to overwrite an old, unencrypted wallet.dat file with pseudo-random data multiple times in an attempt to secure-delete it. Another suggestion is to mark all keys imported from an unencrypted file as "potentially compromised" and never use them for new addresses. This means avoiding the use of the old keypool for getnewaddress, change, and other related functions.In a discussion thread on protecting Bitcoin wallets from theft, Luke-Jr suggested both of these potential measures for securing the wallet.dat file. Arthur Britto agreed with Luke-Jr's second suggestion but noted that writing zeros only once may not be sufficient on modern Unix file systems because the data won't be written in place. Overall, Britto felt that Luke-Jr's second suggestion would make the key leak on a new wallet inconsequential since all keys in it are keypool keys at that point, and therefore should be implemented.In an email conversation between Luke-Jr and Arthur on July 14, 2011, Luke-Jr suggested writing over the old, unencrypted wallet.dat a couple of times with pseudo-random data in an attempt to secure-delete it. However, Arthur responded by stating that writing zeros just once should be sufficient. Arthur provided a link to a tutorial on data sanitization from the University of California, San Diego to support his claim. These suggestions aim to improve the security of cryptocurrency wallets and prevent potential hacks or thefts.
