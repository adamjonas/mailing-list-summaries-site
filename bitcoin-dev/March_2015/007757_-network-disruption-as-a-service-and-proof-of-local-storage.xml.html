Summary:

The email exchange between Robert McKay and Matt Whitlock on 27th and 28th March, 2015 discusses the issue of a single entity running multiple nodes to harvest transaction origin IPs. To address this problem, onion routing is proposed as a solution where transactions would have an additional layer of encryption to prevent IP discovery. However, this would cause overhead and slow down transaction relay slightly, so it would be made an optional feature for those who are concerned about IP association. In response to Matt's suggestion that hosting hundreds of lightweight servers backed by a single copy of the blockchain could be more efficient, Robert points out that someone could set up a single full node with the blockchain and proxy any challenges to it from other non-full nodes. The email thread further discusses the challenge-response protocol to ensure local possession, which relies on the lack of a "partial getdata" command in the Bitcoin protocol. The suggested protocols involve sending encrypted blocks and Merkle tree root of hashes of encrypted blocks with N indexes selected by a pseudo-random function seeded by a challenge value. It is noted that not every node needs to implement the protocol, but only nodes that want to prove full-node-ness, such as those that want to receive bitnodes subsidy.
