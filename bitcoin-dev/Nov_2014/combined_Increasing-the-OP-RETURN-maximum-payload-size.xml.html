Summary:

The Bitcoin community has been discussing the use of OP_RETURN in bitcoin transactions, which allows small amounts of data to be stored within the blockchain. Currently, the limit for OP_RETURN is set at 40 bytes, but there are proposals to increase it to 80 bytes or more. While some developers express concern that this could encourage using the blockchain as a convenient transport channel, others argue that it is necessary for legitimate uses such as storing URLs and signatures.Counterparty currently uses multi-sig outputs instead of OP_RETURN encoding to store more than 40 bytes, and Open Assets needs to store a URL in the OP_RETURN output plus some bytes of overhead, which 40 bytes does not allow for. Flavien Charlon proposed increasing the limit of data that can be embedded as part of an OP_RETURN output from 40 bytes to 80 bytes in November 2014, which would have a negligible impact on bandwidth and storage requirements while being extremely useful for many use cases where a hash only is not enough. However, Pieter Wuille expressed concerns that larger OP_RETURN sizes could encourage people to use the blockchain as a convenient transport channel rather than just for necessary data validation.Adam Back voiced concerns about the potential loss of benefits from hierarchical deterministic (HD) wallets if out-of-band communication is used instead of OP_RETURN. He concludes that OP_RETURN is the best alternative for this use case. In addition to the OP_RETURN size issue, there are concerns about privacy loss when using wallets with functionality beyond maintaining a balance and sending coins.It was suggested that storing only a hash is enough for all timestamping applications while storing a URL in the OP_RETURN output with some bytes of overhead is insufficient for Open Assets. The use of blockchain as a backup mechanism was also discussed, and it was concluded that it is not an efficient or reliable backup method due to its broadcast nature. Instead, cloud storage services or disk encryption with related keys were suggested for decentralized long-term storage.There is also a call for discussion and agreement on a plan for increasing the limit, with the proposal being the second-best option. The first proposal is also considered good as long as the "space budget" is shared between the two outputs. The discussion of "best defaults" including default size for OP_RETURN, minimum fees, anti-dust policies, first-seen vs replace-by-fee, etc. ultimately depends on miners and their configuration policies. However, many miners and pools are apathetic towards configuring their own policy. The solution is to make it easier for miners to implement their own policies by abstracting out those parts of the code.Pull requests like #5071 and #5114 are steps towards making it easier for miners to implement their own policies and increase the acceptance of 80 bytes OP_RETURN transactions among them. Separating script/standard was also a little step in the same direction. Despite concerns about using the blockchain as a convenient transport channel, the decision ultimately depends on miners and their configuration policies. It is important to keep the data design-compressed and minimal for scalability purposes, and other reliable channels are available for backup and transport purposes.
