Summary:

The Bitcoin community has been engaged in several discussions on various aspects of the technology. One such discussion is around proposed changes to the verification equation for Bitcoin and the use of elliptic curve Schnorr signatures. Pieter Wuille has proposed a draft specification for standardizing the signature scheme, which is a first step towards integrating it into Bitcoin. However, there have been discrepancies between the specification and test vectors. Some contributors have suggested alternative equations for the verification process, with some proposing changes to avoid negating coefficients or points.There have also been discussions around the security advantages of using redistributable threshold systems and the use of delinearization in cases where it can be used. There are proposals for creating M-of-N threshold MuSig signatures for any M and N. Discussions have also taken place around the use of delinearization in threshold schemes and the importance of communication efficiency. However, concerns were raised over the loss of a key or aborting participant leading to the failure of the whole process in threshold MuSig signatures.Another topic of discussion is the proposed M-of-N Bitcoin multisig solution by Erik Aronesty, which has been found to be less secure than existing solutions. Andrew Poelstra responded to discussions around building up threshold signatures via concatenation in Bitcoin and provided feedback on Pieter Wuille's proposed BIP for 64-byte elliptic curve Schnorr signatures. In addition, discussions have taken place on the optimal order of inputs for hashing in the context of ECDSA signatures. The argument in favor of nonce first is that it provides a minor security benefit, but others argue that message first follows the standard MD-style hash function design. Another proposal discussed is to add 31 bytes of zeros before the public key in the hash input but is deemed too burdensome for the marginal gain.Regarding Musig, it is considered prone to loss as it is M of N, which means losing or compromising a single key can lead to lost coins and make the system unusable. However, it is possible to create M-of-N threshold MuSig signatures for any M and N with zero security assumptions. Gregory Maxwell clarified that there is no proposal for a "non-redistributable multisig" in Bitcoin, and that Musig as an M of M system is inherently susceptible to loss.There have also been discussions around the use of delinearization for communication efficiency, with both Erik Aronesty and Gregory Maxwell agreeing on its importance in certain applications. It was emphasized that M of M is a specific threshold, and other thresholds can be used if desired. For users who want an "M of N" scheme that doesn't cost more to send funds, allows them to lose a device and keep their coins, and allows them to establish and validate the scheme safely, a simple "verified signer" threshold scheme is probably the best solution.
