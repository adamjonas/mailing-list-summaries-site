Summary:

Bitcoin developers have been discussing the implementation of CheckLockTimeVerify (CLTV) opcode in the Bitcoin protocol. One of the main concerns in the discussion is the use of NOPs and their scarcity in the code. Some developers suggest using the original unparameterized CLTV version, while others argue that adding a parameter to OP_CLTV would make it more flexible and is the most economic use of precious NOPs.Peter Todd suggests that adding a parameter to OP_CLTV would be the most economical use of NOPs. He presents how it could be implemented and acknowledges that this change would require some work to update all the existing tests and example codes and to update the BIP, but believes that it's not too big of a deal. Btc Drak agrees with this proposed change, and suggests that it should be made in time for the feature freeze.Luke Dashjr suggested that it would be straightforward to implement a softfork of RCLTV as a negative CLTV, which means using negative numbers as relative later. This approach makes handling the year 2038 problem more complex, but it would make handling nLockTime values easier. Matt Corallo raised concern over the scarcity of OP_NOP in Bitcoin mempool regarding only CLTV pull-req. He suggested that using one of the precious few OP_NOPs would be more flexible and efficient.The conversation ends with a link to a Bitcoin-development mailing list. Overall, the discussions revolve around finding the most efficient and flexible way to implement CLTV opcode while also addressing the issue of OP_NOP scarcity. The proposed changes require some extra work, but are deemed acceptable and should be made in time for the feature freeze.
