Summary:

In 2015, a Bitcoin Improvement Proposal (BIP) was drafted to enable deterministic pay-to-script-hash multi-signature addresses through public key sorting. The BIP proposes a standard method to generate multi-signature transaction scripts and defines how the public keys must be encoded and sorted to ensure that the redeem script and corresponding P2SH address are always the same for a given set of keys and number of required signatures. This will simplify the derivation of scripts and their addresses by all parties and enable services to declare themselves as BIP compliant, working towards interoperability of services.A discussion thread on the Bitcoin-development mailing list in February 2015 discussed the lack of a specification section in a script template format created by William Swanson and Eric Lombrozo. Luke Dashjr inquired whether the format supported arbitrary scripts or only the simplest CHECKMULTISIG case. Peter Todd suggested that the BIP be rewritten to state that all pubkeys executed by all CHECKMULTISIG opcodes would be in canonical order, with some examples to illustrate this rule. However, Todd expressed concern that the rule might be too restrictive for some cases and should not be enforced as an IsStandard() rule.The email thread discussed modifications that need to be made for a document to be merged. The BIP being discussed is only meant to derive the type of scripts encountered while doing addmultisigaddress. More complicated scripts would require more metadata to be shared, but the only case described is when given public keys and the number of signatures required. The email thread discusses the possibility of enforcing a convention for people to follow regarding canonical ordering of pubkeys executed by all CHECKMULTISIG opcodes. However, this convention should not be brought into validation rules, just a convention for people to follow for now. It is suggested that there is good reason for wallets to opt in to the convention and declare this, for ease of recovery and for interoperability reasons.In addition to the BIP, the context also includes a PGP signature for cryptographic authentication, an HTML attachment that has been scrubbed, meaning it has been cleaned or deleted for some reason, and two non-text attachments in the form of PGP keys and signatures.
