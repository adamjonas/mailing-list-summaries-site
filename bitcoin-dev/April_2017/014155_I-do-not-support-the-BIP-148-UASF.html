Summary:

Chris suggests that instead of rejecting all blocks that do not signal segwit, only those that do not signal but contain segwit transactions should be rejected. He questions the reasoning behind rejecting a block that only has pre-segwit transactions. Praxeology Guy responds by stating that he has requested an alternative to avoid unnecessarily orphaning blocks and agrees with Chris's suggestion. In response to a question about old blocks containing SegWit transactions, it is explained that if a transaction tries to spend a new SegWit format txo in an old block, protocol rules would be broken, particularly for SegWit activated nodes. Old nodes do not have code that can spend SegWit format txos, which may lead to a fork. The assumption of 148 is that irrespective of what happens, there will be a fork, so it is better to do it in a controlled manner before someone creates an invalid SegWit block. It is recommended to implement a boilerplate replay attack prevention just in case the SegWit delayers are not bluffing.
