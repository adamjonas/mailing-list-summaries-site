Summary:

The bitcoin-dev mailing list has been discussing the need for a transaction size limit and how to address the quadratic hashing problem. The existing code at line 1661 in validation.cpp addresses the issue, but some members of the community have proposed adding a transaction-size limit of 10kb or 100kb. Others have suggested making transactions hard enough to confirm that no one will use them without a very good reason by mining any block containing an outsized transaction at a greater difficulty.Luke Dashjr shared a draft BIP proposing a real 2 MB block size hardfork following Segwit BIP148 activation. However, there are concerns that this blunt-style hardfork is risky and requires consensus from the entire Bitcoin community. The proposal includes a block size limit of 2,000,000 bytes being enforced upon activation, while the block weight limit remains at 4,000,000 WU. All witness data is measured as 1 weight-unit (WU), while all other data in the block is measured as 4 WU. The witness commitment in the generation transaction is no longer required, and instead, the txid merkle root in the block header is replaced with a hash of the witness reserved value, the witness merkle root hash, and the transaction ID merkle root hash. The maximum size of a transaction stripped of witness data is limited to 1 MB.In response to the proposal, some members of the community questioned why the redundant 1 MB size limit check could not simply be removed, and the weight limit increased without changing the discount or having two limits. Others have suggested using BIP103 for timing and absorbing up to 20% per year in averages in bandwidth/CPU/RAM growth. There were also proposals to initially set the limit to 1MB and then drop it to 500kb or 100kb at a later block height. However, changing the rules could invalidate pre-signed transactions currently in existence.Overall, the discussions highlight the complex nature of proposing changes to Bitcoin's protocol and the need for consensus among the community. The quadratic hashing problem is not a compelling reason to keep blockmaxsize post-HF, and alternative solutions should be explored.
