Summary:

Several discussions have taken place regarding the implementation of certain opcodes in Bitcoin. In 2010, a bug was found in Core software that allowed denial-of-service attacks due to script execution causing crashes on certain machines. The user believes that disabled opcodes should be reintroduced with standardized behavior definitions, allowing for more sophisticated native smart contracts to be created. Re-enabling old opcodes requires a hard fork, but if SegWit is enabled, new opcodes can be allocated via a soft fork by introducing a new version of Script. The Elements alpha project has experimented with re-enabling old opcodes. One of the requested opcodes is OP_CAT, which concatenates hash outputs together and would significantly reduce the size of transactions for offchain Tumblebit transactions. With OP_CAT, only one hash would need to be stored in the transaction instead of multiple hashes. The author suggests a maximum output size of 512 Bytes and a streaming fashion to avoid exceeding memory usage. The proposed opcode would allow for the secure and compact verification of many hashes and hash preimages.In an email conversation between Ethan Heilman and Peter Todd, Ethan proposes the use of OP_CAT to securely verify many hashes and hash preimages compactly. A version is also suggested which improves upon this but is more complex. Peter asks about the maximum output size and suggests that Ethan's proposal would make a good use-case for a BIP.Overall, the implementation of specific opcodes could enhance the flexibility of scripts and allow for the creation of advanced smart contracts. However, it requires careful consideration and potential changes to the current Bitcoin system. The user is requesting the reintroduction of the OP_CAT instruction, which concatenates vectors and hashes them, with a limit on maximum output size of 512 Bytes. This would enable secure and compact verification of many hashes and hash preimages, reducing the size of offchain Tumblebit transactions significantly.The user believes that most of the math OP codes in Bitcoin aren't particularly helpful due to their 32-bit nature and strange overflow behavior. In addition, the user suggests that disabled opcodes should be reintroduced with standardized behavior definitions. For instance, when executing an opcode that results in an arithmetic error or incorrect arguments, such as the string splice opcodes, the script should exit and fail. Mark Boldyrev, a member of the Bitcoin development mailing list, supports the reintroduction of disabled opcodes to create advanced smart contracts. He cites the example of CHECKSEQUENCEVERIFY and his own CHECKLOCKTIMEVERIFY bip as potential use cases for these opcodes. Boldyrev suggests that the reintroduction of these opcodes should come with standardized behavior definitions.
