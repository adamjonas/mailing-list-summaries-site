Summary:

The distributed algorithm can be changed to make previously valid transactions invalid without causing chain splits. However, this could create issues if invalid transactions are emitted after the block height where the rule takes effect and a super-majority of miners have not yet upgraded. To avoid this problem, upgraded nodes must follow specific rules: never forward or mine a transaction that would be invalid under the new rule, apply old behavior before height X unconditionally (X set far enough in the future for large miners to deploy), and begin applying the new rule only after the first point in the chain after X when none of the last Y blocks have contained an invalid transaction under the new rules. After the software is released, members of the bitcoin community intentionally transmit transactions that are invalid under the new rules. By setting Y high enough for all major miners to have a chance to mine in the window, this becomes an effective vote for the change by miners with a stochastic super-majority threshold. All nodes can determine exactly at what block the election has completed since it is an objective fact of the chain. With this scheme, the new encoding will only become active when mining capacity supports it so that a large reorganization does not happen due to incompatible blocks during deployment.As a safeguard, before applying the new rule, 50% of the last Y blocks must contain a coinbase with an "I am upgraded" code. Until the new rule is active, an "I am upgraded" code should be included in every block; after activation, this can be turned off. The active statement provides a proof of how likely a fork of length X will be if a newly invalid transaction is announced at the time of activation. This scheme is sufficient without conflicting block discouragement and is generally superior to discouragement for this purpose.
