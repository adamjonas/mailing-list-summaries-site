Summary:

The discussion revolves around the implementation of 'green addresses' in the Bitcoin network, which are considered to be controversial. One proposed idea is to have a trust aggregator sign transactions, and merchants can create a transaction that is uploaded to the authority for verification and then signed for broadcast. The author's first approach was to extend the `gettransaction` call to include an extra field "inputaddresses" that would return a list of the Bitcoin addresses associated with the inputs of the transaction. However, this is not generally possible because of the different possible structures enabled through the scripting language. Gavin Andresen suggested that green addresses could be implemented as a second signature in the scriptSig, which would allow the rest of the transaction to be completely 'normal' and not require it to use specific inputs.One suggestion is to implement green addresses as a second signature in the scriptSig, but it violates the design principle that a feature should not cost anything if it is not being used. A possible solution is to hack the Bitcoin client by generating a transaction with the green address key signature instead of using specific inputs. Another approach is to mark certain addresses as trusted and have an RPC call to request all incoming transactions from trusted sources. However, there were concerns about the safety of these methods and their potential impact on performance.In an email exchange between Jan Vornberger and Pieter Wuille, Vornberger proposed extending the `gettransaction` call to include a new field called "inputaddresses," which would return a list of Bitcoin addresses associated with the inputs of the transaction. However, Wuille pointed out that Bitcoin transactions do not have input addresses but rather have optional addresses the input coins were last sent to. He also expressed concern that allowing for a 'from' address on a transaction could encourage using such addresses to identify transactions, which he believed was the wrong approach. In response to Vornberger's green transactions idea, Wuille suggested perhaps creating an interface to mark certain addresses as 'trusted,' and having an RPC call to request all incoming transactions that originate from trusted sources.The email exchange discusses the idea of green transactions and how they could be implemented in Bitcoin. One suggestion is to create a peer-sourced green address implementation where users can individually decide to trust certain addresses as "green" and publish this trust. The discussion also touches on issues with identifying input addresses and implementing an interface to mark certain addresses as 'trusted'. Overall, the exchange provides insight into ongoing development efforts for Bitcoin.There were criticisms ranging from 'unnecessary, as 0-confirmation transactions are fairly safe today' to 'encourages too much centralization and therefore evil.' Additionally, there are concerns about non-standard scriptPubKey scripts that could validate if given a green address as input, and the need for caution in implementing any new features that could potentially encourage centralization or identify transactions.
