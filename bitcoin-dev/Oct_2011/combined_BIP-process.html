Summary:

The Bitcoin community has been discussing the management of changes to the Bitcoin Improvement Proposals (BIPs) repository on GitHub and potential mailing list options. One proposal suggests using wiki pages for BIPs, with the author having the final editing authority, while another proposes a separate mailing list for BIP discussion. However, some members argue that splitting the list would dilute the peer review process and believe that there should be more discussion on the current list.In terms of mailing list options, some members suggest using Google Groups or SourceForge, but concerns are raised about proprietary features and Gmail lock-in. The possibility of using Discourse for BIP discussion is also brought up, with some members liking the feel of it. Meanwhile, there are discussions about problems with Yahoo mail and using mail-archive.com. Wladimir proposed creating a more formal process for managing changes to the BIPs repository on GitHub. He suggested making the author or someone they delegate to be the primary contact for each BIP, responsible for commenting on changes and accepting or rejecting them. However, authors would need to adhere to BIP 1 guidelines, such as not allowing technical changes to final BIPs without community consensus. Wladimir also proposed moving the BIP discussion to a separate mailing list to reduce the workload on the bitcoin-development list, which currently serves both as a forum for Bitcoin Core implementation concerns and global changes to Bitcoin in the form of BIPs. The Bitcoin development mailing list is considering replacing their current mailing software, lists.sf.net, which is incompatible with Yahoo’s “mail signatures must be valid” policy and has poor moderation and mail archiving features. Members have suggested using open source list software such as Mailman or other simple open list software, with a preference against Google Groups due to its proprietary nature and lock-in with Gmail. Overall, the discussions aim to improve the decision-making process for BIPs, distribute responsibility, and reduce the workload on the bitcoin-development list while retaining the quality of peer review. There are also suggestions for improving workflow and avoiding issues with multistage workflows and people failing to comment on BIPs.
