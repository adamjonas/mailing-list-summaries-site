Summary:

Mike Hearn, a former Bitcoin developer, raises concerns about the poor space efficiency and inter-version compatibility problems of Berkeley DB (BDB) files for read only pruned indexes. He also notes that even if a more complex scheme is used, commitments must be in the blockchain, and somebody has to verify that binaries match the source code to prevent software from doing anything malicious. The binaries distributed by bitcoin.org are already compiled deterministically and validated by multiple independent parties, but there will be a downloader tool in the future to check for approvals before accepting an update. This produces a full chain of custody to track actual binaries to specific source code, which can be audited, making substitution attacks detectable. Mike Hearn does not favor unauditable single source material because it greatly increases the attractiveness of compromising developers. If shipping pruned chains was wanted, he would prefer to have a deterministic process to produce archival chains and introduce commitments to them in the blockchain. Then, a client doing a reverse header sync could stop syncing and use the archival chain for points before that. This leaves the distribution of software auditable, and something should be done with service announcements so that full nodes that do not have enough bandwidth can still support syncing without turning off listening.
