In 2012, various discussions related to Bitcoin development were held. One of the discussions focused on improving the performance and efficiency of Bitcoin's block propagation process, which included proposals such as adding the mempool command, changing the "block" message, and introducing getmerkletx to filter transaction hashes. The proposed changes aimed to reduce orphan blocks and spurious chain splits. Additionally, there were proposals for improvement in the format of distributing blocks in the blockchain network, including using 'invgetdatamerkleblock' instead of 'invgetdatablock', which would optimize bandwidth.Another discussion centered around the use of Bloom filters in Bitcoin. While some argued that separating the protocols allows for more flexibility and negotiation between SPV clients and full nodes, others believed that overloaded nodes should offer 50% false positive rate filters as an option instead of disconnecting SPV clients if there aren't many full nodes available. There were also suggestions to combine filterinit and filterload commands, use exact matches in addition to Bloom filters, and optimize query flexibility by using a binary tree data structure of unspent outputs arranged by the Bloom filter bits.Furthermore, discussions related to proposed new P2P commands, such as filterinit, filterload, filterclear, filteradd, and mempool, which enable special behavior changes for remote node's memory pool, were also held. However, concerns were expressed regarding Bloom Filters, with Andreas Petersson proposing a scheme where the client takes a list of addresses they are interested in and hashes them to a fixed-length bit array.The discussions also covered the additional needs of lightweight clients and proposed new P2P commands with associated behavior changes. While ultra-lightweight clients have a fundamentally different security model than SPV clients and cannot connect directly to the P2P network, regardless of added commands or database indexes, the proposed functions could help address their specific needs. Overall, the discussions revolved around optimizing blockchain technology, from reducing server-side load to improving block propagation and reducing wasted work for miners.
