Summary:

Discussions have been held on the best data structure to use for Bitcoin. There was a debate between a raw trie structure or a PATRICIA tree/trie. While a trie has constant time access to elements and deterministic history-independent structure, disk space can be affected by a malicious actor inducing more branching. A PATRICIA tree/trie, while more space-efficient and completely deterministic, is not a simple implementation and may require good specifications and tests to prevent issues. The use of a binary tree could also lead to implementation errors, making it important to choose a standard and write test cases.The conversation also touched on the use of Merkle trees for replacing unneeded transactions in Bitcoin and the importance of balancing binary search trees to prevent unbalanced growth. Recursive data structures can be 'Merkle-ized' by adding a hash of the child node alongside each link/pointer, allowing verification of the data for each node as one traverses the structure. Storage burden can be delegated to an untrusted server, with a lite-client requesting only relevant data.Other discussions included commitments to the state of unspent transactions and the need for a new data structure to manage trees of unspent transactions in a secure, scalable, and DOS resistant manner. A PATRICIA tree/trie has been suggested as an ideal option due to its deterministic structure and space efficiency. However, implementing it is not trivial as it has worst-case O(k) complexity, where k is the number of bits in the key. Furthermore, a trie of any sort is dependent on the distribution of input data for balancing, which could be exploited by a malicious actor under particular timing-sensitive circumstances.In a conversation between Mike Koss and an unknown individual, they discussed commitments to the state of unspent transactions that would enable memoryless nodes to fully validate without trust and securely initializing new pruned nodes without exposing them to the old history of the chain. Adopting a new data structure for managing trees of unspent transactions is necessary for these ideas to be possible without degrading the full node security model significantly. There are many possibilities for this, and it may require adopting a more complex data-structure than the merkle tree for inclusion in the bitcoin protocol in the future.
