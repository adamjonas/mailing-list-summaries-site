Summary:

Bitcoin is facing problems with slow block verification and propagation processes that are CPU and IO intensive. One proposed solution is to make existing nodes faster, which would buy more time before multi-machine nodes become necessary. LevelDB has been introduced as a new database option and has been tested against Regular BDB on a 6 core Ubuntu machine, with results showing that LevelDB is nearly 4x faster when signature checks are not done. Gavin Andresen discusses whether or not to switch to LevelDB as a replacement for bdb, and tweaks to the database schema during migration are also discussed.In another conversation, Gregory Maxwell suggests that ECDSA caching can easily run on multiple cores, providing a linear speedup. Furthermore, discussions have been held on optimizing IO load and threading transmission verification to address block propagation issues. The author of a GitHub repository has added more commits to the codebase, including auto-migration and migration progress UI. Although LevelDB is a fantastic solution for desktop software with multiple use cases, it may become less optimal for Bitcoin over time as SSDs become more prevalent. Additionally, there are concerns about LevelDB's newness, minimalist build system, portability, testing, and compatibility guarantees.Overall, LevelDB works well for the Bitcoin use case anyway, thanks to the LSM trees. However, more testing is needed on LevelDB before it can be fully implemented, and the author of a GitHub repository is asking for help with testing and makefile work. CTxDB::ReadOwnerTxes has never been used in the codebase, and the author plans to delete the code when they next work on the patch.As Bitcoin grows, there is a need to scale the nodes, and eventually, there may be multi-machine nodes. While LevelDB is a great choice for desktop software with multiple use cases, it isn't ideal for Bitcoin. Nonetheless, it works well for the Bitcoin use case anyway, and most Bitcoin nodes on servers will continue to be HDD based for the foreseeable future.
