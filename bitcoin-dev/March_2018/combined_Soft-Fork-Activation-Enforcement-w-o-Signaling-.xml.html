Summary:

The bitcoin-dev mailing list had a discussion about activating soft forks, such as BIP65 and BIP66, without prior signaling from miners. The author of the email thread was working on a project to implement Segwit and other key features introduced by Bitcoin, and wanted to know if they could skip over signaling and go straight into activation/enforcement if the entire network upgrades to the correct version of the software based on Bitcoin 0.15.An intelligent assistant responded that it is possible to activate softforks at a given height, and recommended reading bip34 to understand what it activates in the code. The last thing that was activated was bip65. The assistant also mentioned that activating the soft forks without any signaling beforehand would segregate the 0.15 clients onto their own fork, which must be understood before proceeding.Samad Sajanlal, another participant in the email thread, asked whether the block version gets bumped up automatically at the time that a soft fork activates or if there is additional work needed within the code to ensure it bumps up at the same time. From what Samad saw in the code, it appears that it will bump up automatically, but he wanted confirmation on that.Jorge Timon confirmed that it is possible to activate softforks at a given height and suggested upgrading directly to version 0.16. He also mentioned that block version bumping was a mistake in bip34 and that there is no need to bump the version number.Overall, the email thread provides insight into the technical aspects of implementing soft forks in a Bitcoin codebase. It is possible to activate softforks at a given height, and it appears that the block version will bump up automatically at the time that a soft fork activates. However, the repercussions of activating the soft forks without any signaling beforehand must be understood, as it will segregate the 0.15 clients onto their own fork.
