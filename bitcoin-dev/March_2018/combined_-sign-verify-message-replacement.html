Summary:

In a Bitcoin-dev mailing list, several proposals were made to address issues with message signing tools for legacy 1xx addresses. Karl Johan Alm proposed writing a replacement tool, while Pieter Wuille suggested a script-based approach using a new structure called SignatureProof. This structure would be passed between the signer and verifier using two RPC commands: sign and verify. The proposal also includes adding "proof of funds" to the signature format, which currently does not actually prove funds at all. Luke Dashjr suggested a new signature format that would include "proof of funds" as well as additional features such as avoiding disclosing public keys for future UTXOs.Meanwhile, Anthony Towns proposed that old nodes should check for standardness of the spending script and report nonstandard scripts as either invalid outright or highly questionable to prevent confusion. A warning would also be useful in case the verifier is running old software.The conversation also discusses the handling of scripts with OP_CLTV and OP_CSV by verifiers, with concerns about whether they always succeed or if an nLockTime and nSequence should be included in the proof that can be parsed out and displayed to verifiers. It is suggested that any signatures in the scriptSig/witness data would have no sighash type.Luke-Jr suggests a new signature format that supports both "proof I receive at this address" and "proof of funds" for multiple inputs, while avoiding disclosure of public keys for existing or future UTXOs. He also warns about using prehashed messages for message signing functionality and proposes forcibly prefixing "Bitcoin signed message" to avoid unintentionally signing something.Pieter Wuille suggests adding a version number in the signature that corresponds to a set of validation flags, or using a double verify approach with consensus and standardness rules. He also suggests extending the existing signmessage/verifymessage RPC and warns against using prehashed messages.Overall, these proposals aim to improve the security and functionality of Bitcoin's message signing tools by addressing issues such as proof of funds and avoiding disclosure of public keys for future UTXOs. The addition of a new structure called SignatureProof and improvements to standardness checking could help prevent confusion and improve verification processes.In terms of proving present possession of funds without sacrificing fungibility or hot/cold wallet separation, one proposal suggests using a FORKID in a transaction, allowing for a mempool acceptance test that returns true even if the signature is not valid as per Bitcoin consensus, but only due to the FORKID. Another suggestion is to include time conditions under which the funds may be spent. A soft fork can make old nodes accept invalid message signatures as valid, so it was suggested that old nodes should check for standardness of spending scripts and report non-standard scripts as either invalid outright or highly questionable. Feedback is welcome on the proposed SignatureProof system. The participants suggest assuming sufficient time/height on CLTV/CSV checks when there's no input. They also mention including an optional nLockTime in the proof container which defaults to 0, and nSequence that defaults to 0xFFFF. The default SIGHASH_ALL is suggested for simplicity unless there's a good reason to tweak it. Finally, it is recommended to append a byte to the end of the signature to pass encoding checks, but it could be a 0x00 byte as it's not actually a sighash flag.
