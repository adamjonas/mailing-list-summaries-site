Summary:

There have been recent discussions on the bitcoin-dev mailing list about BIP 117, a proposal for improving the Segregated Witness (SegWit) feature in Bitcoin. Some concerns have been raised about its viability as a soft fork for Script due to the dropping of SIGOP and opcode limits, which has led to questions about its flexibility and whether it is doing too much.The use of altstack has also been found to be awkward, and there are concerns about the potential impact on existing commitments and users' funds, with the possibility of rendering them unspendable and retroactively removing ownership. Rusty Russell has expressed his concern regarding this situation, stating that there are no restrictions on terminating a v0 witness program with a non-empty alt-stack and leaving non-canonical boolean values on the main stack.Russell suggests a simpler implementation of tail recursion and proposes restricting this for a specific tx version to gain experience first. He also calls for more justification for the dropping of SIGOP and opcode limits in BIP 117, particularly measurements and bounds on execution times.In addition, Rusty suggests future improvements to switch to a witness script version and a new segwit output version which supports native MAST to save additional witness bytes. He also advocates for migrating to a model of limiting sigop per weight and counting the actual number of sigops during execution.The article argues that new bitcoin consensus features should follow the UNIX philosophy of being small, modular, incremental, and reusable changes that can be combined and used in innovative ways. This approach would create a platform for unrestricted innovation. The author uses the example of Merklized Abstract Syntax Trees (MAST) to support this argument and notes that there is currently no production use of MAST. Therefore, it is premature to provide a templated, special-cased MAST before it sees wide use. Instead, the proper sequence of events should be to enable features in a generic way and then create specialized templates to save space for common constructions.In another discussion on the bitcoin-dev mailing list, Rusty Russell stated that "standard transactions must still work" and that this principle was violated with low-S. However, he argues that the transformation was arguably trivial. Luke Dashjr expressed his concerns about BIP 117 and its potential impact on existing commitments and user's funds. He argued that this change cannot be compared to previous changes in only the standardness rules. While standard transactions must still work, changes to standardness rules can still cause difficulties for retrieving funds.
