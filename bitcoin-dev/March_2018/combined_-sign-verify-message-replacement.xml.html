Summary:

A proposal to replace the broken message signing tools for all but legacy (1XX) addresses using a script based approach has been suggested. A new structure - SignatureProof, which is a simple scriptSig & witnessProgram container that can be serialized has also been proposed. However, more logic is needed to deal with soft forks and compatibility regarding the verification of fixed, consensus or standardness flags. One solution would be to include a version number in the signature that corresponds to a set of validation flags, allowing inconclusive reports when versions are unknown. RPC commands, sign and verify have been suggested to generate and execute signature proofs respectively. Additional feedback on the proposal is welcome. The danger of using prehashed messages was also highlighted along with the potential for soft forks to make old nodes accept invalid message signatures as valid. It was suggested that old nodes check for the standardness of spending scripts and report non-standard scripts as invalid or highly questionable to prevent confusion.
