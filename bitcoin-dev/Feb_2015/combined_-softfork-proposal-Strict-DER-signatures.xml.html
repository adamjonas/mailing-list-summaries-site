Summary:

The discussion revolves around setting upper bounds on R/S length and incorporating it into BIP66's DERSIG. This change is crucial for ensuring robustness to implementation differences, as allowing overlarge R/S values can have a significant impact. Initially, the function was created for Bitcoin Core v0.8.0 to enforce non-standardness, but not consensus. Therefore, there was no need to impose a maximum length for R and S arguments, as overly-long values would always result in integers >= 2^256, making the encoded signature invalid according to the ECDSA specification.However, in the proposed soft fork, non-DER results in immediate script failure, which is distinguishable from invalid signatures. Adding individual R and S length restrictions would be possible but deviates slightly from the standardness rule implementation that has been deployed for a while. Two options are under consideration: adding this R/S length restriction rule as a standardness requirement or including it in the softfork now. To deploy this change, they plan to reuse the double-threshold switchover mechanism from BIP 34.Pieter Wuille originally suggested a soft fork to make non-DER signatures illegal and proposed incorporating it into BIP66's DERSIG. However, after receiving feedback from implementers such as Suhar Daftuas, he retracted the proposal and expanded comments on the IsValidSignatureEncoding function. It was suggested that only allowing 0 as an invalid signature would align signature validity with decoding. Pieter requested a BIP number for this proposal. The code has been updated to handle the sigsize=0 case elsewhere, and a comment has been added in the BIP text. There was also discussion about edge cases that are not covered and the possibility of a cycle of making this non-standard before a further release enforces it.The proposal for a softfork to make non-DER signatures illegal is seen as a solution to the risk of depending on OpenSSL for consensus rules. Recent events have highlighted the need to address this problem soon. An implementation, including unit tests derived from the BIP text, can be found, which includes motivation and specification. Four implementations now exist, with reduced test code in versions 0.8 and 0.9, but identical implementation code. Work to improve this is welcome. The proposed BIP number is BIP0066. Minor changes were suggested, and some missing comments were fixed in the original document.
