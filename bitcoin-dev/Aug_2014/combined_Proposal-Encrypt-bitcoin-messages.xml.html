Summary:

In a Bitcoin-development mailing list, the topic of encrypted communication between nodes was discussed. The conversation revolves around the importance of secure communication protocols in Bitcoin Core and the potential solutions to achieve this security. While there are various opinions on the usefulness of encrypting communications between nodes in the Bitcoin network, many developers agree that anonymity and privacy are important for users. Some suggest using Tor, Freenet, or other anonymous P2P networks to help achieve anonymity. However, there are concerns about the attack surface this introduces, and some argue that encryption may not be effective in improving network security.One suggestion is to use public and private key pairs for encryption, while others raise concerns about the vulnerabilities of TLS and the attack surface it creates. It was suggested to use ECDH for a symmetric key and transport using a block cipher as a solution. Additionally, there are discussions about the limitations of encrypting non-sensitive inter-node traffic in providing anonymity. However, some argue that encryption is necessary to prevent passive attackers from tracing the origin of Bitcoin transactions. Despite differing opinions on the value of encryption, the email threads emphasize the importance of online privacy and the need to support it whenever possible.Gregory Maxwell responds stating that if your threat model is passive listeners, establishing a symmetric key for each connection using Diffie-Hellman at handshake time is all you need, no public-private crypto needed at all. He goes further to say that those users that do have a need for security should probably connect to the network using Tor or I2P which can give much better security guarantees than anything being discussed here. It was suggested that perfect forward secrecy between all nodes by default would increase the cost for any attackers attempting a MITM attack on a large scale. There were discussions around the use of public and private key pairs for encryption between peers, with concerns raised about the security of sharing public keys. The conversation also discusses the benefits and drawbacks of encrypting all Bitcoin transaction communications, with one person suggesting that all communications should be encrypted by default, except for block packets to avoid increasing propagation time. Another person questions how each node would make use of the blockchain in an encrypted form if all messages were encrypted with an asymmetric cipher.The email also discusses the potential vulnerabilities of TLS and the complexity of auditing its code. One member of the discussion stated that he could see the value of having authenticated traffic with specific nodes using an HMAC for the protocol messages in place of the current checksum. The concern raised was that if traffic in and out of a single node is monitored, it becomes possible to determine which transactions originate from it versus those which it relays. However, it was concluded that the risk/reward payoff of encrypted communications between nodes did not seem worthwhile. The suggestion to use TLS was made, but the responder notes that avoiding its use has prevented emergency scale vulnerabilities. Overall, it was noted that recent serious OpenSSL bugs should be taken into consideration as well.
