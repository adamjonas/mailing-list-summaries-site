Summary:

In 2014, discussions took place among Bitcoin Core developers about the use of GitHub for hosting the project's issues and git repository. Jeff Garzik expressed concern about its centralized nature and suggested moving away from it. However, other developers argued that GitHub had benefits such as allowing contributions from a large number of developers while ensuring the integrity of Bitcoin's code through GnuPG signed commits.There were also suggestions for secure alternatives to GitHub, such as Gerrit and JetBrains' upcoming tool, Upsource. Discussions also took place on the use of Git's rebase functionality and the need for multiple redundant "master" repositories run by different people in different jurisdictions to ensure operational security.While there was a desire for secure alternatives to GitHub for Bitcoin's sustainability, there was also recognition of the convenience and benefits of using it. Developers emphasized the importance of keeping it simple and ensuring digital signatures and cryptographic hashes protect the integrity of software.Wladimir opposed the idea of switching to a self-hosted platform, stating that it would make it difficult for people to contribute and would require them to handle sysadmin work themselves, which wouldn't be a good use of limited manpower available. Moreover, migrating all the current issues and pulls would be a lot of work, which he doesn't see the point of doing. Despite some concerns over GitHub's centralized nature, there are still over 4,000 forks of Bitcoin on the site, providing a large potential contributor base.Some developers proposed implementing multiple redundant 'master' repositories run by different people in different jurisdictions, to maintain Git history integrity and avoid approving unsigned changesets. Signed commits and timestamps have also been recommended as protection against history rewriting. JetBrains is developing Upsource, a code review and repository exploration tool designed for self-hosting, which could be an alternative to GitHub's issue tracker. However, some argue that abandoning GitHub would make it difficult for people to contribute and handle sysadmin work themselves.Overall, while decentralization is desirable in theory, practical considerations such as ease of contribution and limited resources must also be taken into account.In 2014, Pieter Wuille suggested using Github as an easy-access mechanism to contribute and inspect changes made to master would be beneficial due to Git's ability to generate current commit hashes using the previous hash, making tampering noticeable. A formal program of code review, perhaps on a per-release basis, was recommended to ensure operational security, with multiple redundant "master" repositories run by different people in different jurisdictions.An email exchange between Jeff Garzik, Bryan Bishop, Pieter Wuille, and xor discussed the need for alternative solutions to centralized services like Github, highlighting the potential danger of code injection by someone with root access on a Github server. Furthermore, a discussion between Peter Todd and Troy Benjegerdes highlighted the benefits of Git's rebase functionality for communicating changes to developers and the need for signed commits and timestamps to protect against history rewriting. Todd suggested using Bitcoin transactions for signatures and the need for multiple redundant "master" repositories run by different people in different jurisdictions to ensure operational security. Benjegerdes suggested getting paid for reviewing code rather than volunteering and questioned the advantage of PGP, suggesting more people have ECDSA public-private keys than PGP keys.
