Summary:

The email exchange between Bitcoin developers highlights a proposed Bitcoin Improvement Proposal (BIP) aimed at improving block availability. The proposal suggests allowing nodes to request upload and download of blocks that have already been partially downloaded. This would make the blockchain more accessible to nodes with limited internet connectivity while also saving bandwidth among nodes. However, not all developers agree on the proposed improvement. Amir Taaki believes that bandwidth is not a bottleneck for the system and that clients should never send blocks first. He argues that introducing this change would cause disruption without significant benefits. On the other hand, Rebroad suggested adding a hash to the block header to allow nodes to reject already downloaded blocks, thus saving bandwidth among well-behaving nodes.Wladimir, another developer, agreed that it would make sense for clients to be able to reject blocks they already have. However, he thinks introducing a BIP for this is unnecessary since the blockchain can be downloaded from an HTTP server or passed along on a USB stick. Wladimir recommends running a lightweight node like Electrum for those with restrictive or slow internet connections.The discussion highlights different views on optimizing aspects of the Bitcoin system and emphasizes the need to focus on significant improvements while keeping everything working with minimal changes. Currently, nodes with slow or restrictive internet have options such as going via a tor proxy, but the problem of multiple receptions of the same block still occurs due to latency. It is better for those with slow internet connections to run a lightweight node like Electrum rather than implementing unnecessary changes to the system.
