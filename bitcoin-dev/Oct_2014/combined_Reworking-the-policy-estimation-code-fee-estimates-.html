Summary:

The conversation between Gavin Andresen and Alex Morcos revolves around the fee estimation logic in Bitcoin. Specifically, they discuss the threshold percentage for confirmation and the issues with the current fee estimation code. Gavin suggests a default of 80% for the threshold percentage, while Alex suggests making it tunable as an argument to an RPC call.However, Gavin is hesitant to add too many options to the RPC interface and suggests a command-line/bitcoin.conf option for the default percentage instead. Alex has found issues with the current fee estimation code, which can result in high fees being estimated based on little data. This can lead to wallets being emptied quickly and exchanges losing money due to an exploit.To address this issue, a user-configurable failsafe limit is suggested, but Alex believes a more sophisticated approach that takes into account coin-age of observed transactions may be better. He has reworked the code to divide all possible fee rates up into logarithmic fee rate buckets and updates the appropriate bucket with how many blocks it took for a transaction to confirm.While this new code may return a higher rate than the existing code, it is because no one places transactions with such small fees. In their email conversation, Alex suggests making the 90% number an argument to an RPC call, but Gavin believes setting the default percentage as a command-line or bitcoin.conf option would be sufficient. Overall, the discussion highlights the need for a functioning fee market in Bitcoin and finding the best approach to fee estimation.
