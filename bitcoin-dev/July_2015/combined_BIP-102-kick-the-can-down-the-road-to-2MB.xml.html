Summary:

In July 2015, the Bitcoin community was engaged in a discussion about proposals to increase the block size limit. The members of the community proposed various solutions to address this issue, including increasing the block size every 100k blocks or using block time instead of block height to validate block size. Some developers suggested that hard forks should require consensus among nodes rather than majority among miners.Jeff Garzik proposed BIP102 to increase the block size limit to 2MB, but some developers like Peter Todd criticized the proposal, calling it "extremely poorly designed" and suggesting that Garzik may be trolling or incompetent. Another developer, Ross Nicoll, suggested using block time instead of block height to validate block size and proposed a minimum target of 6 months for testing virtually every wallet/node out there. Luke Dashjr argued that hard forks should require consensus among nodes rather than majority among miners.In response, Jeff Garzik proposed BIP173 as a minimum viable alternative plan to his preferred proposal, BIP100, which seeks to provide a backup plan if an agreement cannot be reached on a more comprehensive solution. The benefits of BIP173 are that it provides a conservative increase and allows for added growth while the community gathers data on how an increased block size impacts various metrics.The email exchanges between the developers revealed disagreements on whether to use block height or median time to determine maximum block size. There were also concerns about the risks involved in allocating BIP numbers and implementing hard forks without proper testing and consensus among nodes.Overall, these discussions demonstrated the ongoing evolution of Bitcoin and other cryptocurrencies as developers seek to improve their functionality and usability.
