Summary:

Bitcoin developers have been discussing the need for greater modularization in Bitcoin development to avoid contentious forks. The lack of a layered protocol design presents a challenge, as there is no standards process that is separate from updates to one specific reference implementation. Tamas Blummer suggests a slim protocol server with no wallet, RPC or QT, but with a high-performance remoting API. He also proposed creating an API to the consensus+networking+storage service that is not at the C++ language level but some scalable cross-platform remoting like ZeroMQ.In various email conversations and forum posts, the development of libconsensus, an independent library for Bitcoin consensus rules, is discussed. There is a need for independence and isolation of the library from Bitcoin Core, and the possibility of using libsecp256k1 instead of OpenSSL is explored. The goal of libconsensus is to reduce the "gatekeeping" friction around Bitcoin Core, and competitive alternative implementations are needed. However, any changes that touch consensus code are risky and slow.Bitcoin developers discuss the importance of modularizing the code and separating consensus rules from Bitcoin Core to avoid the need for re-implementation. There is also discussion about integrating secp256k1 as part of consensus and expanding libconsensus' API to offer VerifyTx, VerifyHeader and VerifyBlock, while still allowing for control over storage, concurrency, networking, and policy. The bitcoin-dev mailing list has been discussing the need for an independent consensus library that can be used by alternative implementations, with a simple API for blocks and the current chain. The goal is to have a sufficiently modularized codebase that swapping out consensus rules is straightforward and easy to test, thus reducing the risk of consensus fork bugs. Tamas Blummer proposes a new infrastructure for enterprise applications, suggesting a slim protocol server with no wallet, RPC or QT, but with a high-performance remoting API. The lack of attention towards modularization PRs is identified as an issue, which reduces the risks associated with each change, enables more alternative code bases to be implemented, and allows more people to contribute with code and review.Overall, the conversation emphasizes the need for a modularized codebase that is easy to maintain and test, with a simple API for consensus rules that can be used by alternative implementations. The risks associated with reimplementation, refactoring, and copy-pasting in Bitcoin development are discussed, as well as the importance of functional equivalence and quality control. Encapsulating consensus code so that it can be built without any script-related code is also recommended.
