Summary:

Matt Whitlock, a Bitcoin developer, is running his full node with Peter Todd's full replace-by-fee patch set and is seeing a steady stream of replacement transaction rejection messages due to the ongoing spam attack. He wants to cut off the flood coming into his node by temp-banning peers who relay invalid replacement transactions. This could reduce CPU load, and if other nodes implement this banning rule, it could cut off the flood of spam at the source. However, Patrick Strateman questioned its practicality. Tom Harding suggests that acting against the peer is not the solution. In response, Alex Morcos suggested tracking recently-rejected transaction IDs and not getting data from them, which could work only if conflicts are relayed. The link for the patch set is provided in the email.
