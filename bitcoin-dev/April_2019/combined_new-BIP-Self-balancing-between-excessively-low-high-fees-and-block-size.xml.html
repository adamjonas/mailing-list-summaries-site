Summary:

A new Bitcoin Improvement Proposal (BIP) has been submitted by simondev1 to limit block size in order to prevent spam transactions and keep block space small. The proposal suggests that the logarithm of transaction fees should be used to determine the size of each block. The proposal also includes an auto balance fee system, which would increase low fees and decrease high fees. This would allow for larger sizes when the sender pays more. The BIP specifies that every transaction must fit into a certain block space based on the input variable 'FeeInSatoshiPerByte' and the output type uint, unit bytes. Transactions would be sorted by FeeInSatoshiPerByte from lowest to highest, and the block size would be validated accordingly. If implemented as a soft fork, backward compatibility would be maintained, but if implemented as a hard fork, backward compatibility would not be maintained.Discussions are ongoing within the bitcoin-dev community regarding the implementation of a new algorithm for sorting and validating transactions in a block. The proposed algorithm suggests that transactions should be sorted by their FeeInSatoshiPerByte, with the lowest fee transactions being processed first. Some members of the community have raised concerns about whether this approach would break CPFP (child pays for parent) and suggested that chains of transactions need to be treated as a group to avoid this issue. It has also been noted that multiple transactions in one block could depend on the same "parent" transaction, which may affect the validation process. Despite these concerns, the proposal presents a potential solution to the growing problem of spam transactions and could bring about positive changes to the Bitcoin network. The full BIP can be found on GitHub.
