Summary:

The email exchange between Jeremy and AJ discussed implementing per-input locktimes to disambiguate which branch of a chain split or reorg a transaction is valid for. They discussed various options, including defining a new CSV type and creating a sequenced signature. The conversation also touched upon the disallowance of using both relative locktime and absolute locktime for the same input.Jeremy suggested disallowing the use of both locktimes for the same input, but AJ proposed a more general principle where evaluating a script should only return one bit of information: "bool tx_is_invalid_script_failed." This would make all other information obvious from a simple parsing of the transaction. There was some disagreement on whether overloading nSequence for a per-input absolute locktime would prevent the reuse of the same input's nSequence for a per-input relative locktime.In another email exchange, Jeremy provided an example of a transaction that will never be valid, regardless of the state of the chain it's on. He stated that transactions with timelocks go from invalid to valid once they pass a certain block height but transactions that become invalid cannot become valid again for a given UTXO. This limits the number of cases where transactions (and their dependents) need to be removed from the mempool.Overall, there seems to be agreement that per-input locktimes would be beneficial and satisfy multiple concerns. Additionally, the concept of sequence tagged keys was briefly discussed, as it has the property of a transaction being either valid or invalid without any external information needed to be passed up. Finally, the authors considered defining a new generic place for lock times in the annex.
