Summary:

The discussion on the use of 'h' or apostrophes in Bitcoin descriptors and derivation paths has been a topic of debate. Craig Raw expressed concerns, stating that using an apostrophe allows for more whitespace and easier visual parsing, while David A. Harding argued that using 'h' or 'H' avoids potential errors in shell-quoting and is more efficient for metal backups. Daniel Bayerdorffer recommended the Bitcoin Recovery Tag as a metal backup option that supports both apostrophes and 'h'. The debate highlights the importance of considering user experience and practicality in the design of Bitcoin-related tools and protocols.Andrew Chow proposed to formalize the Output Script Descriptors available in Bitcoin Core into BIPs. He split it into seven BIPs where the first one is informational and the following six specify the actual descriptors themselves. These descriptors offer a generic solution to issues with traditional wallet backups by specifying script types explicitly through the use of Script Expressions and key derivation paths explicitly in Key Expressions. The BIPs specify various descriptors, including segwit descriptors (wpkh, wsh), non-segwit descriptors (pk, pkh, sh), multisig descriptors (multi, sortedmulti), taproot descriptor (tr), combo descriptor, and opaque descriptors (raw, addr).The backward compatibility of these descriptors is discussed, and they have been implemented in Bitcoin Core since version 0.17. However, tr() descriptors have only been implemented since version 22.0. The proposal suggests dropping the combo() descriptor and including metadata format to track birth data, gap limit, and other needed information for future wallets.Sjors Provoost suggested adding a birth date or minimum block height to the BIP 32 and BIP 88 standards for Hierarchical Deterministic Path Templates. He suggested using a metadata format to track this information alongside descriptors, which is supported by the Airgap Wallet Community's UR standards.Andrew Chow updated the text to use "h" instead of apostrophes, citing the need for dumb checksum checkers that don't have to parse descriptors. Raw expressed concerns regarding the use of "h"/"H" in descriptors, but ultimately agreed with Harding's points.Bitcoin Core has implemented all these descriptors since version 0.17, except for tr() expressions which have been implemented since version 22.0. The wpkh() and wsh() output script descriptors have been specified for P2WPKH and P2WSH formats, while the Non-Segwit Output Script Descriptors specify pk(), pkh(), and sh(). Multi-sig output script descriptors are discussed in the MultiSig Output Script Descriptors BIP. The combo() output script descriptors take a key and produce P2PK, P2PKH, P2WPKH, and P2SH-P2WPKH output scripts if applicable to the key. The raw() and addr() output script descriptors allow for wrapping any arbitrary output script or an address into a descriptor, respectively.There is a discussion on whether there are plans to support time locks in descriptors other than 'raw'. Along with this, there are also talks about integrating miniscript more closely with descriptors. Andrew Poelstra expects miniscript to be a follow-up BIP that extends these descriptors and has timelock functionality. Blockchain Commons libraries and tools are multisig descriptor-centric, and there are many scenarios that require describing time locks. In conclusion, the formalization of Output Script Descriptors offers a generic solution to issues with traditional wallet backups and exports. Although there are discussions about the use of 'h' or apostrophe in descriptors and derivation paths, the difference in space is negligible, and the use of checksums helps avoid transcription errors. The Bitcoin Core has implemented these descriptors and backward compatibility has been considered.
