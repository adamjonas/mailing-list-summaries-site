Summary:

In recent email exchanges within the Bitcoin community, several individuals have proposed and discussed changes to the Partially Signed Bitcoin Transaction (PSBT) format to support Taproot. Specifically, there has been confusion over why PSBT_IN_TAP_BIP32_DERIVATION and PSBT_OUT_TAP_BIP32_DERIVATION contain not just the derivation path for the xonlypubkey but also the tapleaf merkle path.Pieter Wuille clarified that these additional fields are necessary for signers who may not have prior knowledge of the script being signed. Salvatore Ingala suggested splitting the control block for the Taproot Leaf Script into two separate PSBT types, but Andrew Chow pointed out that this would not allow multiple leaf scripts to be specified.Jeremy Rubin proposed allowing different keys to specify different sighash flags in order to provide greater flexibility in specifying signature requirements. Meanwhile, Andrew Chow had proposed a BIP which defines new fields for Taproot support in PSBT, including the Taproot Key BIP 32 Derivation Path for both PSBT_IN_TAP_BIP32_DERIVATION and PSBT_OUT_TAP_BIP32_DERIVATION. These fields contain the 32-byte X-only public key, a compact size unsigned integer representing the number of leaf hashes, followed by a list of leaf hashes, followed by the master key fingerprint concatenated with the derivation path of the public key. The proposal also recommends using PSBT_IN_WITNESS_UTXO instead of PSBT_IN_NON_WITNESS_UTXO for Taproot inputs because a Taproot signature will commit to all of the amounts and output scripts spent by the inputs of the transaction, thus preventing any potential attacks involving an updater lying about the amounts in an output.The proposal is a draft and is licensed under the 2-clause BSD license. The compatibility of this proposal with the existing PSBT format is ensured as it is designed to be extensible, and old software will ignore the new fields. Test vectors are yet to be determined.
