Summary:

In August 2020, Suhas Daftuar proposed a Bitcoin Improvement Proposal (BIP) suggesting that Bitcoin network clients should ignore unknown messages received before a VERACK. This proposal aims to make future p2p protocol changes more efficient by eliminating the current pattern of sending a message indicating support for a given feature and replacing it with something more flexible. However, this proposal has been met with criticism for being poor protocol design and a protocol breaking change.Eric Voskuil responded to Suhas' proposal, suggesting that accompanying new messages with a protocol version bump would be a better approach to ensure network clients are never reading unknown messages from a peer. Voskuil further suggested making new features optional at the new protocol level to avoid communication issues and allow each client to limit its communication to the negotiated protocol.In response to Voskuil's suggestion, aj proposed a specific feature negotiation message that would only be valid between VERSION and VERACK. This message would include a length-prefixed-string giving the feature name, optional additional data, and if the feature name is not recognized, the message would be ignored. Matt Corallo also proposed adding a multi-round negotiation feature to Bitcoin protocol messages to improve efficiency.There were also discussions around the security implications of certain features in the protocol, with some suggesting disabling them if privacy is a concern. However, others argued that this approach may not be effective in ensuring security and that changes to version negotiation itself can pose challenges. Another proposal discussed was the creation of a new message, XVERSION, as an alternative to allowing unknown messages on the network. This message would allow for negotiation of non-bilateral feature support and provide a way to communicate maximum unconfirmed chain limits. While initially proposed to establish expectations for new feature deployment, it was eventually withdrawn due to its complexity.Additionally, there were proposals for using a FEATURE wrapper for all messages during the negotiation period and adding a SYNC message for multi-round negotiation. Some participants suggested implementing a more explicit negotiation phase after the version and before VERACK to avoid overloading the network with unnecessary messages. The main concern is acknowledging whether a feature is enabled or disabled, so there is a suggestion of adding a FEATURE wrapper or a rule that all messages in this period should be interpreted as features.The discussions among Bitcoin developers also raised questions about what constitutes the existing Bitcoin protocol and the need for backward compatibility. While some argue for a single message that lists the set of features which a node wishes to support on the connection, others propose extending the VERACK with a matrix of feature support. Overall, the discussions centered around improving the efficiency and compatibility of the Bitcoin protocol while maintaining security and privacy considerations. Proposals such as multi-round negotiations and feature negotiation messages were discussed as potential solutions, though concerns were raised about over-engineering the protocol without a clear need. Ultimately, the principle of rough consensus and running code applies in deciding which proposal to use.
