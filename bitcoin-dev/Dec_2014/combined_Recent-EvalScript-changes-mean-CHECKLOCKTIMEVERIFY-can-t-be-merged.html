Summary:

The Bitcoin Core project has faced criticism for the frequent and speedy refactorings that can cause unforeseen problems. The developers suggest breaking up code into smaller files to optimize review, followed by refactoring in a second wave after a stable release. However, Pieter Todd expressed concerns about the development process of the consensus critical codebase, recommending not upgrading to v0.10 due to these issues.Jeff Garzik suggested that if the majority of code movement was done upfront, followed by simplifications and data structure work, further patches would be easier to review and apply with less impact on unrelated code. BtcDrak found a significant design change that makes merging the CHECKLOCKTIMEVERIFY patch to master impossible. Bitcoin Core developers have also expressed concerns about the modularization process of code movement and refactoring. The process resulted in a blizzard of new files and data structures, making it hard for out-of-tree patches to be applied and reviewed. While Jeff Garzik suggested that most of the code movement should be done upfront, followed by simplifications and data structure work, Cory Fields argued that minor code changes are almost always needed to accommodate useful code movement at the micro level.Pieter Todd raised concerns regarding the safety of changes made to the consensus critical codebase, suggesting not upgrading to v0.10 due to these issues as it would come at the cost of safety. Todd also noted that some refactoring is unavoidable, but having the consensus isolated into a portable self-contained library is the most important goal of Bitcoin Core project.In an email to the Bitcoin development mailing list, Todd pointed out that the large volume of code being moved and changed for the v0.10 release made it difficult for him and others to keep up, especially with significant consensus critical code changes. He recommended a safer approach would be to limit the changes required to simple code movements, accept that the interface to the consensus library won't be ideal, and wait for feedback from multiple open-source projects.In another email conversation, Jeff Garzik and Cory Fields discussed the challenges of code movement and refactoring during Bitcoin's modularization process. While Jeff suggested that most of the code movement should be done upfront, followed by simplifications and data structure work, Cory argued that minor code changes are often necessary to accommodate useful code movement, making it hard to justify code movement for the sake of it. He gave an example of a pull request that unchains Bitcoin's openssl wrapper so that key operations can be performed by the consensus lib without dragging in bitcoind's structures. However, Cory found it challenging to handle these two changes as separate events while making review easier, believing it would be a massive headache in practice.
