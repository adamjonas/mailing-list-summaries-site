Summary:

In an email addressed to Peter, John Light raises questions regarding a proposed policy change that would remove the OP_RETURN output count and data size limits. The first question asks if there are cases where using OP_RETURN to embed data actually results in fewer bytes onchain compared to using segwit/taproot witness space. Robin Linus suggests there may be a case where small inscriptions receive a discount, but further confirmation and specific details are needed. John invites Robin to respond directly.The second question pertains to the prunability of OP_RETURN outputs. John wonders if there are tools available for full node operators to prune this data from their nodes. He references PR #2791 by Pieter Wuille, which implemented pruning of provably-unspendable outputs, assumed to include OP_RETURN outputs. This leads to additional questions: i) Is the unspendable output pruning in PR #2791 enabled by default or does it require a flag to be enabled? If it's a flag, what is it called and how can it be enabled? If it's on by default, how can it be disabled?ii) If a full node operator prunes OP_RETURN outputs, does it affect their ability to help a new node with initial block download (IBD) for blockchain validation? And would the answer differ if Taproot witness space (i.e., "envelopes" or "inscriptions") were pruned instead?John also mentions Sjors Provoost's suggestion to send an email to the mailing list to notify them about the intention to merge the pull request. The pull request aims to remove standardness restrictions on OP_RETURN outputs, allowing more than 80 bytes and 1 output per transaction. The rationale behind this proposal is to potentially open up additional transaction pinning vectors. Transaction pinning involves intentionally including certain transactions in blocks to prevent others from being confirmed. While some may view transaction pinning as a problem, the author of the email does not consider it a significant concern. It is noted that some miners are already mining non-bitcoin-core-standard transactions, implying that pinning vectors may already exist.
