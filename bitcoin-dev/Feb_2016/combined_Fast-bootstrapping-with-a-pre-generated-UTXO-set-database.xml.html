Summary:

The proposals aim to reduce the bootstrap time and network usage of bitcoin nodes. One proposal suggests building the UTXO set backwards, starting from the newest block and working backwards. This involves checking each transaction in last-to-first ordering and adding it to the UTXO set if it's not already in the UFTXI set. The other proposal involves loading a pre-generated UTXO-set datafile, which will be deterministically linearized at a specified height and hashed with a double SHA256 after appending serialized form of the current chain-index. A group of developers will sign this hash with ECDSA, and full node client implementations that support bootstrapping from a pre-generated UTXO-set need to include a set of pubkeys from trusted developers, the hash (or hashes) of the pre-generated UTXO-set-datafile(s), and n signatures of the hash(es) from a subset of developers defined in the pubkeys. To ensure the integrity of developer pubkeys & signatures, methods like the current gitian build must be used. New nodes can download a copy of the pre-generated UTXO-set and continue bootstrapping from the specified height if the users accept the amount of valid signatures. Sharing of the pre-generated UTXO-set can be done over CDNs, BitTorrent or any other file hosting solution. It is also possible to extend the bitcoin p2p layer with features to distribute/share such a pre-generated UTXO-set.
