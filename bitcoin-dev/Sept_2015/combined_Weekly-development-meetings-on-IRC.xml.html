Summary:

In September 2015, the Bitcoin community discussed the idea of scheduling a regular developer meeting in #bitcoin-dev to discuss current issues related to Bitcoin. The attendance would be voluntary, but having a set time for meetings would allow for more people to attend. Several suggestions were made for days and times, with one proposal being every week on Thursday from 15:00-16:00 UTC. A Doodle poll was created to determine the most popular time for the meeting, which resulted in Thursday from 19:00-20:00 UTC.However, there were concerns about using UTC since it may not be available in Google Calendar and could cause inconvenience for those in different time zones due to changes in daylight saving time. It was suggested to use a timezone that observes DST in roughly the same way as everyone else, such as CEST/PDT/EST. There was also mention of the argument against using Google Calendar if it cannot support standard UTC. Overall, the group agreed that it would be best to select a timezone that is available in Google Calendar and observes DST in roughly the same way as everyone else to ensure consistency.During a code sprint meeting, the idea of scheduling a regular developer meeting was proposed and was generally well-received. The purpose of the meeting would be to discuss current issues in the Bitcoin community and to provide an opportunity for developers to collaborate. In an email conversation dated September 21, 2015, a person named gb provided insights based on their experience with other transglobal IT projects. Although the planning for the project was already underway, gb's input could still be valuable.During a discussion on the bitcoin-dev mailing list in 2015, the issue of Daylight Saving Time (DST) usage was brought up. It was pointed out that little global consistency exists in DST usage, with different places implementing it on various dates. This means that if an event is scheduled in a DST timezone, it is likely to move twice for some people who observe it. For instance, Europe and the US end DST one week apart.Btc Drak jokingly suggested that time needs an upgrade and asked if it is possible to hardfork time. Gregory Maxwell responded by stating that using a timezone that observes DST like CEST/PDT/EST might not be feasible due to the inconsistency in DST usage. Matt Corallo suggested using Google Calendar's option to change the timezone of an event but noted that UTC is not available as an option. In response, Btc Drak suggested using Iceland time as an alternative since it uses UTC without DST. Therefore, despite the lack of global consistency in DST usage, using Iceland time as the standard time could potentially resolve the issue of scheduling events in different time zones.
