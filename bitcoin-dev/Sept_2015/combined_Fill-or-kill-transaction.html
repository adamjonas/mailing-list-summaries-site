Summary:

The Bitcoin development mailing list discussed various topics related to Bitcoin development on September 17, 2015. One of the topics was the implementation of a fill-or-kill system using nLockTime and nSequence parameters. The proposed feature allows for flexible confirmation times with two new parameters, nLockTime2 and nKillTime. The system could be activated by setting a bit flag in the transaction nVersion. The proposed system has a maximum confirmation window of 8188 blocks or 16,769,024 seconds, and the resolution is 4 blocks or 2048 seconds.However, height-based OP_CLTV is not compatible with nLockTime2, and a new OP_CLTV2 may be needed to verify both nLockTime and nLockTime2. For height-based nLockTime2, the lock time with the new rule must be 1-2048 seconds later than the original rule for every nLockTime2 value greater than or equal to 720,000, making it a soft fork. In addition, if a hard fork does not happen before 55 years, the height-based nLockTime2 will overflow, requiring a better fill-or-kill system. There is also discussion about implementing the fill-or-kill system as a consensus rule, with suggestions for how to ensure transaction safety and drop the 100 block maturity requirement.Peter Todd suggested using OP_MATURITY 100 in all scriptPubKeys of an expiry transaction to ensure reorg-safe outputs. However, some members of the community pointed out that this could be confusing and suggested alternative solutions. Another suggestion was to implement a "treat like Coinbase" flag in the UTXO set when creating an output to solve this issue.Luke Dashjr responded to Jorge Timón's suggestion that expired transactions could be made as reorg-safe as coinbase transactions by requiring op_maturity/csv/rcltv of 100 for all outputs of an expirable transaction. Dashjr pointed out that this is not quite as safe as coinbase transactions because mined bitcoins have not only a 100-block maturity requirement but are also expensive to generate.To ensure safety, all outputs of a transaction that can expire should have op_maturity/csv/rcltv of 100, making them as reorg-safe as coinbase transactions. Unfortunately, this does not work well with p2sh. A new OP_CLTV2 might be required that can verify both nLockTime and nLockTime2.Peter Todd also emphasized the importance of using accurate terminology to facilitate learning and understanding among members of the community.
