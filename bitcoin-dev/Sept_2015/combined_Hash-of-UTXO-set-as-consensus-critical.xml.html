Summary:

The discussion around UTXO set commitments and their impact on Bitcoin's security model was the topic of a conversation on the bitcoin-dev mailing list. Rune Kjær Svendsen suggested that a newly mined block should include the hash of the current UTXO set to enable new nodes to acquire the UTXO set in a trustless manner. This would force miners to maintain a UTXO set and verify the block they want to build on top of.However, Patrick Strateman argued that full nodes using UTXO set commitments is a change to the Bitcoin security model since an attacker with more than 50% of the network hashrate can rewrite history and create infinite bitcoins. It is important to discuss whether the security model reduction is reasonable before considering mechanisms for UTXO set commitments.Currently, when a new node wants to join the network, it needs to retrieve the entire blockchain history to derive a UTXO set that it can verify new blocks/transactions against. With a blockchain size of 40GB and a UTXO size of around 1GB, the extra bandwidth required is significant. A newly mined block could include the UTXO set hash of the chain up until the previous block, allowing new nodes to acquire the UTXO set trustlessly by only verifying proof-of-work headers. The inclusion of UTXO set commitments alleviates the burden on Bitcoin Core nodes and saves bandwidth while maintaining Bitcoin's security model. However, there are concerns about the security model reduction with UTXO set commitments and the potential for >50% attacks to rewrite history and create an infinite number of bitcoins. The proposal requires additional computation, but it is argued that this is a small sacrifice for the added ease of initial syncing.The context also discusses a proposal for reducing the security model of Bitcoin from 'always secure' to 'secure, as of last year' by hashing the UTXO set and storing it on disk, along with a copy of the UTXO set at that time. Blocks before H1 can be pruned, only keeping D1 for reference/validation. While this proposal increases the difficulty for an attacker with hidden hashing power to overwrite a year's worth of blocks, there are concerns about its feasibility and whether it would provide sufficient security against >50% attacks.The scalability issues of validating the full blockchain history for new nodes joining the network were also discussed, with Svendsen suggesting that including the hash of the current UTXO set in newly mined blocks would allow for a trustless initial sync and alleviate bandwidth issues. The proposal requires additional computation, but most modern CPUs have a SHA256 throughput of around 500 MB/s, making it a small sacrifice for added ease of initial syncing. There is a practical limit to the size of a re-org, with Vincent's proposal suggesting that a year's worth of blocks is a practical limit that an attacker would have difficulty overcoming. Overall, while there were differing opinions on how to handle UTXO set commitments and their impact on Bitcoin's security model, it is important to address the issue of distinguishing between malicious and honest blocks. The discussion highlights the importance of balancing security and scalability in the evolution of Bitcoin's consensus mechanism.
