Summary:

Bitcoin Core's codebase requires extensive modularization to improve its structure. However, this needs proper planning and consideration for downstream projects relying on Core and other contributors to the code. Jeff Garzik highlighted the negative impact of refactors, stating that branches of the source tree are being maintained by software engineers at home, startups, and major companies.The goal of moving consensus code to a separate library is reasonable, but there appears to be a lack of planning and an unfocused approach when it comes to seemingly random stream of refactoring pull requests. Developers require more links or hooks to see the missing aspects in the development process. Jorge Timón has tried many times to communicate his plan for libconsensus, but complex code changes with longer development cycles than simple code movement patches keep breaking. To address these issues, the author recommends time-based bursts of code movement changes and a merge window approach like that used in the Linux kernel maintenance phases. The Bitcoin-dev mailing list recently discussed the ongoing refactoring process in Bitcoin Core, with concerns raised about the lack of a plan or end goal. To address this issue, Btc Drak suggested that refactoring should only be done during scheduled periods and not mixed with other pull requests, while Jeff Garzik recommended time-based bursts of code movement changes and a merge window approach.Gavin Andresen proposed a plan to separate consensus code into libconsensus, which is important for less risky and wider contributions. Companies can maintain forks of Bitcoin Core unnecessarily when a different architecture could do what they need without including new logic in the codebase itself. The ideal solution is to build functionality without changing bitcoind's code at all or to change the code in as few places as possible. Wladimir's long-term vision of bitcoind is a P2P node with validation and blockchain store, with a couple of data sources that can be subscribed to or pulled from.The article discusses the possibility of implementing an index for simplewallets against one node daemon through a library that communicates with the node. This library would offer functionality similar to 'git pull' and enable communication between the two. Unlike bitcoind's wallet, the simplewallet would not need to run as always-on daemon itself.The article suggests that SPV libraries like bitcoinj could be used to build extra indexes and calculate a UTXO set indexed by address. These libraries know how to build a block locator, request blocks forward from the common branch point, and handle re-orgs by downloading data from a full node. By using a library to communicate with the node, it would be possible to implement the index entirely externally while fully handling reorgs. This approach offers a more efficient and practical solution compared to running multiple daemons for each simplewallet. The focus should be on maintaining the core bitcoin full node P2P engine. Overall, the context provides insights into the challenges faced by Bitcoin Core and the efforts made by contributors to address them. Developers are encouraged to imagine several different ways people might try to find this description in the future and make them work. Further discussion on the bitcoin-dev mailing list is also encouraged.
