Summary:

The Bitcoin community has been discussing the issue of block size for some time. While there are concerns about a potential selfish mining attack on the network, which could be amplified by excess block size, it would have to be sustained for at least 2000 blocks over a period of 4 weeks to achieve a meager 10% increase in the block size. To address the issue of block size while also considering the fee market, a proposal for a dynamic limit on the block size has been made. The proposal suggests a conservative increase in the maximum block size while providing relief to transaction volume pressure in response to true market demand. The goal is to set a "triggering level" for an increase in block size, indicating an increase in transaction volume that would fill up blocks. The appropriate triggering level is difficult to determine and consists of three parameters: evaluation period, capacity, and threshold. The proposal takes a decentralized approach similar to Satoshi's decision on altering the target difficulty. The maximum block size can only ever be increased, not decreased. The proposal allows for at least 4-5 years to develop efficiency improvements and network infrastructure.Developers agree that changing the fundamental rules for Bitcoin should be hard, and for this reason, scaling BIPs such as BIP101, Pieter's proposal are preferable for their predictability and simplicity. However, some developers are unclear on what problem(s) this proposal is trying to solve. They suggest implementing simple rules such as specifying a maximum block size based on the average block size over the last N blocks or implementing an absolute maximum block size that increases by a maximum of 3.45 times every year.There is a discussion around reducing the maximum block size limit for Bitcoin, but it is argued that there is no need to reduce the max block size unless technology advances or the internet is impacted by WW3. However, if the block size can only increase, it will hurt the fee market as transaction volume may ebb and flow significantly. Therefore, the block size needs to be elastic and allow contraction as well as expansion.The proposal emphasizes the need for an economic discussion rather than a technology scaling discussion, identifying the risks of unbounded growth and selfish mining attacks. There is also a concern about the potential need for a hard fork in the future and the difficulty of upgrading embedded firmware. The discussion highlights the importance of a technical approach to block scaling rather than a political one, and the need for decentralized decision-making in determining how full blocks "should" be.Sanchez, the author of the proposal, is seeking feedback, suggestions, and alternatives related to some of the variables in his specification. Additional calculations show theoretical maximums for block sizes and transaction processing capacity for each year until 2020. Overall, the proposal allows for a conservative increase in the maximum block size while providing relief to transaction volume pressure in response to true market demand.
