Summary:

The standardization of op_return outputs with multiple pushdatas is being questioned by the author of a script wiki, as there is uncertainty on whether it would be considered standard. The policy.cpp file on Github does not limit the number of pushdatas allowed in op_return, but it is unclear whether the script wiki is up to date. A user named Chris proposed making op_return outputs with two pushdatas standard, with the same maximum data size, which would allow transactions to be easily tagged and returned by bloom filtering nodes. Luke Dashjr responded that standards (BIPs) need to describe a specific use case and protocol for implementation, although most nodes already allow such outputs by default.This proposal aims to facilitate the tagging of transactions so that they can be filtered by nodes using op_return output scripts. This provides an easy way to fetch transactions where the tag matches a specific pattern. It appears that many nodes and miners already accept such transactions, as the first block was mined using this method. Therefore, the proposal seeks to establish a clear protocol for standardization of op_return outputs with multiple pushdatas. On January 9, 2017, Chris via bitcoin-dev raised the question about standardizing op_return outputs with two pushdatas. Luke responded that BIPs should describe a specific use case and protocol for implementation, and most nodes already allow such outputs by default. However, the author of the script wiki remains uncertain about whether multiple pushdatas in op_return would normally be considered standard.
