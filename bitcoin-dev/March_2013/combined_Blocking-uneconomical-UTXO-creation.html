Summary:

The conversation around scalability in Bitcoin revolves around the issue of unspent transaction outputs (UTXO) and its cost. Peter Todd, a Bitcoin developer, suggests that the cost of storing data in UTXO set is more than storing the data itself. He proposes a fee system that captures the long-term costs of UTXO creation and discusses timestamping and secure data backups as potential uses of provably unspendable txouts for data applications. He even suggests making a standard transaction type for unspendable but not provably so txouts for secure data backups. Stephen Pair proposes a different way of thinking about transactions in Bitcoin, suggesting paying for a feed of profitable transactions instead of blocking uneconomical transactions. He argues that this system would determine the probability of confirmation for a given fee within a given timeframe. The email also includes a promotion for Symantec Endpoint Protection 12 for endpoint security, along with a link to access the full report for selecting the right partner to tackle endpoint security challenges.In another discussion, Gavin Andresen and Jeff Garzik exchange emails about the possibility of demurrage. While Andresen believes that demurrage would never happen due to negative publicity, Garzik suggests the idea of a well-done coin that is not Bitcoin. He calls it "tempcoin" and suggests that it could have a finite lifespan. Although many alt-coins have been pre-mined scams, Garzik believes that the concept of alt-coins is great for experimenting. If the idea could be proven in the field without modifying mainnet bitcoind, then perhaps it should go onto the mainnet hard fork wishlist, many years in the future.The issue of scalability in Bitcoin also touches on the UTXO problem, which increases with the block size limit. Benjamin Lindner points out that systems like SatoshiDice contribute to a bloat of UTXO data at a speed of 52 GB/year. Mike Hearn suggests that if necessary, outputs that are unlikely to be spent can be moved into a separate database to lighten the load on LevelDB. There was also a concern raised about an attack if UTXO is not stored in fast storage.Peter Todd suggests making transaction outputs with values less than the transaction fee non-standard in order to prevent spam transactions. However, Jorge Timón disagrees with this approach, stating that it would disable legitimate use cases for payments worth less than the fees. He proposes activating a non-proportional demurrage instead, which would incentivize old transactions to be moved and destroy unspent transactions with small amounts that don't move to another address periodically. While this proposal makes sense economically, Rune suggests that it isn't feasible if Bitcoin wants to maintain people's confidence in the currency. Instead, Jeff's proposal to let an alt-coin implement this feature could be a better option.The discussion also touches on the use of RAM as a database cache and potential attack scenarios on the UTXO. Suggestions were made for incentivizing the movement of old transactions and discouraging spamming at the protocol level. Additionally, there were discussions around the use of Distributed Hash Table (DHT) to share the load and accommodate a growing set of UTXO data due to abuse/misuse or massive use. Wallet apps have incentives to find a balance in output sizes and defragment themselves if their average output gets too low in value.Lastly, the conversation includes proposals for improving scalability in Bitcoin, such as making transaction validation multi-threaded in every case, transmitting merkle blocks instead of full blocks, moving blocking disk IO off the main loop so nodes don't go unresponsive when somebody downloads the chain from them, and finishing the payment protocol work so there's less incentive to replicate the SD "transactions as messages" design. Overall, the email exchanges and discussions highlighted the need for caution when dealing with large amounts of data and the importance of finding scalable solutions for the Bitcoin network.
