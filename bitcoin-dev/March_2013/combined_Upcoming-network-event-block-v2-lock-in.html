Summary:

In March 2013, the Bitcoin network underwent a fork to version 2 due to issues with version 1. Bitcoin developer Jeff Garzik advised miners to use a recent bitcoind (0.7 or later) to mine as some older miners may produce newly-invalid blocks that get ignored. Garzik encouraged miners who needed help upgrading to version 2 to reach out for assistance via #bitcoin-dev or bitcoin-development at lists.sourceforge.net.Jeff Garzik and Luke-Jr had discussions about the use of bitcoind for mining in March 2013. In one email exchange on March 23, 2013, Luke-Jr questioned whether anyone was using bitcoind 0.7 or later for mining purposes, to which Garzik responded that several pools, including slush, BTC Guild, and ozcoin, were using it. However, Luke-Jr clarified that these pools were not using bitcoind for producing coinbases where BIP 34 is implemented. In another discussion between Garzik and Luke-Jr on March 23, 2013, they debated the relevance of Bitcoind in relation to BIP 34's implementation. While Luke-Jr argued that Bitcoind was not relevant for producing coinbases, Jeff Garzik pointed out that it was misleading to suggest that Bitcoind played no role in the stack. Both parties agreed that upgrading Bitcoind alone would be sufficient to prevent the creation of forks.Although bitcoind was not used in the implementation of the miner end of BIP 34, it was still present in the stack. Luke-Jr claimed that bitcoind was nowhere in the implementation of the miner end of BIP 34, but Jeff Garzik corrected him by saying that it's not entirely true. He mentioned that bitcoind's 'getblocktemplate' RPC call is used by some to supply the block version, select transactions for the miner, and perform other tasks integral to the implementation of the miner and BIP 34. However, Luke-Jr argued that transaction selection and everything else done by bitcoind is irrelevant to BIP 34.It was noted that several open source poolservers had already supported v2 blocks for months, including p2pool since August 8, 2012, Eloipool since September 1, 2012, and slush's stratum server since its creation. However, PoolServerJ and ecoinpool were known to break when v2 blocks became mandatory. Miners must update both Bitcoind and the pool server layer to prevent the creation of invalid blocks or mine on invalid forks to avoid losing income.Overall, the implementation of BIP 34 in mining using bitcoind 0.7 or later was a topic of discussion among Bitcoin developers in March 2013. While some believed that Bitcoind was not relevant for producing coinbases where BIP 34 was implemented, others argued that it played a role in the stack. Several major mining pools were using bitcoind 0.7 or later, but some open source poolservers had already supported v2 blocks for months. Upgrading to a recent bitcoind (0.7 or later) is the easy solution for miners to prevent the creation of invalid blocks.
