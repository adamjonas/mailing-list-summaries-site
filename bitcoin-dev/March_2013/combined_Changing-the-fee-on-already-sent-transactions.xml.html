Summary:

The proposal of allowing transaction replacement for adding fees to existing transactions without the risk of double-spending has been suggested. The stubbed out replacement code would be used to replace a transaction only if another transaction spending the same tx is not already present in the mempool. The replacement code checks for conflicts with in-memory transactions and sets up a new transaction in place of the old one if the condition mentioned above is met. To ensure safety, it is crucial to check for any conflicting transactions in the mempool before replacing the old one. Peter Todd, a Bitcoin developer, proposed changing the replacement logic for Bitcoin transactions to evaluate if the candidate replacement does not remove or decrease the value of any existing outputs. This change would prevent DoS attacks by only accepting replacements that increase the fees paid by at least MIN_RELAY_TX_FEE * size. He believes that implementing this new logic correctly would be easier than child-pays-for-parent and could reduce the urgency of including child-pays-for-parent in Bitcoin. In a message sent on March 12, 2013, Peter Todd discussed the possibility of transaction replacement for adding fees to existing transactions. He noted that adding fees was already possible by respending change, at least for miners running an Eligius branch (both 0.6.0 and 0.8.0).
