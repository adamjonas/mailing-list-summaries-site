Summary:

In the Bitcoin development community, there have been various discussions regarding the security and efficiency of multisig constructions. One proposed solution involves using blinding factors, interpolation, and threshold hash with Shamir polynomial interpolation to achieve multisig constructions without new opcodes. Another proposed solution is a non-interactive multisig scheme based on threshold hash and Shamir polynomial interpolation. The scheme involves each party having a public key g*x', where x' is their private key, and rolling a random k' to compute r' = g*k'. A share of r' is broadcasted and used in lagrange interpolation across shares to compute g*k, e' = H(M) * r' and s' = k'-x*e'. The Bitcoin-dev mailing list has also seen discussions regarding the implementation of schnorr signatures and potential changes to the algorithm, such as using the hash of the message instead of concatenation. There are concerns about the security of Schnorr signatures with weaker assumptions and adaptive key choice and R choice attacks. Additionally, there is discussion about the security assumptions of multiparty signing using the Schnorr construction and the vulnerability of naive interpolation of schnorr signatures to adaptive key choice attacks.Another conversation on the bitcoin-dev mailing list discusses the security of non-interactive signatures. There is a suggestion that these types of signatures may be more secure, and it is noted that Schnorr signatures already allow for the functionality desired. Gregory Maxwell and Tim Ruffing discuss the security of non-interactive signatures and the potential for more advanced multisignature transactions using Schnorr signatures. Overall, the conversations revolve around improving the security and functionality of digital signatures in Bitcoin. There is caution advised against including novel cryptography in Bitcoin without a more complete writeup. Adam Back notes that delinearization mechanism is necessary to prevent adaptive public key choice from being used to break the scheme using Wagner's attack which is not specific to addition but a generalized birthday attack. He suggests that all public keys should be hashed along with per value hash for non-adaptively chosen public keys. Adaptively chosen public keys are dangerous and simple to exploit.
