Summary:

A discussion on the bitcoin-dev mailing list has taken place regarding the Bitcoin Improvement Proposal (BIP) 174, which proposes a standard for partially signed Bitcoin transactions (PSBT). The proposal outlines a way to allow multiple parties, such as hardware wallets, to cooperatively construct and sign Bitcoin transactions. Participants in the discussion suggested changes to the proposal, including using key-value models instead of simple lists of records, making the sighash type mandatory, and requiring the transaction record inside a PSBT to be in a canonical form.There was also a discussion on whether to include the full public key and chaincode of the parent or master things derived from in BIP32 derivation paths. Additionally, there were suggestions to improve the original proposal, such as defining symbols for the numeric key values in Appendix A and suggesting a filename extension for wallets to register with their operating systems.The discussion also covered concerns about the strictness and security properties related to the BIP, including the issue of choosing from duplicate keys when combining and signing records with unknown keys. The importance of Combiners in correctly combining PSBTs by agents that do not understand them was also discussed.Overall, the participants discussed various aspects of PSBT standards and considered alternatives to improve them. It is important to have public discussion before implementing proposals to refine the work and prevent toxic behavior from vested interests. While it's permissible for signers to implement any policy about what they accept in signing, allowing them to refuse to sign because there is extra metadata might make it hard to implement new features that require extra data.
