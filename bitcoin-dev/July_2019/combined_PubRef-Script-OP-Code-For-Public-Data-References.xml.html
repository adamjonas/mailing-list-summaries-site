In the bitcoin-dev community, a discussion was held regarding the implementation of a new feature that would allow scripts to reference data on the blockchain. While this could reduce transaction sizes and allow miners to fit more transactions into blocks, concerns were raised about address re-use and its negative impact on fungibility. Additionally, there were concerns about increased validation costs and growing unprunable data. Other proposals were suggested, such as batching, lightning, MAST, Schnorr signatures, and taproot/graftroot.The conversation also touched on potential attacks on the blockchain due to the use of `OP_PUBREF` in `scriptPubKey`. The attack involves replacing a recently-confirmed transaction with one that pays to a different public key via a history-rewrite attack. This attack is doable by miners, and it is suggested that 100 blocks might be an acceptable amount of time for the attack to occur, similar to the acceptance of miner coinbase maturity as "acceptably low risk." There are concerns about the size of the data set required to validate the scripts, which would not fit in memory for many devices practical for running full nodes.Furthermore, the conversation between Mike Brooks and ZmnSCPxj discussed the potential implications of using the OP_PUBREF4 operation in the blockchain. They suggest using 100 blocks, which is considered "deep enough" to risk allowing miners to sell their coins. Although dealing with larger data sets may be challenging, there are other open source projects that have accounted for the real-world limits on computability, such as Apache HTTPD's Bucket-Brigade and leveldb. The proposal aims to reduce the cost of building a Merkle tree by utilizing a dictionary of repetitive strings.
