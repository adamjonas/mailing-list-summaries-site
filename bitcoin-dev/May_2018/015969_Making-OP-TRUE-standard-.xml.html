Summary:

Luke Dashjr, a Bitcoin Core developer, suggests having a consensus rule that a bit must be set for an expected behavior and the bit may only be set when the last output is exactly 00000000000000000181. He also proposes having a template that when combined with a 0-value is always valid only if spent in the same block. Meanwhile, Rusty Russell, another Bitcoin Core developer, believes the best mitigation against UTXO bloat is to make the fees as low as possible, put a CSV delay on the to-remote output, and attach more value to the OP_TRUE output, say 1000 satoshi. However, it turns out they probably don't want an OP_TRUE output nor P2SH, because then the spending tx would be malleable. So P2WSH is preferred. Russell suggests that the network benefits from using OP_TRUE outweighs the risk, but it would be nice if OP_TRUE P2WSH spends were always considered RBF.
