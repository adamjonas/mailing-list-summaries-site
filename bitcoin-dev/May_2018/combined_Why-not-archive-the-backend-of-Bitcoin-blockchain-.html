Summary:

One user on the bitcoin-dev mailing list inquired about whether a Lightning node can access an already-existing full node instead of requiring its own dedicated instance of bitcoind. Christian Decker responded that Lightning nodes need to monitor the network for transactions and suspicious activity, but the bitcoind sitting next to the Lightning node does not need to keep an index of the transactions. Therefore, running with "-txindex" is not necessary to run a Lightning node. Pruned nodes should work as long as the current blockchain head that the Lightning node has seen does not fall into the pruned range since, in that case, it won't be able to fetch and process the blocks anymore.In response to another user named Yubin Ruan who asked about running the Lightning network without a fully-fledged Bitcoin blockchain due to space limitations on their laptop, Segue suggested the idea of archiving chunks of the blockchain periodically on at least 150 computers across seven continents to avoid continuously adding onto an increasingly long chain. This suggestion was later posted on the Bitcoin-dev list, and it was noted that the "prune" flag could be used to get a snapshot of the blockchain, but it may not be compatible with "txindex" and "rescan," potentially causing issues for lightning nodes.Additionally, it is possible for a Lightning node to access an already-existing node rather than running multiple bitcoind full nodes. One method of achieving this is by configuring `bitcoin-cli` to connect to a remote node with the `rpcconnect`, `rpcuser`, and `rpcpassword` options in the `bitcoin.conf` file. This can be done with c-lightning, lnd, and Eclair. With c-lightning, you could also use command line options when starting `lightningd`. Similarly, lnd allows you to specify the node to connect to using command line options. On the other hand, Eclair requires editing the configuration file before compiling. It should be noted that C-lightning is not yet rated for pruned bitcoind use officially, but it could be used if you installed and started running bitcoind before installing lightningd, caught up to the chain, and then installed lightningd and set things up so that bitcoind will get killed if lightningd stops running. Pruned bitcoind is not supported for c-lightning officially, thus loss of funds due to doing the above idea is entirely one's own fault.On the topic of "chapter-based" archiving, it needs to get implemented and reviewed. There are certain questions that need to be addressed, such as how do we select the archive servers, and how can we ensure that no chapter has only a small number of actual owners who could easily coordinate to deny access to historical blockchain data to those they deem undesirable? Overall, chunking the blockchain periodically and archiving it on at least 150 computers across seven continents would be a solution to allow people with limited storage space to still use Bitcoin. This suggestion was posted on the Bitcoin-dev list as well, where it was noted that pruning can be used to only store the last X MB of the blockchain while still keeping the UTXO set, which takes up a few GBs of space. Christian Decker, a developer at Blockstream, confirmed that Lightning nodes need to monitor the network for transactions and suspicious activity but do not need to keep an index of the transactions. It is possible to allow a Lightning node to access an already-existing full node instead of requiring its own dedicated instance of bitcoind. All three implementations support this method. Some implementations of Lightning use the additional txindex, which is not compatible with pruned mode. In response to this, Segue suggested the idea of archiving chunks of the blockchain periodically on at least 150 computers across seven continents to avoid continuously adding onto an increasingly long chain. This would allow for outdated data to be stored in an evenly distributed manner across the planet, and a widely distributed index would be needed to reconnect with a chapter if needed. Additionally, running a pruned bitcoind instance with txindex may not be useful since indexing pruned nodes does not make sense for Lightning nodes.
