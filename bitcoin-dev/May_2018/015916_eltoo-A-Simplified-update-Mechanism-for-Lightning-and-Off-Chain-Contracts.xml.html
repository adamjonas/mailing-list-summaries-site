Summary:

In response to a query regarding fixed offset in Lightning Network (LN), Jim Posen explains how it ensures safety. To explain this, he assumes a path `A->B->C->D->E` with each having a CLTV delta and CSV timeout of 144 blocks, and an HTLC with the following CLTV deltas: `A -(576)-> B -(432)-> C -(288)-> D -(144)-> E`. When the absolute CLTV approaches, channel closure is initiated for safe preimage fetching. This duration can be higher if the final destination is obfuscated by building a shadow route behind the real recipient. With eltoo, settlement transaction confirmation must precede the desired CLTV timeout delta countdown, which means that the time difference between the current time and the last HTLC's CLTV delta increases, but only as a fixed offset, not accumulating along the path. The length of the route also impacts this offset's actual value, resulting in a smaller impact for longer routes.
