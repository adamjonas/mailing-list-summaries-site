Summary:

In an email thread on the bitcoin-dev mailing list, concerns were raised about the security of P2SH (pay to script hash) used in Bitcoin transactions. Some members suggested that if there is an attack on RIPEMD160, which is used with SHA256 to create unique addresses, it could leave P2SH vulnerable. However, others argued that even if a collision is found in RIPEMD160, it would not help an attacker as they would still need to create a specific SHA256 hash for the RIPEMD160 preimage.Furthermore, there were discussions about potential vulnerabilities and attacks on ECDSA private to public key derivation and Schnorr 2-of-2 signatures. In one instance, Tier Nolan pointed out an error in a code block related to CHECKSIG and OP_DROP parameters. The discussions revolved around the importance of maintaining security standards and keeping things simple to mitigate implementation errors and side-channel attacks.In a Bitcoin-dev mailing list, the discussion revolves around the hash function used by Bitcoin and whether or not extra bytes should be added to it. Gavin Andresen suggests using RIPEMD160(SHA256()) as the hash function, but Pieter Wuille argues that 80 bits of collision security is necessary. There are concerns about collision attacks in contract setups, where an attacker can steal coins by constructing scripts with identical hashes. It is suggested that contract wallets can protect against these attacks by refusing certain types of scripts or looking for specific proof of private keys.The discussion on the vulnerability of Bitcoin's 128-bit security target has raised concerns about its crypto rule of thumb. RIPEMD, which is believed to be less well-reviewed than SHA2, has been identified as one of the exceptions to the security target. It has been suggested that using RIPEMD160(SHA256()) as the hash function for segregated witness transactions would save 12 bytes, but Pieter Wuille argued that collision security is necessary and that 80 bits of storage should be encouraged. Andresen countered that it is trivial for contract wallets to protect against collision attacks and that adding extra bytes to prevent an attack that takes 2^80 computation and 2^80 storage is unnecessary.In January 2016, Gavin Andresen discussed a specific collision attack in which an attacker creates an innocent fund-locking script and doesn't give it to the victim yet. Using Ethan Heilman's procedure, the attacker can create two scripts and find values A1 and A2 that hash the scripts to the same result with under 3*2**80 work. The approach suggests using the next private key as the result of RIPEMD(SHA256(script with pubkey)). Five attempts should give a ~96% chance of hitting a usable pair, and it should take under 15*2**80 work ~= 2**84 work, with trivial memory use.Overall, there are ongoing discussions and debates among developers regarding the security of various hash functions used in Bitcoin transactions, particularly in the context of segregated witness BIP. While some argue for the use of RIPEMD160(SHA256()) as a hash function to save space, others raise concerns about the potential vulnerability to collision attacks. Additionally, targeted collision attacks and length extension attacks can be performed with RIPEMD160, suggesting that security should not solely rely on the notion that an attacker requires 2**80 memory. Nonetheless, the consensus is that Bitcoin is currently secure and any potential attacks would require significant computational resources.
