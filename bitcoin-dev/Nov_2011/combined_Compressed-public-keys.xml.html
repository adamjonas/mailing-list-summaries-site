Summary:

A pull request (#649) has been submitted to enable the use of compressed public keys in Bitcoin. These keys are 33 bytes long and contain only the X coordinate of the point, while the value of the Y-coordinate is reconstructed upon use, which requires some CPU but only a fraction of the cost of verifying a signature. This implementation chooses only one address out of two possible addresses created by one private key corresponding to two public keys for compressed pubkeys at key generation time to avoid confusion.Old clients can verify and relay these keys without any problems as they are supported by default by OpenSSL and all alternative implementations that use OpenSSL should support them as well. However, it remains to be seen if all client implementations support compressed public keys. Additionally, there is a need to figure out how to represent secrets for compressed pubkeys. Lastly, send-to-pubkey transactions are untested for now and require attention.On November 21, 2011, Pieter Wuille, a computer programmer, sent an email requesting attention to a specific matter. He asked if all client implementations supported a particular thing, but did not specify what that thing was. To aid in testing this, he provided a link to an address that corresponds to a compressed public key. It is unclear what the purpose of this compressed public key is or how it relates to the matter that needs attention.
