Summary:

A discussion took place about the transaction format of Bitcoin, particularly focusing on finding ways to save space and make transactions faster and more efficient. One suggestion was to include the full public key instead of just the address hash in each output. However, some participants were against this idea due to scalability issues and potential incentives for chain spamming.Several possible improvements were suggested to save space, including using compressed public keys, compact signatures, and pubkey recovery. Gregory Maxwell created a table comparing these options, which showed that all proposed options were better than the current send-to-address method. However, it was noted that the advantages of using these new methods may be smaller if pruning is implemented in the future.There was also discussion about the use of QR codes and their limitations. It was suggested that standardizing and supporting public key addresses could be a better fit for large QR codes. Proposals were made for different types of transactions, and their advantages were compared.Matt Corallo strongly opposed the use of firstbits, citing scalability issues and potential incentives for chain spamming. He believed that it should never be supported, particularly as SPV clients become more prevalent.Luke-Jr expressed frustration with the discussion on recovery in OP_EVAL being dropped because "input script size doesn't matter because of pruning." He also disagreed with the proposal for full public key addresses to be required to be "compact" and use version 21, as this would introduce another address type that services will have to cope with and no currently deployed software knows how to spend or receive it.Lastly, the use of 'green addresses' was discouraged as they are considered to be a broken-by-design feature. Overall, while there were disagreements about the best approach, everyone agreed that finding a solution for improving Bitcoin transactions was important.
