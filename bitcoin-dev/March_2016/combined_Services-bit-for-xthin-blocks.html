Summary:

A proposal has been made by G. Andrew Stone to use a services bit to communicate whether or not thin block communication messages are supported. The preferred signaling mechanism is probably the one in BIP 130. The Bitcoin Unlimited client requires a services bit to indicate its capability of processing thin blocks. The proposal suggests using bit 4 as bit 3 is already reserved for Segregated Witness. However, Gregory Maxwell raises a question about whether this functionality will change peer selection or not. If it does not, he suggests using the signaling mechanism proposed in BIP 130. If it does, then he recommends writing a BIP to document the usage.Furthermore, Luke Dashjr clarifies that the BIP process was created by Amir Taaki, who was a libbitcoin developer, and is not affiliated with Core at all. Andrew Stone states that they are happy with their existing process and view the mailing list and BIP process as very Core specific. In another email thread, Stone expressed satisfaction with their own process and sees this mailing list and the BIP process as very Core specific but proposes a process for claiming service bits. It includes claiming a bit with a reasonable justification, having a finalized description of the feature within three months that lets other clients implement it, and having working software that deploys the feature within six months. The discussion also touches upon the standard method for obtaining numbers, with some suggesting that it requires writing a BIP documenting the usage.
