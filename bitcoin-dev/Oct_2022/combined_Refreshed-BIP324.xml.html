Summary:

The Bitcoin community is discussing the BIP324 protocol, which would increase the number of message types on the network. Currently, there are about 35 message types, and the addition of BIP324 would further increase this number. However, assigning message type IDs between the 1-byte message type IDs and the commands they represent has raised some concerns.One possible solution being discussed is to use a fixed initial table with the possibility of using new mapping entries in future BIPs. Another idea proposed is to introduce an encoding scheme where every alphabetic command becomes L bytes, and non-alphabetic commands are assigned a value from a set of 102 elements. There is also discussion on how to implement negotiated assignment of shortids. A proposed solution is for each peer to choose a mapping that minimizes their own outbound traffic and not coordinate with the other peer to decide a common optimal mapping. Additionally, a single global map for serialization can be used, reducing memory usage and implementation complexity.The Bitcoin community plans to start with the first approach but may switch to the third approach if collisions become a concern. Pieter Wuille has gathered statistics from two nodes showing that around 4.5% of sent bytes are bytes 2-12 of the command for an outbound-only node, and around 1.6% of sent bytes are bytes 2-12 of the command for an outbound and inbound node.Another suggestion to increase the number of message types is to use the first bit to signal a 2-byte message ID, allowing for 2^15 IDs across 2 bytes. However, there is debate about reserving a byte for one-shot messages, and it is suggested to exclude rarely-sent or at-most-once-sent commands.
