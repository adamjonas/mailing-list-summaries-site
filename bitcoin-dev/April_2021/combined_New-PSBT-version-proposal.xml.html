Summary:

A proposal has been made to introduce a new version of Partially Signed Bitcoin Transactions (PSBTs) called PSBT v1. This version will contain all input and output data for each in their respective maps, allowing PSBT to properly support adding inputs and outputs, a feature that many people have requested. Additionally, several new fields including PSBT_GLOBAL_PREFERRED_LOCKTIME, PSBT_GLOBAL_INPUT_COUNT, and PSBT_GLOBAL_UNDER_CONSTRUCTION will be added.However, there are concerns about software compatibility issues between those that support version 1 and those supporting version 2. Andrew Chow has also proposed PSBTv2, a backward-incompatible update to the PSBT protocol. It allows for inputs and outputs to be added after creation, introduces new fields, and modifies existing ones. The update splits lock time fields into two separate fields and introduces new roles such as Creator and Constructor. PSBTv2 disallows the PSBT_GLOBAL_UNSIGNED_TX field and includes a new global field, PSBT_GLOBAL_UNDER_CONSTRUCTION, to signal whether inputs and outputs can be added to the PSBT. Andrew Chow's proposal includes several important fields, such as PSBT_GLOBAL_PREFERRED_LOCKTIME and PSBT_IN_REQUIRED_LOCKTIME, which must be considered by finalizers when choosing the correct lock time for the transaction. There is also a concern about software compatibility issues between versions 1 and 2, but developers must deal with this fact of life when supporting older hardware wallets.During a recent discussion, finalizers were advised to choose the highest *_LOCKTIME field when deciding on locktime for transactions. While output descriptors were mentioned in relation to PSBTs, it was determined that they are not essential for communication with signing devices. However, one group member did question how output descriptors could fit into PSBTs.The proposed changes would allow PSBTs to be used in the construction of transactions, with inputs and outputs that can be added as required. These modifications are not backwards-compatible with PSBTv0, but parsers for PSBTv0 will still be able to deserialize PSBTv2 with only minor changes to support the new fields. If the changes are considered reasonable, a pull request will be created to modify BIP 174 to incorporate them.The bitcoin-dev mailing list serves as a valuable platform for developers working on the Bitcoin network to collaborate and discuss various aspects of Bitcoin development.
