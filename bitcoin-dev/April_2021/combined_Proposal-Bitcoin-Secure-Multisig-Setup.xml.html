Summary:

A Bitcoin Improvement Proposal (BIP) called the Bitcoin Secure Multisig Setup (BSMS) has been proposed to establish a standardized process for securely setting up multisig wallets across different vendors. The proposal introduces two layers of protection, including a temporary secret token used to encrypt communication between signers and coordinators, and a descriptor checksum to prevent malicious members from being inserted into the set. However, concerns have been raised about the effectiveness of the proposal in addressing security issues such as xpub validation and risks for stateless hardware wallets.The setup process involves a Coordinator and multiple Signers who use a two-round communication system to set up a multisig wallet. If encryption is enabled, the Coordinator distributes a shared secret or shared secrets to the parties involved for secure communication. Other experts have suggested measures to improve privacy and avoid xpub reuse in Bitcoin multisig accounts. Christopher Allen proposed several best practices to avoid xpub reuse and improve privacy, while Michael Flaxman recommended using a checksum with higher entropy to reduce xpub validation and create a better user experience for signers.Various proposals have been made to address concerns about the vulnerability of the encryption key generation scheme used in BSMS, including using PBKDF2 as the key derivation function or BIP32 seed words list to generate the encryption key. However, these solutions may add complexity and negatively impact multisig adoption if not addressed effectively. In conclusion, while the BSMS proposal aims to address concerns about securely setting up multisig wallets across different vendors, there are still concerns and ongoing discussions about its effectiveness in addressing security issues.In addition to the BSMS proposal, Hugo Nguyen has proposed a solution to address trust issues in the form of a shared secret approach where the token doubles as a session ID to establish a common state on both sides. Dmitry Petukhov has suggested an alternative approach using a key derivation function designed for passwords such as PBKDF2 instead of ENCRYPTION_KEY = SHA256(SHA256(TOKEN)), which could be vulnerable to rainbow table attacks. The BSMS proposal focuses on an interoperable multisig configuration format and outlines an encryption scheme for a setup that involves multiple signers. The proposal introduces three modes of encryption: NO_ENCRYPTION, STANDARD (64-bit nonce), and EXTENDED (128-bit nonce). To ensure maximum security, the proposal suggests two layers of protection: a temporary, secret token used to encrypt communication between the signer and coordinator during setup, and a descriptor checksum and visual inspection to prevent malicious members from being inserted into the set.The document also discusses the importance of multisig setup context and the need for unambiguous formats for wallet ID and policy maps, as well as distinguishing between a "Signer" and a "Signing device." There are concerns about xpub key reuse in multisig transactions, which could be addressed by creating an index using a PBKDF of the Account Policy. Losing multisig setup context is flagged as an issue, particularly if one device is lost in a 2-of-3 setup without metadata. The use of a Coordinator can help mitigate the O(n^2) communication problem and reduce the need for large amounts of sensitive data to be shared by simplifying the setup process. Overall, these proposals aim to provide a secure and efficient method for multiple signers to work together while ensuring the privacy and trust of all parties involved. However, ongoing discussions and further improvements are necessary to address potential vulnerabilities and ensure the effectiveness of these solutions in addressing security issues.
