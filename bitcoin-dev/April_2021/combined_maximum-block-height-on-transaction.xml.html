Summary:

The email thread discussed the challenges involved in adding a field or opcode to transactions. The proposal was to define a "hidden" field with a default value compatible with pre-softfork rules, along with an opcode that manipulates this field. However, there were concerns about the increase in CPU usage and code complexity with these changes.Another topic discussed was the possibility of specifying a maximum block height for a transaction. It was deemed to have limitations and potential risks similar to the OP_BLOCKNUMBER transaction. Instead, nTimeLock was suggested as an alternative option that allows for escrow transactions that automatically lock and go through unless revoked before the deadline. This feature is not yet enabled or used, but support is available for its implementation later.The discussion emphasized the importance of maintaining the integrity and security of the blockchain technology. Any proposed modifications must be carefully considered and evaluated before making a decision. Any changes to the protocol must undergo rigorous testing to ensure that they do not compromise the security of the system. Ultimately, the decision to implement such a feature would depend on the consensus of the community and developers behind the blockchain technology.
