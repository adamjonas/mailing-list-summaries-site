Summary:

The sorting of public keys plays a crucial role in the functioning of cryptocurrencies. Eric Lombrozo clarifies that internally, pubkeys are DER-encoded integers, while in Script, they are represented as raw integers. OP_CHECKSIG takes a DER-encoded pubkey and a DER-encoded signature on the stack. Ruben de Vries suggests the need for a standard way of sorting public keys in P2SH multisig transactions to allow easier compatibility between different services.Ryan from BitPay had earlier raised this issue and implemented lexicographical sorting on the hex of the public key in bitcore. However, there was no other library available with a sorting function at the time. In response to this, CryptoCorp recommended sorting lexicographically by the public key bytes of the leaf public keys, which is also followed by Haskoin. Jeff Garzik agrees that a micro-BIP should be created to establish a standard and get everyone on the same page.There is also a discussion about narrowing down how a multisig scripthash address should be determined. Ruben de Vries suggests sorting the pubkey by the way it is represented in the context of creating the scriptPubKey since sorting is only needed for creating the scriptPubKey. Alan Reiner sees no reason to restrict compressed/uncompressed strings, stating that strings do not have to be the same length to sort them lexicographically.In summary, the conversation revolves around the importance of public key sorting, the representation of pubkeys in Script, and the need for a standard way of sorting public keys in P2SH multisig transactions. The developers agree on the need for a BIP outlining this process, and the writer is building a list of implementations which currently support sorting.Lastly, an advertisement from GigeNET offering a free month of service with a new server in Ashburn can be found at the end of the email thread.
