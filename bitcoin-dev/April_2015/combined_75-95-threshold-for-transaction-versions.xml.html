Summary:

The email conversation between William Swanson and s7r talks about the possibility of altering transaction IDs in Bitcoin transactions. While BIP 62 approach is one way to tackle this problem, another approach is to sign the transaction in such a way that input txids can change without invalidating the signatures. However, not covering the ID allows for transaction replay, which can lead to monetary losses. Moreover, it can result in backward computation of signatures to produce limited covenant-coins. The proposal linked to there is a game of wack-a-mole with assorted masking flags, which doesn't provide tools to address the replay issue. It exposes the sender to replay even in specialized 'contract' applications where "there won't be address reuse" could be a strong assumption enforced by the application.While the number of altered txids in practice is not high, such alterations have happened in the past. Anyone can alter the txid without invalidating it, without mining power, and without access to the sender's private keys. All it requires is seeing a transaction on the network, doing a trivial modification to it, and rebroadcasting it quickly. If the modified version gets mined, it can cause financial loss. Having mining power can help.BIP62 does not make v1 transactions invalid as it is opt-in. The creator of a transaction needs to agree to protect it from malleability, and this subjects him to extra rules in the creation. Forcing v3 transactions would require every piece of wallet software to be changed.
