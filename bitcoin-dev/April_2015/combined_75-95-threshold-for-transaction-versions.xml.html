Summary:

The issue of transaction malleability in Bitcoin is the focus of various email exchanges that propose different solutions to prevent third parties from mutating transactions. One proposal involves signing the transaction in a way that allows input txids to change without invalidating signatures, while another proposes adding a consensus rule for transaction versions that marks previous versions as non-standard or invalid if the majority of recent blocks contain a certain version.BIP62 is mentioned as a solution for preventing non-signers from mutating transactions, but may require every wallet software to be changed if v3 transactions were forced. Additionally, there is discussion about the possibility of skipping version 2 and jumping directly to version 3 for transactions and blocks. Concerns are raised about the malleability of transactions on the network and who can alter a txid without invalidating the transaction. Some users express confusion about whether certain features of the Bitcoin protocol will be supported in any future transaction version or block version, regardless of changes made to Bitcoin core.There are debates about the effectiveness of different solutions, with some proposing a simple change to solve the problem and others arguing for integrated approaches that use purpose-built signature hashing algorithms. Despite differences in opinion, it is acknowledged that both proposals could be useful for enabling more complicated uses of chained transactions like micropayment channels.The conversation also touches on the risks involved in using pre-signed txes with nLockTime and the potential impact on security when building a Bitcoin contract that relies on coinjoin/txes with inputs from multiple users signed by all users after they are merged together, pre-signed txes with nLockTime, and P2SH.Joseph Poon discusses the current problems with the SIGHASH_NORMALIZED_TXID and how it would be implemented in the Lightning Network. The SIGHASH_WITHOUT_PREV_VALUE flag is suggested as a possible mitigation strategy for replay attacks.Ultimately, it is suggested to integrate multiple proposals to provide the best defense against transaction malleability and enable more complicated uses of chained transactions such as micropayment channels.
