Summary:

In a discussion on the bitcoin-dev mailing list, users are discussing the implementation of a blinded statechain protocol using Schnorr MuSig2. The goal is to have a server (party 1) that is fully 'blinded' and does not learn certain information such as the aggregate public key, the aggregate signature, or the message being signed. The security relies on trusting the statechain server to report the number of partial signatures it has generated for a particular key. The full set of signatures generated is then verified client-side.The MuSig2 protocol involves party 1 generating a private key and public key, while party 2 does the same. The shared (aggregate) public key is calculated as a linear combination of the individual public keys. To sign a message, both parties generate nonce values and compute a 'challenge' value. The final signature is obtained by combining the aggregated nonce with the sum of the individual signatures.However, concerns are raised about the vulnerability of this scheme to certain attacks. It is mentioned that MuSig2 requires multiple 'R' points per signatory to prevent these attacks. The current scheme described only has one 'R' per party, raising questions about its vulnerability.The discussion also touches on the potential vulnerabilities of blind MuSig schemes in general. An attack called "Wagner" is mentioned, which could be avoided by implementing the third round of MuSig1 involving R commitments. Alternative schemes are proposed to address these issues, including sending a commitment to the public key, nonce, and message to the server, and the server replying with its own nonce.Overall, the conversation highlights the need for further research and exploration into the vulnerabilities and possible solutions related to blind MuSig schemes like MuSig1 and MuSig2.
