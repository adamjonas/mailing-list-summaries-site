Summary:

Jeff Garzik, a developer for the Bitcoin protocol, proposed several mini-proposals in June 2011 to better prepare the Bitcoin protocol for the long term. The proposals aimed at avoiding creating four-component version numbers and using pszSubVer as a client identification string instead. He also suggested removing IP transactions in v0.4 and creating constants for protocol version while auditing code to split client version from protocol version. Garzik recognized that splitting protocol and client versions would be a difficult patch and proposed leaving the version number as is, simply not incrementing it, which would allow for a gradual switch. He suggested using a single bit inside 'nServices' to indicate the presence of an optional "capabilities" message, enabling minor protocol changes without having to change the protocol version.Garzik discussed the benefits of alternative implementations of a protocol, stating that tying together protocol and client version is sub-optimal and undesirable long term. The current, coarse-grained scheme was preferred by Satoshi, who discouraged alternative client efforts. However, protocol development had slowed, and there was at least one major alternative client in the works (bitcoinj), so it seemed fair to revisit those assumptions and preferences.In a mailing list discussion with Christian Decker, Jeff Garzik proposed these mini-proposals for improving the Bitcoin protocol. They both expressed excitement about the proposed changes. Christian also proposed calling the Mainline client the Reference client.
