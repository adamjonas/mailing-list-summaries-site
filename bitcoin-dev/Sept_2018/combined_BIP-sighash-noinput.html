Summary:

Bitcoin developers are discussing the reintroduction of the no_input feature in Bitcoin, which would allow flexibility for changing scripts attached to the output being spent. The proposal suggests introducing a new sighash flag alongside no_input, which could bundle additional flexible sighash flags simultaneously. A modification to the NOINPUT implementation has been proposed to make eltoo unilateral closes taprootifiable. There is also discussion about using `SIGHASH_NOINPUT` to make protocols integrate better with existing wallets like CoinSwap, Poon-Dryja, and Decker-Russell-Osuntokun.Christian Decker proposed a new sighash flag called "SIGHASH_NOINPUT" that removes the commitment to the previous output. This proposal is believed to be useful for Lightning as it enables simple watch-towers and allows users to outsource the need to watch the blockchain for channel closures. However, there are concerns about its potential risks, such as address reuse and confusion with existing wallets. Some developers suggest changing the name of the flag to better describe its potential dangers.The wider community input is required with regards to the implementation details such as whether to use a separate opcode or a sighash flag. Despite the potential risks, the proposal is well-received in general. The proposal is meant to enable some off-chain use-cases and should not be used unless the tradeoffs are well-known. There have been ongoing discussions on the bitcoin-dev mailing list regarding the use of NOINPUT in Bitcoin's code. The main concern is that if the same key is used for multiple inputs and one of them is signed with NOINPUT, all of them will be spent, potentially leading to the loss of other coins. It has been suggested to drop support for SIGHASH_NONE or limit it similarly to limiting SIGHASH_NOINPUT. Taproot and graftroot scripts have the same tradeoffs as NOINPUT. A new segwit version with taproot that includes an opcode was proposed.The context refers to the funding output and settlement branch of a multisig transaction which utilizes relative timelocks using OP_CSV. The transaction requires both parties' signatures to spend the update transaction, which will only occur if the input has the correct sequence numbers. However, there is a possibility of errors when using NOINPUT with SINGLE due to the inclusion of sequence numbers of all transaction inputs. To avoid these errors, the hashSequence must be zeroed out. This approach also makes it impossible to rebind to an output with an OP_CSV that requires a larger sequence number without signing with SIGHASH_SINGLE. Despite this limitation, the use of NOINPUT enables taprootifiability of eltoo unilateral closings in the current proposal.Overall, the proposal for `SIGHASH_NOINPUT` is meant to enable some off-chain use-cases and should not be used unless the tradeoffs are well-known. It is suggested that contract-specific key-pairs be used in order to avoid having any unwanted rebinding opportunities. The deployment part of the proposal is left vague on purpose in order not to collide with any other proposals. The proposal has been well-received, and the applications proposed so far are quite interesting. There are many more applications that can be enabled with this change.
