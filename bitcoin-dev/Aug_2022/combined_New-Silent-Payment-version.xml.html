Summary:

Bitcoin's Silent Payment implementation has been updated through a Pull Request (#24897) on GitHub to introduce a new version that eliminates manual steps. The new version uses a new descriptor type called "sp()" which generates a unique 'silent-payment' address with no range and exactly one key. This address can be used as one of the outputs in a transaction, alongside standard addresses. The "send" RPC will automatically identify and tweak the silent payment address. The output generated returns a standard Taproot script with HRP changed from "bc" to "sp" on the mainnet or "tsp" on testnet and signet. The transaction can contain multiple outputs, combining silent and standard addresses. A step-by-step signet tutorial is available for reviewers to test this new version easily. Despite the improvements made, some people have raised concerns about the auditability of these transactions. In response, Ali explains that the silent payments are still transactions included in publicly auditable blocks. However, the only thing that changes is that the addresses cannot be heirarchically derived with BIP44 or any other path. Some critics warn that if Bitcoin allows secret payments that cannot be publicly audited, then it would exceed the original consensus that gave Bitcoin fungibility. Overall, the new version of the Silent Payment implementation makes the process of making silent payments easier and more efficient.
