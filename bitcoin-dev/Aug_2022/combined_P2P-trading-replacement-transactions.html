Summary:

The email thread discusses the proposal of trading replacement transactions for privacy, which involves Alice, Bob, and Carol. The idea is to swap outputs with a counterparty through an API managed by Carol, using a 2-of-3 multisig setup. Alice and Bob create orders for replacement transactions, which could be matched automatically or done manually by Bob. Both parties lock their bitcoin and share outputs with each other before signing and broadcasting the transaction. Two of three multisig is created with Alice, Bob, and Carol keys. Bob locks 0.01 BTC in it and shares outputs e2, f2 with Alice. Alice signs tx2 and shares the transaction with Bob. Alice locks 0.011 BTC in it and shares outputs b2, c2 with Bob. Bob signs tx2 and shares it with Alice. Both replacement transactions can be broadcasted, and funds are released from 2 of 3 multisig with a transaction having three outputs (one to pay fee which goes to Carol).The positives of this scheme are increased privacy, while the negatives include extra fees, time, and the need to lock bitcoin with the same amount used in the initial transaction. However, there are some uncertainties on how private keys would be shared securely, and it is unclear how Taproot could help with this issue.Although this method would improve privacy, there are negatives like extra fees, time consumption, knowing details by Carol and other peers, and need to lock bitcoin with the same amount as used in tx1. However, if this method makes sense or a similar market to trade replacements in the future, it could help create a process in which a chain of replacements happens before Bitcoin reaches the destination, similar to a tor circuit. Nonetheless, the additional complexity raises questions about whether this method is worth it.
