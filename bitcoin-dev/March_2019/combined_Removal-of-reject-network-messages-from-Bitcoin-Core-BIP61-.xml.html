Summary:

Several email threads on the bitcoin-dev mailing list have discussed the handling of reject messages in Bitcoin Core. Reject messages are responses sent to network peers when a message cannot be accepted. These messages have been disabled by default since version 0.18.0 due to nodes on the network not always being trusted to send valid messages. Despite previous discussions about removing support for BIP 61 REJECT messages, they will not be removed and instead, will be disabled by default in the upcoming release. Support for REJECT messages will be removed entirely in Bitcoin Core V0.20, expected for release in mid-2020.Developers have proposed alternatives for testing, debugging, and validating transactions and blocks, which include inspecting log messages produced by a recent version of Bitcoin Core, using specific RPCs, and relying on fee estimation. The proposal is to remove "reject" messages from Bitcoin Core 0.19.0 unless there are valid concerns about its removal. Reject messages play a crucial role in confirming transactions, indicating that a transaction will not confirm, but alternative handling must be provided to ensure reliable software function.Testing the validity of a block or transaction can be achieved through specific RPCs, such as `submitblock` and `getblocktemplate` with `'mode'` set to `'proposal'`, `sendrawtransaction`, and `testmempoolaccept`. Wallets should use fee estimation to determine transaction fees and set replace-by-fee if desired instead of relying on "reject" messages. Developers have proposed alternative testing methods such as inspecting log messages produced by a recent version of Bitcoin Core and using specific RPCs to test the validity of blocks and transactions.Users have had issues with receiving reject messages for both valid and invalid transactions in Bitcoin Core, and the feature has been disabled by default since version 0.18.0. However, Andreas Schildbach via bitcoin-dev recommends re-enabling reject messages by default before version 0.18, while Marco Falke via bitcoin-dev proposes removing "reject" messages from Bitcoin Core 0.19.0 unless there are valid concerns about its removal. Dustin Dettmer suggested waiting 16 seconds after receiving the first error message before processing errors to solve the issue of not receiving reject messages, which works well in testing. Furthermore, RBF attempts with too small a fee and accidental double spends both generate the same reject code: not enough fee. Dettmer suggested that a new reject code for RBF based on too small a fee would make for a better user experience.The P2P protocol change to support the reject message should be exposed as a version increment and a BIP, according to Eric Voskuil. Despite this, David A. Harding did not think a new BIP or version number increment was necessary, as "should support" is not the same as "must support". Additionally, Bitcoin Core no longer supports BIP61 per its documentation. In summary, reject messages in Bitcoin Core have been disabled by default since version 0.18.0, and will be removed entirely in Bitcoin Core V0.20. Testing or debugging of implementations of the Bitcoin P2P network protocol should be done through specific RPCs and logging messages. Wallets should not rely on reject messages to indicate transaction propagation or set transaction fees, but rather use fee estimation and replace-by-fee if desired. If any applications rely on this feature, they are encouraged to let the developers know.
