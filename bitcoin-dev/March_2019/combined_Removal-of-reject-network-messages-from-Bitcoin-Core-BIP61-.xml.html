Summary:

The discussion in the bitcoin-dev mailing list revolves around the handling of reject messages and their importance. Andreas Schildbach emphasizes that without reject messages, software must rely on timeouts for reception of network confirmations, which is not reliable. The rejection causes are unstable and unreliable due to the validity criteria being unable to be tested independently. Rejection on low fee only occurs when the node's mempool is full. Output already spent is indistinguishable from a missing parent. Invalid signature error states are often not very useful. The idea is to take reject messages seriously but not overrate the lack of them. However, current code handles these two cases well. The user asks for pointers regarding where the code handles reject messages in Android Bitcoin Wallet and BitcoinJ but couldn't find any except for logging exceptions.
