Summary:

On September 7th, 2017, Pavol Rusnak proposed two ideas for changing bip32 version bytes for SegWit and adding an additional byte field called OutputType for wallets that do not follow BIP43. He shared his proposals on the bitcoin-dev forum and requested comments from the community. The proposal is hosted in Satoshilabs' SL repo but can be added to BIP repo as an extension to BIP32 if there is interest.In regards to the SegWit proposal, some have pointed out that this would make it only feasible within the context of BIP43. As a result, it should be considered an improvement to BIP43 rather than BIP32. Nonetheless, a solution still needs to be found for wallets that do not wish to use BIP43. Discussions also revolved around optimizing space usage while maintaining privacy and ease of use for wallets.Pavol has also suggested adding a birthday field to the blockchain for SPV wallet developers and reducing the resolution of this field to save space while maintaining privacy. Jonas Schnelli has suggested rounding down to month-level resolution for private key birthdays, which should still be precise enough to handle any likely reorg scenario. During these discussions, concerns were raised about the potential privacy implications of revealing the derivation path for non BIP-44, 45, 49 HD nodes. Andreas Schildbach and Pavol Rusnak discussed the issue of exported public key depth in Bitcoin Wallet. Thomas Voegtlin proposed solutions for wallets that do not wish to use BIP43 by adding another byte field for OutputType.Furthermore, Kabuto Samourai suggested including a 'Birthday' field in SPV wallet, which was welcomed by developers in the community. This development has been long-awaited as it aids with initial scans for SPV wallets. Additionally, test vectors were requested to be added, which would be implemented once agreed upon.Overall, the email exchanges and discussions were focused on finding ways to improve wallet functionality and maintain security. The proposals will be considered for pull request stage on bitcoin/bips.
