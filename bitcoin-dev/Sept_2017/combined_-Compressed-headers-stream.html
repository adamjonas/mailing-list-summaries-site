Summary:

Bitcoin developers are proposing optimizations to the transmission of block headers to save bandwidth. One proposal is to remove the nBits field, which contains difficulty information, from headers and include it only on headers with a height that is a multiple of 2016 or whenever it changes from the previous header's nBits. Another proposal suggests omitting prev_block instead. The discussion also covers the possibility of parallel header fetching to optimize header download and using fixed position chunks to leverage HTTP caching.Riccardo Casatta suggested an optimization to avoid transmitting the previous hash after the first header, which could result in about 40% savings in bandwidth for a long stream of headers. He also proposed an HTTP API that uses fixed position chunks to speed up synchronization for new clients. OpenTimestamps calendars plan to use this compression to give lite-clients reasonable secure proofs without relying on a full node.Developers Suhas Daftuar and Gregory Maxwell discuss including nBits in ambiguous messages by using a one-byte flag to indicate changes from the previous header's nBits. They suggest introducing a new 'cmpctheaders'/'getcmpcthdrs' message pair for syncing using this new message type while leaving the existing 'headers'/'getheaders' messages unchanged.In addition, there is a discussion about timestamping as a non-financial piggy-back application that does not directly interact with the Bitcoin economy beyond a trivial number of timestamp transactions. Peter Todd notes that timestamping can be more vulnerable to malicious miners than financial applications due to the lack of a financial feedback loop. Riccardo Casatta noted that timestamping is used to prove date and integrity of private databases containing a lot of value, and in those cases he would recommend using Bitcoin's "full security. "The discussions acknowledge the need to support the existing protocol for a long time but suggest that splitting off the headers chain-sync functionality to a new message pair seems like a nice side-effect benefit, in the long run.
