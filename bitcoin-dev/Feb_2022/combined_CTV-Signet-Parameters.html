Summary:

The bitcoin-dev mailing list has been discussing the importance of decision-making and testing before implementing any changes to Bitcoin's consensus code. The conversation revolved around the use of hard numbers to measure proposals and the number of unique spends and use cases necessary for a proposal to be considered sufficiently tested.CheckTemplateVerify (CTV), a feature that enables the creation of more complex smart contracts, was mentioned, with uncertainty about its usefulness. Participants emphasized the need to demonstrate real-life use cases and sufficient testing before implementing any changes. Jeremy Rubin outlined three questions to determine whether a new feature proposal is ready for implementation, including if usable prototypes of major use cases have been created for the new feature.While CTV has been used in many different ways and highlighted Sapio as an example of how CTV can be used for a diverse set of applications, Rubin argued that it has not yet met the standards for being considered sufficiently tested. The use cases for CTV are targeted for long-term store of value rather than a medium of exchange and measured in decades rather than minutes or days.The context explores the potential advantages of using bare CTV over Segwit v0/v1, highlighting that it could save bytes per histogram bucket, resulting in greater feerate savings for users. However, it is acknowledged that predicting the knock-on effects of such changes can be difficult and simple comparisons may not always be adequate due to the cheapness of nodes in the graph potentially changing the optimal graph. The author notes that the proposed changes are still in the "interesting thought experiment" stage and suggests that bugs such as nonstandard broadcasting would be fixed through testing.Regarding the use of bare CTV, the author notes that it only saves bytes when spending and argues that an extra 34 or 67 bytes of witness data is fairly immaterial. Furthermore, they suggest that nobody will benefit from the possibility of using bare CTV based on the closest thing to real user data. If new features are desired, old hardware may need to be upgraded to support them. The potential benefits of using Segwit v0 CTV for users with hardware modules or MPC Threshold Crypto that only support ECDSA signatures are also discussed.A new explorer for CTV Signet has been established, and Jeremy Rubin shared his experience of running a CTV signet for around a year, which saw little use. He provided details on how to sync with the signet using https://github.com/JeremyRubin/bitcoin/tree/checktemplateverify-signet-23.0-alpha. Additionally, Rubin invited developers to test the signet and requested help in setting up an automatic faucet.In another thread, AJ expressed concern over the lack of third-party experimentation on Jeremy's CTV signet, suggesting that merging the code (w/o activation) for signet would likely help users versus custom builds/parameters. Rubin agreed that real experimentation would be great and suggested that more regtest RPC test samples for regtests could be a sufficient in-between. The author of the message concurs with AJ's suggestion and believes that merging the code (without activation) for signet would be beneficial for users. They note that they regularly conduct experiments on regtest and mainnet but setting up signet requires more time and effort, suggesting that seeing more RPC test samples for regtests could be a sufficient alternative.The discussion also touched on the potential benefits of CTV, such as combining batches into a root and N batch outputs, which eliminates the need for inputs, signatures, change outputs, etc. per batch, making it slightly smaller. However, there are fee-rate situations where it would be cheaper to wait to do the payout at a low-fee time, and some participants noted that even if bare-CTV were removed from the spec, CTV in Segwitv0 would still exist.There was also concern expressed about the lack of third-party experimentation with CTV signet, with AJ asserting that proving the usefulness of a consensus change is an essential part of the "proof of work" for deploying it. Evaluating proposals and defining clear metrics for evaluating them could make the lives of key Bitcoin core maintainers easier and remove politics from technical decisions.The conversation also moved to the criteria for soft-forking a proposal into consensus code, emphasizing the importance of demonstrating real-life use cases and sufficient testing before implementing any changes. Anthony Towns expressed skepticism about the bare-CTV part of BIP-119, which he says lacks any branches and can only commit to a subsequent transaction's outputs. Despite this, there have been over 2,200 transactions on the CTV signet, with 317 being CTV spends.Finally, the group discusses the possibility of using OP_SUCCESSx in Segwitv0 CTV. The author notes that if new features are desired, old hardware may need to be upgraded to support them.
