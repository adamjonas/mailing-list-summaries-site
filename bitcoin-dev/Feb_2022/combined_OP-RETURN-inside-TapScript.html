Summary:

After the activation of Taproot, it is now possible to store data on-chain without the need for separate OP_RETURN outputs. Instead, users can create an "OP_RETURN" branch in the TapScript to attach data to a transaction and store it off-chain, allowing for more than 80 bytes of data to be stored for free. This change has many use cases, including committing to an ECC point, which allows users to tweak it to commit data inside it and store up to 1.5GB of data using OP_RETURN or even an entire merkle tree of hashes.However, non-deterministic tweaking of an ECC point can make it harder to recover from backup without knowing the full commitment. Zac proposed a new approach to Bitcoin transactions to save space by not requiring outputs and fees to be specified in data-transaction. The conversation also touched on saving space in UTXO systems like Bitcoin, with Zac proposing partial spending of a UTXO without specifying an output or fee. ZmnSCPxj argued against this proposal, stating that the core design philosophy of the UTXO system requires either complete consumption or no touch at all, and fees are always implicit in Bitcoin.In addition to the discussion about storing data on-chain, there is also talk about improving efficiency when paying for data on the blockchain. Zac suggests embedding the data into the same transaction used to pay the miner would be more efficient. However, this raises the question of how one would pay a miner to include the data on the blockchain if it is not embedded inside a transaction. The pros and cons of both approaches are discussed, and any softfork would likely spend more bytes than it could save.The conversation also notes that publishing the hash of the data is typically done, not the actual data itself. While committing to data has many use cases, publishing data can be expensive and useful only for the transaction maker. However, there are new possibilities with TapScript that could potentially make OP_RETURN obsolete by using unspendable public keys to push data.Overall, there are ongoing discussions about improving efficiency when it comes to storing and paying for data on the blockchain while considering solutions that minimize waste. The implementation may need to change to place the "data" field in "createrawtransaction" into a TapScript instead of creating a separate OP_RETURN output. It's important to consider these solutions to maximize the potential of the blockchain technology and minimize its drawbacks.
