Summary:

In May 2014, Bitcoin Core developers discussed the possibility of offering statistics with the installer for running a wallet-less node as an OS background service. Jameson Lopp suggested using Graphite to generate charts and chose StatsD because it sends stats via UDP, which does not affect the node's performance. However, he noted that his approach had dependencies on several other pieces of software to do all the heavy lifting of stats aggregation and chart rendering. Wladimir suggested that since the chart rendering happens client-side, the web service just has to collect and provide the data, and serve static HTML/JS files. He also emphasized that implementing things like web services in C++ is not realistic given the time constraints and the great already-written code that is out there. Therefore, something like Python would be necessary for the stats service. Mike Hearn suggested exporting stats over regular TCP and making them public so that anyone can listen to the stats feed for any node.The Tor network uses a similar approach to make its statistics public through torstatus.blutmagie.de. The network operates on a centralized but theoretically federated trusted directory service, which is responsible for much of the data that is not related to relay. Semi-trusted bandwidth authorities generate data by measuring node performance back to the directory authorities, similar to how Bitcoin DNS seeds work. The Tor network's P2P architecture is somewhat similar to how the Bitcoin DNS seeds work but more extensive. However, there are differences in authentication as Tor directory authorities are authenticated while DNS seeds are not. Additionally, Bitcoin places a greater emphasis on decentralization than Tor does.Nelson Castillo suggested using SNMP as there are many existing tools written for it. However, Wladimir expressed concern that even implementing SNMP on a client device/application is complex and suggests that registering custom OIDs can be complicated. Jameson Lopp created a Bitcoin Core fork that outputs statistics to StatsD. Pavol Rusnak asked about the complexity of the patchset and whether it would be possible to merge it into the mainline and enable compilation of this feature conditionally by some build-time option.Charlie Shrem shared his 9-step guide for installing a full node in a universal way. In an email conversation between Jameson Lopp and Wladimir, they discuss the possibility of offering statistics with the installer for running a wallet-less node as an OS background service. As an optional tool, it should be external and not part of bitcoind itself. Statsd expects a simple packet format, making it easy to experiment with. To further improve the patch, it is suggested that someone can obtain a feed of the stats packets via TCP by using ssh tunnelling to their host. Once a feed of simple stats packets is easily accessible, individuals can try different kinds of aggregation and monitoring software from their desktops.Bitcoin Core developer Jameson Lopp created a fork of the platform called 'Statoshi' which outputs statistics to StatsD. The fork was made in order to gain insight into what messages and requests a node is processing, and feedback from users is appreciated. Wladimir, another Bitcoin Core developer, suggested incorporating this feature into future node-only installers so that users can track their nodes. However, Lopp suggested packaging other software along with the installer as his route has dependencies on several other pieces of software for stats aggregation and chart rendering.In a conversation between Wladimir and Mike Hearn, they discussed the idea of exporting stats over regular TCP, which would allow anyone to listen to the stats feed for any node. They also touched on the similarities between how Bitcoin DNS seeds work and Tor directory authorities, and the possibility of having authenticated "directory authorities" in the Bitcoin source code. Additionally, Python was considered necessary for the stats service since implementing web services in C++ is not realistic given the time constraints and great already-written code that is out there.The conversation among Bitcoin Core developers revolves around simplifying the setup of components for Graphite and StatsD, which are tools used for stats aggregation and chart rendering to gain more insight into what messages and requests a node is processing. The idea is raised to simplify the process by running a community StatsD and Graphite instance to allow for aggregate statistics across many nodes without individual node operators having to set everything up themselves. There is also discussion about offering statistics with the installer for running a (wallet-less) node as an OS background service. In terms of packaging other software along with the installer, it depends on the amount of stuff required, but it should be external and not part of bitcoind itself.Finally, the email thread ends with a message about a Perforce event focused on SCM systems.
