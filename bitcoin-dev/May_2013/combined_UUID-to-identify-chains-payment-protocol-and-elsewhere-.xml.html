Summary:

Bitcoin developer Mark Friedenbach proposed the use of Version 4 UUIDs to identify coloured coins and chains. The proposal involves using random bits extracted from the double-SHA256 hash of the genesis block of the chain to create a unique identifier for any chain. Jeff Garzik suggested a bitcoin-specific Version 6 could be necessary, while Melvin Carvalho argued that an out-of-band algo/hash would work if there was a one-to-one relationship between the object and UUID. Gavin Andresen voted against the proposal, stating the payment protocol would become at least three BIPs. Bitcoinj discussed creating a global naming system using Version 4 UUIDs, with Jeff Garzik suggesting a bitcoin-specific Version 6 may be necessary. MULTI, Inc. and Gavin Andresen, Chief Scientist at Bitcoin Foundation, have both shared their opinions on proposals to use UUIDs for identifying chains and coins in the payment protocol. Mark Friedenbach proposed using Version 4 UUIDs with random bits extracted from the double-SHA256 hash of the genesis block of the chain. He argued that namespaces become more important as registries grow and suggested allowing it to be a URI. Jeff Garzik proposed a bitcoin-specific Version 6 and discussed the possibility of multiple currencies using the same blockchain. Gavin Andresen noted that the protocol has a UTF-8 encoded string identifying the network as "main" or "test". Melvin Carvalho suggested an out-of-band algorithm or hash could work if there was a one-to-one relationship between the object and the UUID. However, Jeff disagreed, stating that it would go against the point of the Bitcoin consensus algorithm. The Bitcoinj project has also discussed using Version 4 UUIDs with example identifiers such as mainnet, testnet3, and namecoin. Despite attempts to avoid it, humans still require a readable shortname string to make it usable. In an email conversation between Jeff Garzik and Mark Friedenbach, they discussed identifying chains/coins using UUIDs. Friedenbach suggested using Version 4 UUIDs with the random bits extracted from the double-SHA256 hash of the genesis block of the chain to remain compliant with RFC 4122. However, Garzik proposed a double-SHA256 Version 6 as it is more applicable for identifying chains/coins. He gave examples of chain identifiers for mainnet, testnet3, and namecoin, stating that humans will want a side-by-side mapping to make it readable and usable to them despite this proposal wishing to avoid it. Friedenbach explained his upcoming colored coin architecture that allows issuance of new coins for a fee using a special category of transaction. The hash of that issuing transaction would then be used to generate a UUID identifying the asset for the payment protocol and other purposes as well. Melvin Carvalho suggested that an out-of-band algo/hash could work if there was a one-to-one relationship between the described object and the UUID. He also mentioned that in some cases, the gensis block may not uniquely identify a coin. However, Jeff Garzik countered this by saying that it seems unlikely for two different genesis blocks to have the same hash.Melvin Carvalho and Jeff Garzik further discussed the possibility of two coin ecosystems having the same genesis block. Melvin suggested that an out-of-band algorithm or hash could work as long as there was a one-to-one relationship between the described object and the UUID. Jeff disagreed, saying that it would have bad side effects and go against the point of the Bitcoin consensus algorithm. In another email conversation dated May 22, 2013, between Luke-Jr and Jeff Garzik, they discussed the possibility of multiple currencies using the same blockchain. They agreed that encouraging such a use case would be beneficial for altcoins that fit within the scope of Bitcoin or another existing altchain. However, they also agreed that they did not want to support cases like a fork that leaves the genesis block intact and leaves the subsidy at 50.0 BTC forever.
