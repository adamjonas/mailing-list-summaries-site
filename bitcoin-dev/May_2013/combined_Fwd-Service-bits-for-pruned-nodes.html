Summary:

In an email thread on Bitcoin development, Jeff Garzik defended the storage format of Bitcoin protocol blocks as the most efficient and supported format for multiple applications. However, Andy Parkins suggested adding support for more Bitcoin-protocol-oriented HTTP requests to allow any client to supply the same interface. This would create a block explorer's raw mode in every bitcoind, making it easier for light clients to find the block containing a particular transaction. Additionally, support for HTTP POST/PUT of signed transactions and block announcements was suggested.Garzik suggested using the current format of bitcoind which is tied too much to the storage format of bitcoind. He also dismissed the alternative of giving access to a file on disk since it hardly needs discussion or the involvement of bitcoind, as Apache can do it right now. However, he acknowledged that a generalized HTTP REST query protocol would be beneficial, though it was off-topic for this thread. In the context of this thread, all current users require the native binary block format.Regarding finding an ideal domain name to use for caching content, Garzik suggested using the existing DNS seed mechanism with :8333 added after the name. Andy Parkins suggested that this would make it easier for light clients to find the block that contains a particular transaction. Rebroad from sourceforge suggested an HTTP/HTTPS protocol for block downloading, which Garzik supported, but he mentioned issues with large files when using HTTP and emphasized the need for proper headers for web caches to function properly. He suggested using the format currently used by bitcoind for raw data, limiting the size to below 1GB, and adding a small metadata download to serve the raw block files. However, Garzik argued that the proposal to enable HTTP POST and PUT of signed transactions and block announcements would require a whole new client interface and was far outside the scope of an efficient HTTP protocol that downloads blocks. He also suggested the need for a well-defined HTTP-retrievable layout with proper headers for web caches to function properly. In another email thread, Garzik proposed the idea of a generalized HTTP REST query protocol, which was acknowledged as useful but off-topic for the current thread. Overall, the conversation involved various suggestions and proposals for improving the efficiency and accessibility of Bitcoin protocol blocks through different storage formats and HTTP requests.
