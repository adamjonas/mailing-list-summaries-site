Summary:

A proposal for a secure and convenient backup method using shuffled decks of cards was discussed on the bitcoin-dev mailing list. Users must shuffle their deck of cards at least seven times to achieve true cryptographic security, and write out the order in comma-separated format. They then SHA512 it to create a seed which can be interpreted based on the specific use case. However, SHA512 alone is not enough for generating a master key as it requires an additional condition. The writer suggests a bring-your-own-entropy approach, where users generate entropy by shuffling a physical deck of cards that provides enough space to be secure against collision and brute force attacks, with 52! different shuffle orders. A suggestion is made to use 50+ six-sided dice rolls to ensure sufficient entropy. The writer also notes that a shuffled deck of cards can serve as a physical backup that is easy to hide with plausible deniability, but it is important to prevent someone from mixing up the order. To address this issue, the writer suggested drawing a diagonal line on the side of the deck from corner to corner using a permanent marker. This way, if the cards ever get mixed up, one can put them back in order by making sure the diagonal line matches up. Additionally, the writer questions how this method could be used to seed a hardware wallet that requires a BIP39 seed, proposing a scheme that takes cards and generates entropy, which can then be used to generate BIP39 or non-BIP39 seeds. He references Ian Coleman's BIP39 site, which already supports card-based entropy generation and provides a link to the source code. Finally, the writer corrects a previous calculation of entropy, stating that log2(52!) is actually 225.58 bits, not 219. Discussions on the bitcoin-dev mailing list suggest inventing/formalizing a scheme that takes Cards -> Entropy so that one can generate BIP39 and non-BIP39 seeds. Prior art was noted in Ian Coleman's BIP39 site which already supports card-based entropy generation.
