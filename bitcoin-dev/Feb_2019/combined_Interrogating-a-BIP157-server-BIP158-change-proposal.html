Summary:

In recent discussions on the Bitcoin-dev mailing list, Tamas Blummer proposed changes to BIP157 and BIP158. He suggested implementing a base filter that only includes spent outpoints and the scriptPubKey of each output for each transaction in a block, aside from all OP_RETURN output scripts. This proposal would simplify interrogation of filter servers and serve wallets well. However, Olaoluwa Osuntokun disagreed with the proposal, suggesting adding new filter types instead.Jim Posen favored option two, which involves clients tracking multiple possible filter header chains and considering the union of their matches. Additionally, Tamas expressed skepticism about the commitment of a filter into Core anytime soon. Despite some disagreements on the best approach, there is an ongoing effort to explore and address these issues in order to improve the reliability and efficiency of light clients and the Bitcoin network as a whole.Blummer also proposed modifying BIP158 to have a base filter that contains spent outpoints and the scriptPubKey of each output, aside from all OP_RETURN output scripts. Gregory Maxwell expressed support for the proposal but noted that it may require additional development work. Other discussions on the bitcoin-dev mailing list also explored different possibilities for improving client protocol and filter correctness, including introducing new P2P messages or committing filters into the chain via witness reserved value or coinbase OP_RETURN.There were also discussions on introducing a new P2P message to retrieve all prev-outputs for a given block and verifying the scripts against the block by executing them. Laolu Osuntokun suggested using input scripts for multi-party protocols such as atomic swaps, while Tamas Blummer proposed implementing an interrogation by downloading blocks at checkpoints since it is much simpler than following multiple possible filter paths. Lastly, there were discussions on the decision to remove outpoints from the filter and the need for new filter types in the future.Despite some disagreement and skepticism, there is significant interest from bitcoin developers to eventually commit an output filter with a significant investment of effort in improving the proposal. Tamas has put in a significant amount of effort to improve an output filter proposal. When he asked if there was anything technically wrong with his submission, he was told that there was not. However, he was also informed that a commitment to his proposal was not yet in sight. Tamas had written code, gathered and shared data, and demonstrated an understanding of non-technical issues. Despite this, he was still unsure why his proposal had not received a commitment.
