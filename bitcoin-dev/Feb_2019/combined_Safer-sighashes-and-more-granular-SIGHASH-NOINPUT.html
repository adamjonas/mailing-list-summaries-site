Summary:

The Bitcoin development community is engaged in discussions around several proposals that aim to improve the security, flexibility, and efficiency of the Bitcoin protocol while balancing the tradeoffs between these factors. One such proposal by Pieter Wuille includes adding three new sighash flags: SIGHASH_NOINPUT, SIGHASH_NOFEE, and SIGHASH_SCRIPTMASK, along with a new opcode OP_MASK. There are concerns raised around the necessity of the SIGHASH_SCRIPTMASK flag, the usability, and security of OP_CODESEPARATOR with NOINPUT, and the need for an alternative approach to witness weight malleability.In addition, there are proposals to replace OP_CODESEPARATOR with an internal counter, use Merklized Abstract Syntax Trees (MAST) as a replacement for redundant operations like OP_IF, and add additional features to the Segwit version's Sighash. Pieter Wuille suggests signing the amounts of all inputs instead of explicitly signing the transaction fee, creating a new `hashValues` field to the hashed representation, with `hashValues` being the double SHA256 of all values. He also proposes a combined proposal that includes all existing sighash flags, plus NOINPUT and MASK, encoded in two bits.Overall, the discussions among Bitcoin developers center around finding solutions to improve the security, flexibility, and efficiency of the Bitcoin protocol, while balancing the tradeoffs between these factors. The focus is on simplifying wallets, reducing complexity, and making it easier to implement covenants/vaults given the uncertainty of which features will be deployed on mainnet.Bitcoin developers have been discussing proposed changes to Bitcoin Script on the bitcoin-dev mailing list. One issue discussed is witness weight malleability, caused by certain opcodes accepting variable size inputs provided by the puzzle solver. The proposal includes the addition of "signature covers witness weight" to make it easier to compute the final witness weight prior to signing, especially in complex multi-party signing protocols. Another issue discussed is the NOINPUT feature, with some developers expressing concerns over its safety and security. The discussion revolved around finding a replacement for OP_CODESEPARATOR, which is currently used to create a signature that covers a specific branch within an input Script code. Suggesting alternatives, a few ideas were discussed such as adding an internal counter or dividing every Bitcoin Script program into blocks delimited by control flow operators. Additionally, the developers proposed several changes to the sighash for future segwit versions, including committing to the absolute transaction fee and scriptPubkey in addition to the scriptCode. They also considered the idea of signing witness weight, which has been previously brought up.The discussions aim to improve efficiency, usability, and security measures in the Bitcoin protocol while maintaining core protocol simplicity. The trade-off between adding security measures and increasing software complexity was debated, with prioritizing core protocol simplicity being recommended.
