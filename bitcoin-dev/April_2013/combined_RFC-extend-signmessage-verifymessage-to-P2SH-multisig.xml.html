Summary:

In April 2013, Peter Todd proposed extending the signmessage/verifymessage feature to support messages signed by n-of-m keys or P2SH multisig addresses. He suggested a new header byte format for multisig signmessage signatures that would be backward compatible and keep the size down to an absolute minimum. The format would have three forms for each signature or key: sig, compressed key, or uncompressed key. All pubkeys must be provided despite not being used for a given signature to allow the P2SH address to be reconstructed. However, figuring out HD wallet changes was necessary before making any such extensions as it is essential to avoid signing with the same private key twice under any circumstances. Alan Reiner responded by suggesting the addition of an ASCII-armored format for message signing in response to the demand for Armory to become compatible with Bitcoin-Qt signing. This format would encode the signed message next to the signature, eliminating any ambiguity about what was signed. Users could copy an ASCII-armored block of text into the client app, which would then display a pop-up window showing the message along with the valid signature from a particular address. Following RFC2440, section 7 "Cleartext signature framework," Peter Todd recommended implementing this method. Alan agreed to implement the format in Armory but noted that there might be an issue passing \r's through the RPC interface since the RFC specifies CRLF line endings.Peter Todd also suggested creating a new address format that represents a chain of keys instead of one key or combination of keys to ensure privacy and safety. Luke-Jr recommended figuring out HD wallet changes before making any extensions to prevent signing with the same private key twice under any circumstances. Todd agreed that multisig signmessage would be useful and suggested extending the RPC signmessage command with an optional "existing signature" option.
