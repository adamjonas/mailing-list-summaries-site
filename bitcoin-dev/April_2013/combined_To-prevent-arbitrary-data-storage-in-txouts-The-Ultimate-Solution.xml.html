Summary:

In 2013, there were various discussions and proposals related to Bitcoin's P2SH addresses and transaction hash algorithm. One proposal suggested creating a new address type called P2SH^2, which would limit data stuffing to brute forcing hash collisions and eliminate non-prunable locations for users to store data in the blockchain, except for values. It was also believed that it could discourage miners from mining P2SH^2 without relaying the inner P2SH address.Another proposal was made to change the transaction hash algorithm to extend the Merkle tree into the transaction outputs/inputs, allowing pruning of transactions bit by bit while still being able to serve nodes requesting proof of their UTXO. However, there were concerns about potential technical risks involved in implementing UTXO fraud proofs and how they could cause an arbitrarily deep re-org.In addition, the email conversation between Peter Todd and Gregory Maxwell discussed the difficulty of paying further funds to a random txout seen on the blockchain, which could potentially be solved by making it difficult to pay to random transaction outputs. There were also discussions about using P2SH to provide the full 256 bits and proposing a new scriptPubKey that could make it difficult to pay further funds to a random transaction output seen on the blockchain.In yet another email exchange from 2013, Robert Backhaus and others discussed the use of P2SH addresses in Bitcoin and confirmed that they do not prevent value commitment, but only prevent data storage. While it is possible to store some data in values, it incurs a computational cost that increases exponentially with the amount of data stored per output.Overall, these discussions and proposals aimed to improve Bitcoin's functionality and security while addressing concerns related to data storage and transaction hash algorithms.
