Summary:

Automated tests are conducted on every proposed patch to the reference software, and pulltester comments on GitHub requests and produces logs. However, the Bitcoin community has expressed the need for integration testing for Bitcoin. Adam Ritter proposed this in a 2013 post on the Bitcoin-development mailing list and his plan can be found on a GitHub repository proposal. The aim is to make the code more stable while enabling testing of attack scenarios and easier refactoring and extension of code.Gregory Maxwell recommended using blocktester, a system based on BitcoinJ that simulates various network scenarios to test the boundaries of different rules of blockchain validation protocol. These tests are run as part of the automated tests on every proposed patch to the reference software. Pulltester comments on GitHub requests and produces logs.The Bitcoin community also established a public secondary test Bitcoin network called Testnet where anyone can access it by starting the reference software with testnet=1. Most of the testing work has been focused on validating the blockchain behavior because it has serious systemic risk. However, JSON-RPC behavior is also interesting but less important.To exercise the wallet code, integration tests are needed in bitcoinj. In the past, unit tests were not realistic enough and did not simulate real network scenarios, which resulted in bugs. Integration tests should handle re-orgs correctly and update the wallet properly in all cases. A standalone tool that stress-tests the system would also be useful to catch bugs earlier. This tool could randomly generate chains of transactions and could be written using bitcoinj or the raw transaction APIs on bitcoind.In summary, the Bitcoin community has developed multiple systems for testing and validating blockchain behavior, including the use of blocktester and Testnet. The importance of integration testing has been emphasized and efforts are being made to implement them effectively.
