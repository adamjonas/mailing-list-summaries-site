Summary:

The Bitcoin community is proposing a solution to make descriptors more user-friendly and efficient. Descriptors have become popular for working with master public keys, but users find it difficult to copy and paste them due to the presence of commas and parentheses that prevent highlighting by double-clicking. A proposed solution is to base64 encode the descriptors, resulting in a text blob as the master public key without any extra details to bother users. However, there is an issue with the descriptor checksum. If there are typos in the base64 text, it may decode into multiple character errors in the descriptor, which could be problematic for the checksum. Therefore, it was suggested to attach the checksum to the end of the base64 text.These ideas were generated from discussions with achow101. Chris Belcher proposed that descriptors should become the standard for master public keys, but there is a downside as users cannot easily copy-paste them due to containing parentheses and commas which prevent highlighting by double-clicking. A possible solution is to base64 encode the descriptors, resulting in a text blob as the master public key without any extra details to bother users. However, some members of the community prefer Bech32, Base58, or Base62 due to their user-friendly characteristics.The use of Base64 encoding is not optimal for URL/QR code friendly purposes, and instead, alternatives like Base58 or Bech32 could be used. Some members expressed skepticism about the usefulness of payment descriptors for quick transactions that QR codes work well for. In terms of error detection, QR codes have built-in error detection using its own BCH code scheme which mitigates issues when used there. Nonetheless, it was suggested to define derivatives of bech32 that retain error detection properties for longer message lengths. The proposal addresses concerns that users will be unwilling or unable to use descriptors due to their code-like syntax and use of commas, parentheses, and brackets. Base64 encoding was suggested as a possible solution, but some members of the community prefer Bech32, Base58, or Base62 due to their user-friendly characteristics. However, the checksum scheme used by Base58 and Bech32 already exists, while descriptors have their own BCH code for descriptor checksums optimized for their length and character set. This can be repurposed to work with any encoding scheme that is suggested, as long as the encoding's character set is covered by the descriptor checksum character set. Descriptors' checksum character set covers all characters on a standard keyboard, making it suitable for future expansion of descriptors.Overall, this improvement would make using descriptors more user-friendly and efficient. Further information on the descriptor checksum can be found here: https://github.com/bitcoin/bitcoin/blob/master/src/script/descriptor.cpp#L26.
