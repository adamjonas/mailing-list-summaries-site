Summary:

Bitcoin developer Luke Dashjr has proposed a standard for the exchange of fiat currency information in Bitcoin Knots to simplify interoperation between wallets and exchange rate providers. However, there is an issue with the proposal requiring XBT instead of satoshis for Bitcoin amounts.On the bitcoin-dev mailing list, Tim Ruffing suggested the need for periodic messages system in longpolling so that clients can request updates from a server. Clients cannot distinguish between a situation where the value is still within requested bounds and when the connection has dropped without periodic messages. Long polling is useful for Bitcoin as rates can change significantly in a short period of time. Luke-jr noticed that Electrum had many implementations for each exchange rate provider due to lack of a common format for such data while investigating the possibility of adding fiat currency information to Bitcoin Knots. In response, he created an initial draft of a BIP (Bitcoin Improvement Proposal) that would standardize this process so wallets and exchange rate providers could easily interoperate without having to reimplement the same thing many times.In recent discussions on the bitcoin-dev mailing list, it was suggested that servers should be authenticated at a minimum to prevent attackers from manipulating exchange rates. HTTPS would be sufficient for this purpose as it operates at the same layer.Various discussions on the bitcoin-dev mailing list have addressed the need for standardization of fiat currency information in Bitcoin Knots to facilitate interoperation between exchange rate providers and wallets. A draft of a Bitcoin Improvement Proposal (BIP) has been created, but it requires using XBT (as BTC) for Bitcoin amounts, which may not be ideal. The proposal also lacks critical components such as authentication for preventing manipulation of exchange rates by attackers.There is a suggestion to simplify the proposal by allowing clients to choose the time scale of data and clearly defining the type field without flexibility. Clients should also be able to request periodic messages from the server during longpolling to avoid wrong decisions due to dropped connections. The use of line-feed format in JSON has also come under scrutiny for being a premature optimization at the cost of interoperability.In another discussion, it was suggested that historical rates are necessary as all incoming transactions are historical, and wallets are often offline when these transactions occur. However, its usefulness is questionable, and clients should be allowed to decide on which time scale the data should be displayed and clearly define the type field. To address the bounds issue, Ruffing specified that when from/to don't have an exact record, the previous/next (respectively) is provided. The client or server cannot specify granularity in the current draft.
