Summary:

Pieter Wuille, a developer, has proposed a new Bitcoin Improvement Proposal (BIP) for native SegWit addresses to replace BIP 142. These addresses are optional but offer better efficiency, flexibility, and user-friendliness. The format of these addresses is base 32 with a simple checksum algorithm that provides strong error detection properties, and reference code in multiple languages as well as a website demonstrating it are included.Meanwhile, the discussion on Bitcoin-dev mailing list was about the ability of non-native English speakers to communicate BTC addresses with English speakers. In recent weeks, two new reference implementations (Haskell and Rust) have been contributed and C++ and Go ones are underway. In response to the suggestion of using Base 43 for QR code encoding, Peter Todd expressed concerns about the downsides of using non-alphanumeric characters and questioned the need for transmitting addresses via voice communication. He suggested using something like "Base 64url" or preferably Base 94 for shorter QR codes. Additionally, Pieter Wuille addressed concerns about transmitting addresses via voice and emphasized his aim to address all places where humans interact with addresses, while also mentioning that QR codes can be made more compact, but that is not the goal of the BIP.In email exchanges, Pieter was advised to add a section to the BIP that stated that the Human Readable Part (HRP) in the proposal should be lowercase, and Russell O'Connor suggested adding a note that the HRP should be specified in lowercase or at least mention that uppercase and lowercase HRPs are considered equivalent and will be canonicalized to lowercase during validation. The HRP is required to contain 1 to 83 US-ASCII characters, with each character having a value in the range of 33-126, and its validity may be further restricted by specific applications. Bech32 forbids mixed-case and converts everything to lowercase, so it is good to warn against using uppercase in the HRP.Finally, there was a discussion about the potential need for a new type of address in the future, which may not be representable as existing address types, to avoid confusion and lost privacy and funds. The writer argued that if a new type of address can be encoded with bech32 in the future, a new address type may be needed anyway.While Pieter Wuille requested a BIP number assignment for his proposal, people have contributed two new reference implementations (Haskell and Rust), and a C++ and Go one are underway.
