Summary:

Recently, there have been discussions on peer-to-peer implementation details in Bitcoin Core's Segregated Witness code that require clarification. After SegWit activation, Bitcoin Core nodes will not download blocks from non-SegWit peers to prevent downloading invalid blocks without the necessary witness data. However, non-SegWit blocks will still be evaluated under the same rules as pre-SegWit versions of the software. At the consensus layer, non-SegWit blocks that are valid to older nodes are also valid to SegWit nodes. But at the peer-to-peer layer, SegWit-enabled nodes will only download those blocks if announced by a SegWit-enabled peer. The phrase "protocol requirements of SegWit" is confusing because it involves two different layers: the consensus protocol layer and the peer-to-peer protocol layer. In neither layer is the behavior of not downloading blocks from non-NODE WITNESS peers a "requirement". It is an implementation detail in the Bitcoin Core code that alternate implementations compliant with BIP 144 could implement differently.If a miner wants to continue mining post-SegWit activation but using pre-SegWit software, they would need to relay their blocks to a SegWit-enabled peer. There are several ways miners can do this, including using the RPC call "submitblock" on a SegWit-enabled node or explicitly delivering the block to a SegWit node over the P2P network. It is incorrect to say that "A miner running pre-SegWit code will produce blocks that no SegWit node will ever receive." Even though SegWit nodes will only download those blocks if announced by a SegWit-enabled peer, there are bridge nodes on the network, and miners have other options to relay their block.Therefore, it is important to highlight this detail and better document this behavior as it is essential for miners to be aware of. The issue deserves specific mention in a BIP. Although the testnet setup has only a few nodes acting as bridges, mainnet already has many systems that act as effective bridges today, including several relay networks in use that bypass the P2P network.In summary, Bitcoin Core nodes will not download blocks from non-SegWit peers after SegWit activation. Non-SegWit blocks are evaluated under the same rules as prior versions of the software and are valid to SegWit nodes. SegWit-enabled nodes will only download those blocks if announced by a SegWit-enabled peer at the P2P layer. Miners using pre-SegWit software can still relay their blocks to a SegWit-enabled peer through various means, such as using the RPC call "submitblock" or running a bridge node. It is important to better document this behavior for miners to be aware of, although there are currently bridge nodes on the network and miners have other options to relay their blocks.
