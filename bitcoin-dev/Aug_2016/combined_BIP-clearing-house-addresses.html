Summary:

The recent hack on Bitfinex has sparked discussions about the security of Bitcoin exchanges. One proposed solution is to create a new address type with a reversal key and settlement layer to prevent unauthorized transactions. However, this conflicts with Bitcoin's design, which makes transaction reversals impossible. Luke Dashjr suggests using nLockTime withdrawals instead to prevent unauthorized transactions, but this approach has limitations for centralized exchanges. Alternatively, TierNolan suggests creating a new type of output script that forces the redeemer to use a designated output script template. Many consider this to be the best solution as it allows people to be forced to send coins into another transaction with "relative lock-timed" outputs.Matthew Roberts also proposes adding an opcode called SPENDTO, which requires an output to be set to a specific script, allowing for a purely local check and can be used with check sequence verify to lock the spending script for a period of time. Centralized exchanges play an important role in the Bitcoin ecosystem by allowing efficient price discovery and improving liquidity, but they also have security concerns. Matthew Roberts proposes a protection mechanism for hot wallet coins. The output script of the hot wallet would only allow coins to be sent to a new transaction whose output script could only be redeemed after N confirmations, but can also be recovered to a fixed fail-safe address before the time-lock is reached. Deposits to the hot wallet would need to be encumbered by the same protection so that the time-lock to redeem coins can be applied to every new transaction trying to move coins. This approach is good for crucial services like wallets, gambling websites, e-commerce websites, provable dead man's switches on the protocol level, and obviously exchange hot (and cold) wallets.Decentralized exchanges may offer greater security, but they are not as efficient for speculators who need to enter and exit complex positions quickly. Erik Aronesty suggests using lightning networks for margin trading to avoid centralization that is currently happening on the network. He advocates for a P2P order routing system with market makers running nodes to facilitate routing, resulting in a decentralized exchange with orders matched to users directly, and channel trades executed instantly.Matthew Roberts also discusses the usage of centralized exchanges for speculative purposes, stating that many users do not care about decentralization, security, or even the asset they are buying, but rather use these exchanges to enter and exit complex positions in milliseconds. However, using channels alongside a centralized exchange can provide the benefits of a distributed exchange. Channels allow instant funding while giving customers control over their funds. Even margin account holders appreciate the clarity of knowing which funds are under their control versus those held by the exchange. Using a 2 of 3 system where the customer, exchange, and third-party arbitration agency hold keys may be acceptable to the exchange if bitcoin funds are being used as collateral for a margin trade.The conversation also touches on the topic of protecting coins in hot wallets from theft by using output scripts that only allow coins to be sent to a new transaction whose output script is then redeemable after several confirmations. Managing risk by keeping some keys offline is an important part of key management. BitGo has implemented an "instant" system to prevent double spending, but reversible payments should be treated as such. Vaults are used to prevent theft by hackers who gain access to exchange servers.On the bitcoin-dev mailing list, Tier Nolan and Matthew Roberts discussed a clearing mechanism for exchanges, with Nolan suggesting adding an opcode that requires an output to be set to a specific value to lock the spending script for a period of time. The email conversation also explores the possibility of doing instant trades between altcoins using channels and the exchange as a hub, and the potential risks associated with a "100% reserve" company issuing fiat tokens.In summary, discussions around Bitcoin's security have led to proposals for new address types, output scripts, and protection mechanisms for hot wallets. There are also discussions around the usage of decentralized exchanges and lightning networks for margin trading. While centralized exchanges have their own security concerns, using channels alongside them can provide the benefits of a distributed exchange. Finally, the conversation touches on the importance of managing risk through key management and vaults to prevent theft.
