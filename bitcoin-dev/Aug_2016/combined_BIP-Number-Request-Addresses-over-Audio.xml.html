Summary:

A new Bitcoin Improvement Proposal (BIP) has been proposed on the Bitcoin-dev mailing list. The proposal aims to represent Bitcoin addresses over audio using a standardized method. The binary representation of the address would be chopped up into 4 or 2 bit chunks and tones generated based on those values, with error correction built into the protocol. The proposed method could be used for donating Bitcoins to podcasts and other audio content. Justin Newton suggested adding additional tones to indicate amounts in satoshis. However, there are concerns about making the signal detectable and easy to decode without interfering with speech frequencies. Using Mozart symphonies as a carrier wave has been suggested to modulate the signal.Another proposal suggests using BIP32 public seeds instead of plain addresses to protect privacy. There were discussions around revealing a public seed, which could compromise privacy, and a proposal was made to encode a BIP32 public seed rather than broadcasting addresses to prevent tracing the total amount of payments. Erik Aronesty raised a question about determining if an address was derived from a BIP32 public seed and suggested computing all possible public child addresses. On the other hand, Gregory Maxwell suggests that generating all possible addresses from a BIP32 public seed would take about 8 hours with competent software on a quad-core laptop. Discussions also centered around creating a "publishable seed" that can allow someone to derive random Bitcoin addresses and receive funds, with suggestions including homomorphic encryption and shared secrets.The bitcoin-dev community is exploring innovative ways to transmit Bitcoin payment information while protecting privacy and security. Other discussions included using trellis modulation or other convolutional codes for error correction, defining channel models to simulate various use cases for testing, using CDMA for sending multiple bits at once with orthogonal codes that sound like noise, and why telephones are considered 4-tone when DTMF has 16 tones.Overall, there is ongoing discussion about the potential use cases and feasibility of these proposals, with some suggesting further research and implementation to fully gauge their viability. The full specification for the BIP can be found on various GitHub pages, and authors are working on decoders while preparing documentation per the official BIP spec.
