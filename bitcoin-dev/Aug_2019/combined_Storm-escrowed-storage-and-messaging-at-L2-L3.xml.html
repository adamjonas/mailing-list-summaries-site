Summary:

The proposed solution called Storm, which uses the Lightning Network (LN) for decentralized data storage, is the main topic of discussion in an email conversation between Dr. Maxim Orlovsky and ZmnSCPxj. However, several issues are raised that need to be addressed before implementing the solution. These issues include insufficient/unclear description of probabilistic proof, asymmetrical encryption, and transportability over LN channels.To address these concerns, ZmnSCPxj suggests using EC for encryption/decryption and threshold ECDSA signatures. The need for an asymmetrical encryption that separates encryption and decryption keys is highlighted. They also discuss the possibility of implementing the solution with threshold ECDSA signatures.Moreover, the Deaf Bob Attack, a scenario where Bob can ignore Alice and still receive the stake and reward if he had already selected the decryption key at setup time, is raised on the bitcoin-dev mailing list by ZmnSCPxj. This issue could potentially create problems in implementing the proposed solution. To solve this problem, Dr. Maxim Orlovsky proposes adding two new intermediary HTLC transactions called confirmation and fallbacks. If Alice doesn't get the data, she can sign the HTLC fallback tx and get her money back as compensation for Bob's stake. Bob can prove that he still has the data by providing a "preimage" to the secret hashed by Alice.Additionally, Fulgurite, a project to split a channel into Lightning part and DLC (discreet log contract) part, is mentioned as a possible solution to the Deaf Bob Attack.Dr. Maxim Orlovsky has also proposed a design for distributed storage and messaging with escrow/economic incentivization that works at Layer 2 and 3 and leverages the LNP/BP ecosystem. This design allows constructing a special type of payment channel that guarantees remote data storage and retrieval while mitigating counterparty risks through economic stimulus such as stakes. The proposal came as a side-effect of joint work on RGB and single-use seals technologies, but it can be useful for other L2/L3 technologies requiring client-stored data.Interested parties are welcome to provide comments, criticisms or discuss driving Storm development forward. An issue will be opened on GitHub to continue the discussion around these proposals.
