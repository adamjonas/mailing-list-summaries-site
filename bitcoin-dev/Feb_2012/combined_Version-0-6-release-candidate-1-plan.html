Summary:

On February 6, 2012, Luke-Jr proposed to make transactions with extra data in scriptSig non-standard, expressing concern that this could harm the replacement of the spammy "green address" scheme with a sensible extra signature system. He suggested stripping extra data from transactions as an alternative solution that would have no negative impact on normal transactions. However, Gavin Andresen believes that any design relying on clients recognizing two variations of a transaction is a bad idea. The problem of green addresses spamming the blockchain can be solved through a REST-ful API provided by green address providers. This API offers functionality that allows users to declare transactions as "green" by providing their transaction ID and credentials.In addition, the head of Git has announced several major changes ready for wider testing, including bug fixes and displaying multiple outputs in GUI. A release candidate will be pulled together in the next day or two, with the goal of having at least a full month of release candidate review and testing before releasing a 0.6 final version with zero High Priority bugs. The proposed TODO list for rc1 includes making transactions with extra data in scriptSig non-standard, a modified version of fixing minimize to tray, and not voting for /P2SH/ unless -p2sh specified. However, CAddrMan: stochastic address manager was not included due to not passing sanity tests. The discussion thread will include thoughts on rolling out higher-level multisignature support.Luke-Jr raised an objection on making transactions with extra data in scriptSig non-standard on February 6, 2012, but withdrew his objection on February 7, 2012, after gmaxwell explained why it was no longer needed on IRC. Lastly, Gavin Andresen expressed interest in pulling a modified version of 755, which would not vote for /P2SH/ unless -p2sh specified. However, he did not mention what else needed to be changed for this. Additionally, he wanted to pull 787, which involved CAddrMan: stochastic address manager, but it did not pass his sanity tests. He confirmed seeing at least one addr.db corruption with this.
