Summary:

The Bitcoin community is currently discussing various proposals for creating difficulty futures instruments. One suggestion involves using a new opcode for binary options on difficulty, while another proposal entails using Taproot MAST to enable difficulty futures on-chain. The idea of using annex restrictions, as outlined in BIP-0341, was also suggested to lock in the difficulty range for participants.There are discussions about the feasibility of placing bets on specific time slices and difficulty thresholds using Powswap, a platform that allows users to bet on the future hash rate of the Bitcoin network without requiring any protocol changes. However, some members raised concerns about the need for watchtowers to prevent the loser of a bet from sweeping funds if the winner is not vigilant. Tier Nolan proposed a new method for speculation contracts to consider the ratio between future and current difficulty instead of difficulty itself. This involves dividing current difficulty by last difficulty using packed representations already included in blocks. Jeremy Rubin proposed an alternate scheme that uses the difference in maturation time between timelocks and height-locks to enable difficulty futures without changes to Bitcoin. Another suggestion involves using covenants to check output values and scripts and the op_difficulty opcode to get the difficulty. The discussion also explores the use of synthetic hashrate binaries and the possibility of making difficulty future purchases on-chain in a trustless way using covenants.Overall, there are many proposals and discussions happening within the Bitcoin community regarding difficulty futures instruments and alternative approaches to enable them without requiring changes to the Bitcoin protocol. The goal is to eventually have these implemented with lightning and eventually HFT.
