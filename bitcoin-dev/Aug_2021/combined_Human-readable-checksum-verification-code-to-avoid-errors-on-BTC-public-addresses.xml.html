Summary:

A proposal has been made to add a 3 or 4 digit verification code, called 4DC, to Bitcoin addresses in order to reduce the risk of sending funds to the wrong address and improve user experience. The proposed code would be generated as a checksum from the address and shown in all wallets next to the receiving address. However, some members of the community argue that such a code is unnecessary due to the existing built-in checksums. Despite this disagreement, proponents of the proposal argue that it could still prevent copy-and-paste errors.Pieter Wuille has suggested an alternative solution to improve the user experience while verifying Bitcoin addresses. He suggests adding a special 4-character "external" checksum on top of the built-in one or displaying 4 or 5 characters from a fixed position separately in a clear box beside or below the address. This "Quick Verification Box" would encourage the user to look at a given number of characters and make it easier to see them. There are also discussions about different ways to make Bitcoin addresses more visually distinct. One suggestion is to use a checksum for bright colors, weights, sizes, capitalizations, and/or spacing of the characters of the address. Ethereum already uses mixed case to differentiate addresses. However, there are concerns about the difficulty of getting the ecosystem to adopt such changes.Christopher Allen suggests LifeHash as an alternative to a visual checksum proposal, which includes both a visual indicator and a readable fingerprint value. LifeHash is an open-source visual hashing algorithm with desirable qualities such as high complexity, good aesthetics, and robustness when transformed to grayscale. BlockchainCommons uses LifeHash for all their projects. Agreeing on a checksum standard to generate the proposed code is needed. Once the standard is established, wallet and exchange developers can start implementing this solution. A good name for this code would be helpful for fast adoption, such as human readable checksum, verification code or 4DC. The proposed solution can also be used for other coins/networks, but ideally, each of them should have its own checksum algorithm to avoid sending funds to the wrong network.On August 28th, 2021, ts via bitcoin-dev proposed the idea of adding an additional text-based checksum to Bitcoin addresses and requested feedback from the community on the feasibility of implementing it. Pieter, a member of the community, expressed confusion about the need for such a feature. He argued that random typos are already automatically protected against through the checksum, and comparing a few characters of the address is just as effective as any other method. Despite some disagreements within the community, the proposed solution for addressing BTC transaction errors is easy to implement and could greatly improve the user experience while reducing the probability of performing transactions to a wrong address.
