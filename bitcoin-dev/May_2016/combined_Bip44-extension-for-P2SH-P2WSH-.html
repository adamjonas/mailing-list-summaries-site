Summary:

In May 2016, a discussion on the bitcoin-dev mailing list centered around improving the wallet experience. The conversation covered several topics related to Bitcoin Improvement Proposals (BIPs) and wallet compatibility. One option proposed was defining a new derivation path parallel to Bip44 but with a different purpose, allowing users to choose which account they want to add (normal or witness) for handling P2(W)SH receiving addresses with the approaching implementation of SegWit. Another option involved staying with the current BIP44 account and giving users the option to display public keys as either P2PKH-Address or P2SH address.The email thread also discussed the complexities of importing a BIP32 wallet, which some argue is still an expert job despite most reasonable wallets now using BIP39 mnemonic. Several potential problems with the process were raised, such as cross-wallet imports and concerns about security. When moving all funds to a new wallet, there is a high chance of losing funds if the old wallet has used more than 1000 addresses, which some wallets don't even create. A proposed solution involves using a little GUI tool or cli-app/script that accepts different bip32 schematics, different bip39 implementation, creates large lookup windows, creates a sweep transaction to a new address/wallet, and exports transaction history in CSV format. Despite the concerns, some wallets have implemented BIP39 mnemonic for imports, and there are plans to implement a new derivation path for wallets that is parallel to Bip44 but with a different purpose. However, there are no visible results yet.Furthermore, there were several potential issues discussed in the email thread, including cross-wallet imports using BIP39, the lack of support for Bip39 import by some wallets like Electrum, and whether importing a BIP39 mnemonic into a hardware wallet could bypass some of the security that the hardware wallet provides, reducing the overall security of the wallet. To prepare for SegWit, a common derivation scheme needs to be defined for BIP44 compatible wallets to handle P2(W)SH and P2WPKH receiving addresses. Two options have been proposed, including sticking with the current Bip44 account and giving users the option to show public keys as either a P2PKH-Address or P2SH address or defining a new derivation path parallel to Bip44 but with a different 'purpose', letting the user choose which account they want to add ("Normal account" or "Witness account"). However, there is a major problem with overloading the account index to indicate the type of address, creating confusion regarding whether a wallet supports segwit or not.The writer suggests having another BIPXX for segwit so that a wallet can claim it is BIP44, BIP44+BIPXX, or BIPXX compatible, making it clear which wallets are compatible with it. Pavol Rusnak disagreed with Jonas Schnelli's opinion that importing a BIP32 wallet is still an expert job, noting that most reasonable wallets now use BIP39 mnemonic for this purpose. Andreas Schildbach explains that the idea behind BIP43 is to ensure that balance retrieval never changes so that the same balance can always be seen on "same BIP" wallets.
