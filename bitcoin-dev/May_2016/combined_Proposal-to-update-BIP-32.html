Summary:

Jochen Hoenicke has proposed an update to the Bitcoin Improvement Proposal 32 (BIP-32) which would allow for invalid nodes to be skipped in the BIP-32 tree. The current requirement of checking for errors and handling them appropriately shifts the burden to the application developer, causing additional complexity and effort. By modifying the procedure to repeat hashing with slightly different input data until a valid private key is found, the library will always return a valid node for all paths. While some app developers feel that the complexity is not worth the effort, Jochen believes that the backward compatibility issues are minimal. He suggests updating other derivation functions in a similar manner. Pavol Rusnak wrote to the bitcoin-dev mailing list on May 8, 2016, requesting feedback from Sipa regarding a proposed backwards compatible fix. Marek Palatinus also expressed his support for the proposal. However, Sipa had not been active on the mailing list for some time and did not receive the message.Jochen's proposed modification addresses the issue of uncertainty when an intermediate node is missing. If the computed hash is larger or equal to the prime or 0, then the node is considered invalid and skipped. The chance of this affecting anyone is less than 10^-30. By encapsulating the difficulty within the library, the burden is shifted from the application developer to the library itself, resulting in simpler implementation and program flow isolation.The current specification of BIP-32 requires all callers of CKDpriv or CKDpub to check for errors when the computed hash I_L is larger or equal to the prime or zero, creating an additional burden for application developers. The proposal also includes suggestions for updating the derivation functions and root node derivation from the seed.Eric Lombrozo raised a concern on bitcoin-dev regarding the handling of cases where the BIP-32 derivation path is invalid, which can be difficult to handle since skipping may not always be an option. Jochen found an example of an invalid BIP-32 path at m/28578'/33941 derived from a hexadecimal seed. In an email exchange between Lombrozo and Hoenicke, the probability of this case was discussed as being extremely small at 2^-128.Some app developers feel that dealing with this level of complexity is not worth the effort, but if handling the case is simple to implement and isolate in program flow, Lombrozo would be in favor of implementing such a solution.
