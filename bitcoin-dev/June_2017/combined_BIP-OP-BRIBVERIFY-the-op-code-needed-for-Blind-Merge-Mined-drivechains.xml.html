The Bitcoin development community has been discussing the implementation of sidechains and blind merge mining (BMM). One issue that arose was the blind merge mining, which contradicts the concept that a block list node is valid only if its block is valid. To address this, proposals were made to improve the OP_BRIBEVERIFY proposal, including removing the necessity of coinbase commitments and using the hash of the sidechain's name or genesis block to identify them instead of numbering them sequentially.Another discussion focused on the BRIBEVERIFY opcode, which allows users to bribe miners to include a hash in the coinbase transaction's output. Concerns have been raised about bribe attempts taking up space in main chain blocks. An alternate proposal suggests using a 0 output value with some currently non-standard OP_RETURN output script that specifies the critical hash and sidechain ID.The Blind Merge Mined Sidechains concept involves locking the miner's coinbase reward and fixing the witness commitment output to enable reliable identification of corresponding vouts for each drivechain. A new rule has been proposed that forces each new BlockMod to be in range [-X000,+1] relative to the old BlockMod, which means no skipping ahead but allowing reorg by starting a new chain from up to a=-X000 blocks ago. BMM also has a new rule that bribe isn't paid unless the sidechain block in question has been buried by [for example] 100 sidechain blocks.The proposed opcode called OP_BRIBEVERIFY allows a user to bribe a miner to include a hash in the coinbase transaction's output. Blind Merge Mined Sidechains (BMM) were discussed, which involves locking the miner's coinbase reward at index 0 in the coinbase transaction's output vector and fixing the witness commitment output to be at index 1 of the same vector. The discussions highlight the complexities and challenges associated with implementing sidechains on the Bitcoin blockchain, as well as the various proposals and solutions that have been suggested to address these challenges.The proposed op code allows users to bribe miners to include a hash in the coinbase transaction's output without necessarily following the sidechain. This feature enables different factions of the community to realize their own separate vision on a blockchain that is interoperable with the Bitcoin blockchain and use Bitcoin as a 'reserve currency' for their own network. The commitment structure requires that the miner's coinbase reward be at index 0 in the coinbase transaction's output vector and fixes the witness commitment output to be at index 1 of the same vector.The proposal suggests improvements to the OP_BRIBEVERIFY proposal, including removing the necessity of coinbase commitments, verifying a block by keeping a set of sidechain IDs, and not numbering sidechain IDs sequentially. The ratchet system plays a significant role in linking the h* data obtained from bribes to sidechain blocks. This is achieved by adding the sidechain block hashes to the ratchet system. Furthermore, a sidechain has the ability to request verification of headers downloaded from peers and form the side chain through their local mainchain node.Overall, the proposed op code enables sidechains to be created on the Bitcoin blockchain while allowing different factions of the community to realize their own visions on their own separate blockchain that is interoperable with the Bitcoin blockchain. The context highlights the importance of the ratchet system in connecting the h* data from bribes to sidechain blocks while also emphasizing the functionality of sidechains in the verification process.
