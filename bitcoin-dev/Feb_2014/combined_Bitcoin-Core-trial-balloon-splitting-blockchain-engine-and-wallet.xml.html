Summary:

The Bitcoin Core reference implementation has various roles, including a payment network access and management tool. Version 0.9 will allow the disabling of the bitcoind wallet for an optimized border router profile. Recent discussions propose running the Bitcoin Core wallet as a separate process to become a real SPV client while sandboxing wallet keys and sensitive data from the network-exposed P2P engine.The move towards process separation is a security goal, and while simple forking was explored, separate programs seem better since Windows requires an exec-like method for process separation. Pieter Wuille's work on headers-first sync requires increased review and test bandwidth for this wallet/blockchain engine split.Jeff Garzik argues that process separation increases the difficulty of accessing key data from the blockchain engine (BCE) and changes the attack surface. Mike Hearn agrees that process separation is necessary for sandboxing but is not convinced about Garzik's argument. He suggests using seccomp mechanism, a syscall whitelister that makes ptrace useless, among other things.Peter Todd suggests using seccomp with chroot and whitelisting the open() call so that the existing code can create new blockfiles and do whatever leveldb does. The discussion is still ongoing, and it remains unclear whether the separate-binary proposal will be implemented. However, it is a good security move to separate keys and sensitive wallet data away from the attack surface introduced by network interfaces into another separate process.
