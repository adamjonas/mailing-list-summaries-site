Summary:

The Bitcoin community has been actively engaged in discussions to address transaction malleability. The issue has been a topic of discussion among developers for some time, with proposals ranging from defining a new canonical transaction hash/ID (cTxID) to modifying wallet software and potentially invalidating some script functionality. However, both solutions come with their own set of challenges, including potential fork risks and problems with retroactively changing the meaning of nLockTime in Bitcoin transactions. Despite these difficulties, many agree that eliminating malleability entirely would be ideal.Various proposals have been made, including using static IDs instead of full transaction hashes, modifying signatures to include the entire transaction, and introducing new rules or versions. One proposed solution involves adding new rules, such as modifying wallet software and potentially invalidating some script functionality, but these rules would remain optional and controlled by an nVersion increase. Another suggestion was to define a new cTxID that would be used internally by clients instead of traditional transaction hashes, although this would require nodes to index all transactions.There is debate over how to handle malleability in Bitcoin, with some suggesting making CHECKMULTISIG require the dummy value to be exactly equal to OP_FALSE while others believe it is better to fix wallet software to handle malleability issues. Pieter Wuille's proposal to implement transaction.version=3 has received consensus from the group, but there are concerns about undiscovered malleability routes. Additionally, there are concerns about introducing new rules or versions that could lead to fork risk.Overall, the proposed changes aim to provide non-malleability guarantees for clients willing to restrict their functionality, rather than outlawing malleability altogether. New transaction and block versions allow implementing and automatically scheduling a soft fork without waiting for wallets to upgrade. Despite some reservations, the group agreed to proceed with Pieter's plan, which includes a version check to make it sane to complete. It is acknowledged that getting wallets to start generating transaction.version=3 might take years, but this is deemed acceptable.
