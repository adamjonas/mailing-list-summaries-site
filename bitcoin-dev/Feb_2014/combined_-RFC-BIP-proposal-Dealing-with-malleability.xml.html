Summary:

In February 2014, Bitcoin developers discussed the issue of transaction malleability. Pieter Wuille proposed a BIP to eliminate malleability over time, which was agreed upon by Gavin Andresen. Michael Gronager suggested making version 1 transactions unmalleable and introducing a version 3 that supported the malleability feature, but this would require blocking all ongoing transactions before deployment.The idea of modifying signatures to be of the entire transaction was proposed, but deemed unnecessary by Pieter Wuille. Instead, he proposed moving towards eliminating malleability without changing what is signed. Rune Kj√¶r Svendsen suggested defining a new canonical transaction hash/ID (cTxID), but Jeff Garzik pointed out that it was not a complete solution for all transaction types.Michael Gronager recommended blocking relay of non-deterministic transactions in a later version and forcing unmalleable transactions in blocks after having only conforming transactions for 1000 blocks. However, Pieter Wuille disagreed, citing that the rules may limit the possibilities of the script language further. He proposed introducing a new transaction version to protect senders who care about non-malleability, with the new block version number coordinating the rollout and choosing an automatic forking point.Alan Reiner suggested encouraging Bitcoin software developers to design their software to use a static ID instead of the full transaction hash, but this would not have prevented MtGox's double payments. Alex Morcos apologized for asking a question that had been discussed many times before. Pavol Rusnak proposed dropping DER encoding completely and using just 64 bytes directly for signature, but Pieter Wuille stated that it would be a hard fork and only something to be discussed if a version-2 scripting language is introduced, which is a long-term thing.
