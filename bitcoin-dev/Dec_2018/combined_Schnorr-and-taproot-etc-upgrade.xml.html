Summary:

In a discussion among developers on the bitcoin-dev mailing list, there were several proposals for changes to the Bitcoin system. One proposal involved disabling CHECKSIG and CHECKMULTISIG in favor of using CHECKDLS, CHECKDLSVERIFY, and CHECKDLSADD. Another suggestion was to pop the stack after OP_CLTV and OP_CSV, which would save one weight unit per OP_CSV. However, this change could cause issues for constructions that avoid the DROP. Additionally, there was a proposal to change the rule from "one true stack item" to "empty stack" for CLTV/CSV.In an email conversation on December 17, 2018, Johnson Lau proposed a change to Bitcoin Improvement Proposal (BIP) 114, which was previously suggested by Satoshi. Though he expressed concerns about the potential negative consequences of the change, such as people losing money by copying an existing script template. Lau also suggested popping the stack after OP_CLTV and OP_CSV, which he believed would save one weight unit per OP_CSV, but it is yet to be decided whether or not to implement these changes.In another email thread, Anthony Towns proposed a change to the way scripts are composed, arguing that making the stack exactly empty would make the script more composable and eliminate the need for special logic to handle top-level CHECKSIG versus mid-level CHECKSIGVERIFY. There was some discussion on the mailing list regarding this proposal.The article also discussed potential improvements to the Bitcoin system, including a new segwit version, different length segwit v1 public keys, a new segwit v1 script version, and additional opcodes. The writer proposed introducing 33-byte v1 witness addresses that encode a secp256k1 ECC point and using new Schnorr ops to replace the ECDSA CHECKSIG/CHECKMULTISIG ops.There were also discussions about changes to the Script language, with proposals to make the stack exactly empty instead of having one non-zero item in order to make it more composable and eliminate the need for special logic to handle top-level CHECKSIG versus mid-level CHECKSIGVERIFY. The developers concluded that changing the rule from "one true stack item" to "empty stack" could make more sense for CLTV/CSV. The developers acknowledged that fixing these issues creates further tech debt but did not have strong opinions on the topic.
