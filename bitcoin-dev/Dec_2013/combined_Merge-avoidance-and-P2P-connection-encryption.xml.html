The discussion between Mike Hearn and Gavin Andresen focused on the integration of BIP32 and BIP70, suggesting output scripts for backward compatibility. They propose allowing each output to have an additional xpubkey and iteration count field. The article discussed Bitcoin privacy for non-developers, covering P2P connection encryption, address reuse/payment protocol, CoinJoin, and merge avoidance. It provides insights into the importance of privacy in Bitcoin transactions and highlights various methods for improving it. The author believes that such proposals could help improve Bitcoin's privacy features.The email conversation between Mike Hearn and another person suggests the implementation of "super addresses" in Bitcoin wallets, which would improve privacy as users won't need to merge coin if they had one of these super addresses. The proposal fits nicely into BIP70, where the PaymentDetails message allows a merchant to provide multiple outputs. The author believes that such proposals could help improve Bitcoin's privacy features.A possible use case for pastebin style hosting sites is also discussed for payment requests, where regular end-users can upload a signed or unsigned payment request, and the site takes care of incrementing the iteration count on each download.Mike Hearn proposed the idea of pastebin style hosting sites for payment requests, where regular end-users can upload a payment request to payr.com, with an iteration count on each download and can be possibly signed. The user-signed payment request without iteration count can be enclosed within a payr.com-signed envelope that contains the iteration count. However, it is unclear how this signature would show up in the user interface, and what actions should be taken if the signature is broken or missing.
