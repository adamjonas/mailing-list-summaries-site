Summary:

The context discusses a comparison between the defunct OP_EVAL proposal and the current pay to script hash proposal. A diff of the two proposals can be found at https://github.com/bitcoin/bitcoin/pull/748/files. It is suggested that it would be more beneficial for code review purposes if the old pull is reverted and the new one re-requested. This would simplify the process of identifying the actual changes made between the two proposals.The purpose of counting the number of sig ops after executing a script in ConnectInputs is to ensure compliance with the rule that "a block shall not contain more than MAX_BLOCK_SIGOPS (20,000)". There is concern that this counting may be too late in the process. To address this, further refactoring of ConnectInputs is suggested, including the addition of FetchInputs() and AreInputsStandard() functions.Gavin Andresen and Pieter discussed the compressed-public-keys patch on January 7, 2012. They discussed how compressed public keys interact with pay-to-script-hash to make ECDSA denial-of-service attacks less expensive and how transaction fees need to be reconsidered before releasing version 0.6. They suggested tweaking the fee policy so that denial-of-service attacks using compressed public keys and 1-of-3 CHECKMULTISIG transactions are expensive enough to deter attackers. Gavin Andresen suggests reverting an old pull to make it easier to see the real changes. He updates his gavinandresen/master GitHub branch to the state of the tree just before the OP_EVAL merge for code review purposes. The changes include unrelated 0.6 pulls, but it should be pretty obvious what is what. However, he notes that there were one major merge and eight commits to fix bugs or tweak things and is hesitant to revert them and risk losing a fix that he wants to keep.
