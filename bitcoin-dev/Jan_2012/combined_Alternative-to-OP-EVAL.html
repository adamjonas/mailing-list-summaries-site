Summary:

The Bitcoin scripting language has been the subject of ongoing discussions and proposals aimed at improving its security and minimizing potential risks associated with various operators and codes. One of the most significant debates centered around the proposed addition of a new functionality called OP_EVAL, which added expressive power to the language but also eliminated the ability to statically reason about scripts. Gavin Andresen, the lead developer of Bitcoin, requested an alternative to OP_EVAL, leading to several proposals for OP_EVAL that allow static analysis being suggested, such as using a fixed position reference prefix, using an execute bit on data set by an opcode, using OP_CODEHASH, using OP_CHECKEDEVAL, and using OP_HASH160 OP_EQUALVERIFY as a special sigPubKey. However, the debate on whether it is worth the hassle to get a statically verifiable OP_EVAL is still ongoing.In response to Gavin's request for an alternative to OP_EVAL, Russell O'Connor proposed adding a third "codehash" stack with OP_CODESEPARATOR and OP_CODEHASH operators in a new transaction format that can only be redeemed by knowing the hash of the script. Gavin expressed his opinion that the proposed alternative would not be any better than OP_EVAL. Roconnor proposes OP_CHECKEDEVAL which defines its own literal-only stack where all literals push to and only OP_CHECKEDEVAL pops from. It has several advantages over OP_EVAL, including easy spend-to-script-hash and static analyzability. In contrast, Pieter Wuille proposed a more restricted version of OP_EVAL called OP_CHECKEDEVAL.There were concerns raised about opening up a hole that could allow someone to satisfy arbitrary scripts or create one-packet DoS/crash attacks. Suggestions have been made for preventing OP_EVAL from executing the result of calculations and for releasing new features first on testnet. Gavin Andresen proposed a new 'standard' transaction type called "pay to Script hash," which allows users to publish a short "funding address" that is the hash of an arbitrary redemption script revealed when they spend the funds. This proposal is considered cleaner and more straightforward to implement than OP_EVAL.Overall, the discussions highlight the importance of prioritizing security in Bitcoin scripting language and the need for careful consideration of potential risks associated with various operators and codes. While there are ongoing debates about whether to implement new functionality like OP_EVAL, proposals like Gavin Andresen's pay-to-script-hash transaction type offer cleaner alternatives that prioritize security.
