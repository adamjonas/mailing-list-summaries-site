Summary:

The email thread on the bitcoin-dev mailing list covers various topics related to Bitcoin transactions, including spamming, censorship resistance, and storage of arbitrary amounts of data on-chain. One proposed solution for timestamping and validating data is to use a commitment to a signature, which would only be stored by nodes that explicitly enable them. This is seen as more efficient than using OP_RETURN or witness nodes, as all nodes would store such data. The issue of censorship resistance is also discussed, with concerns about some projects censoring transactions and government agencies taking advantage of such practices. Two tools are mentioned to test if transactions are getting relayed by connected peers, including counting the number of INV messages and broadcasting transactions to specific nodes using libbtc.In terms of storing arbitrary amounts of data on-chain, one solution suggested is to store the first and last chunk of data and keep the internal state of SHA-256 before entering the last chunk. This way, unused parts can be pruned without invalidating signatures. However, spammers could provide "FALSE" as a separate witness element rather than being part of the witnessScript, making it difficult to ban specific script fragments and leading to legitimate uses of "IF ENDIF."The writer seeks feedback on an ideal tool for testing censorship resistance, which includes allowing users to construct different types of transactions, broadcast them to specific nodes, verify successful relay or rejection, and ban peers if necessary. The script's primary goal is for testing, but it can also be used by anyone who wants to avoid wasting resources connecting to peers that do not relay transactions. Inspections have some positives, including more users interested in running full nodes (non-pruned), trying bitcoin wallets, lightning, etc., increased Taproot usage, more developers interested in learning bitcoin development and looking for libraries, docs, etc., and demand for block space has increased. The discussions highlight the need for protocols relying on data embedded in Bitcoin transactions to have flexible encoding rules and the challenges of preventing spam and censorship. Different solutions, such as pruning unused parts, testing censorship resistance, and using commitments, are proposed to address these challenges.
