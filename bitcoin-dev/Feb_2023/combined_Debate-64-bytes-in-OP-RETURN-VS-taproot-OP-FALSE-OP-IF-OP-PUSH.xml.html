Summary:

Bitcoin developers are currently discussing the best practices for storing data on the blockchain. One of the primary debates concerns the use of OP_RETURN and potential alternatives. While the current standard allows for a maximum of 80 bytes and only one output per transaction, some suggest increasing the size of OP_RETURN to support more significant data. Others argue that it may not reduce inscription-style use cases significantly. In addition, there is ongoing discussion surrounding the crossover point in data size where using witness data would be cheaper than using script pubkey. Various proposals have been put forward, including requiring all data to be in the op_return output and adding a required op_return_hash field that is checked by consensus.During a recent discussion, a potential crossover point was identified in terms of data size where it may be more cost-effective to use witness data rather than script pubkey. This has led to the proposal of an alternative method for storing data in taproot annex using Datacarrier with a single transaction and a script validation extension called PUSH_ANNEX. However, there are concerns about potential issues with "partial" block validation and skipping witness data.Overall, it is essential to weigh the advantages and drawbacks of each approach before making any decisions. Collaboration and consensus-building remain crucial in ensuring the continued success and growth of Bitcoin. Finding the most efficient and effective way to store and validate data remains a top priority for those involved in this ongoing discussion.
