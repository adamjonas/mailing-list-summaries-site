Summary:

The conversation mainly revolves around the efficient protocol for block relaying and validation in Bitcoin proposed by Matthew Mitchell. Mitchell suggests that all the transaction hashes could be given using a simpler protocol, which would allow nodes to request multiple transactions at once. However, some are concerned about the inefficiency of downloading all the transactions individually. There is also discussion around adding a few additional components to the protocol to allow requests for any level of the Merkle tree.The issue of block size limits is raised, with the current limit being ~0.95MB. Mitchell argues that the limit will need to be lifted eventually to satisfy Bitcoin demand, but others argue that the finite size and contention for space creates non-trivial fees, which are necessary for security. The scarcity of block space is what makes the security model economically sane. However, Mitchell notes that it is possible to make transactions denominated in Bitcoin outside of the blockchain, such as through Open Transactions.The context also touches on Bitcoin banks, trust, and Open Transactions contracts that contain proof of agreement and offer legal protection. It is emphasized that Bitcoin is great but its greatness depends on there being a way to fund enough computation to maintain a system to fully validate the system's rules.The conversation further discusses the scalability of Bitcoin and whether it can replace every other system due to slow transactions. It is expected that any increase in block size should not degrade decentralization or mining economics. The discussion goes on to consider segmenting blocks, ultimately concluding that it doesn't make much sense to design for that. Instead, focus should be on decreasing block relay times for the current network. A BIP draft for improving block relaying and validation has been proposed, which focuses on sending the hash tree. However, some argue that the block header, transaction list, and transactions a node doesn't already know are enough. In terms of parallelization of block downloads, the approach suggested involves requesting 8 level-3 hashes and beginning 8 parallel threads to download the transactions from various peers. It is suggested that processing different layers of the Merkle tree iteratively after starting the initial parallel segments is not useful as it adds several round-trips and transfers more data than the transactions themselves.Other potential solutions discussed include Ripple, which is a sort of p2p hawala that allows the circulation of tokens between a community of users and settlement against bitcoin. Changing the block size limit would require replacing all Bitcoin node software with incompatible software, which would be technically difficult and economically risky.
