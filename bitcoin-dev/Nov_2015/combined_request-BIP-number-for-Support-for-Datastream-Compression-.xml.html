Summary:

Bitcoin Core developers have proposed to incorporate datastream compression into the block relay mechanism to reduce bandwidth usage. The proposal suggests using Zlib compression for blocks and shows an average of 20% block compression, with the primary benefit being less bandwidth usage. However, there are concerns about the security of the Zlib compression library, and some suggest using LZO instead. Additionally, a new "cblock" message could be created that compresses only block messages, which shouldn't reduce efficiency by much. The bitcoin-dev mailing list discussed the use of compressed blocks and transactions to save on bandwidth. The latest test results on compression ratios for the first 295,000 blocks show that serving up compressed blocks is beneficial for those with bandwidth caps on their internet connections. LZO provides better compression, but it would come at a cost of CPU performance and using a less-reviewed and less-field-tested library. Some engineers believe that data compression should be left to the application layer rather than the transport layer or lower in order to give application designers more control over tradeoffs.The proposal is to compress blocks using a compression library and advertise compression as a service in case compression or decompression fails or crashes. Compression should be done at the datastream level in the code. Trade-off decisions should be local and negotiated between peers rather than being a required feature of the network P2P. The proposal suggests using a BIP titled "Support for Datastream Compression."There was some debate over the value of the proposed 25% bandwidth savings. A suggestion was made to give the BIP a more generic title, "Support for Datastream Compression," rather than the specific PR title of "Zlib Compression for block relay," since it could also be used for transactions at a later time. The aim of these suggestions is to optimize compression and reduce network buffer requirements. Additionally, a message holder could be defined for compressed messages, but this would likely be too complex to be worth the effort.Peter Tschipper notes that while there may be better ways of sending new blocks, the PR is particularly useful for historical blocks and sending transactions. In terms of improving the efficiency of block and transaction transfer, Tschipper suggests defining a "cblocks" message to handle multiple blocks and combining transactions together and compressing them into "ctxs." Jonathan Corgan reinforces the idea that such trade-off decisions should be local and negotiated between peers, rather than a required feature of the network P2P.In conclusion, the proposal to incorporate datastream compression into Bitcoin's block relay mechanism has been submitted, and the aim is to compress the data before sending it, initially for blocks only, but theoretically could be done for transactions as well. The idea of compressing blocks using a compression library and advertising compression as a service in case compression or decompression fails or crashes was discussed. However, trade-off decisions should be local and negotiated between peers, not a required feature of the network P2P.
