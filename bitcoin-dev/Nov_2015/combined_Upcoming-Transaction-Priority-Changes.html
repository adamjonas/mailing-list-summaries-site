Summary:

The bitcoin-dev mailing list recently discussed mining code and block priority size. The discussion centered around finding a balance between maintaining the mining policy and preserving the ability of users to place priority-based transactions in a limited mempool world. Some members suggested that the starting priority should be optional for ease of implementation, while others supported customizable cost and reward.During the discussion, there was debate about changing the default block priority size to 0. However, Luke Dashjr argued that this would not be helpful as priority is the best metric for ensuring legitimate transactions get mined despite spam attacks. Instead, he proposed merging priority, feerate, and sigoprate into one number, which he believed would be a better solution.In November 2015, there was a similar discussion about the priority space in Bitcoin blocks. Chun Wang claimed that it was "always the best place for spam nowadays." However, Luke Dashjr contradicted this claim, stating that spammers generally cannot use the priority space at all, and that it is actually a major way for legitimate users to have their transactions processed cheaply despite ongoing spam attempts.Matt Corallo wrote a post on bitcoin-dev suggesting certain measures for version 0.12, such as using starting priority for mining code for ease of implementation and setting the default block priority size to 0. However, Luke-Jr disagreed with changing defaults and influencing miner policy.The conversation also highlighted the importance of updating things in the right order when blocks are connected or disconnected to prevent logic breaking in future changes. The Bitcoin network is currently experiencing mempool bloating and GBT latency caused by the priority space. Many free transactions are not getting mined and cleared out of the mempool anymore. To prevent GBT latency degradation, limitfreerelay=0 is necessary in addition to setting minrelaytxfee=0.0001.
