Summary:

A discussion on a Bitcoin forum is focused on the comparison of locktimes for including a transaction in a block. The current consensus rule is that a transaction can only be included in a block if its locktime is smaller than the timestamp of the block. However, BIP113 proposes a new rule whereby a transaction can only be included in a block if its locktime is smaller than the median-time-past, which is already a consensus rule. This constitutes a hardfork as any transaction with a locktime between the median-time-past and the block nTime will now be valid.Bitcoin Core recently merged #6566, which added BIP113 logic to the mempool and block creation process. However, this may produce invalid blocks, and Rusty suggests adding N hours to the median-time-past for comparison and max()'d with the block nTime for it to be a proper hardfork. Alternatively, it may be best to wait and deploy BIP113 as part of a planned hardfork in the future. Further input and thoughts are welcomed.Meanwhile, a user named jl2012 queries whether their interpretation of the new rule is correct and receives confirmation. Another user named Luke apologizes for causing confusion earlier. In conclusion, there are concerns about the implementation of BIP113 as it introduces a new consensus rule and constitutes a hardfork. Further discussion and input are necessary before a solution can be determined.
