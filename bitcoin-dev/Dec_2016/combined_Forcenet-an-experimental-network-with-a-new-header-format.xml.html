Summary:

Various proposals for updates to Bitcoin's protocol have been discussed on the bitcoin-dev mailing list, including a potential hard fork that would remove the block size limit. Johnson Lau suggested a second version of bitcoin's hardfork, called forcenet2, with experimental features including a new header format, anti-replay, SigHash limit for old scripts, and new tx weight accounting. Matt Corallo suggested minimizing header size, having only one merkle tree, avoiding variable-length header fields, and avoiding complicated max() limits. The proposed changes include a maximum limit on block resources, a smooth halving mechanism, and a new merkle root algorithm using a sum tree. During discussions on the bitcoin-dev mailing list, concerns were raised regarding the use of variable-length fields in the header, the need for both TMR and WMR, timestamp size, and the use of a sum tree for partial validation and fraud proofs. Solutions were proposed, including using a fixed-size header instead of variable length fields, reducing the number of trees to keep hashing to a minimum, and explicitly committing to weight and nSigOp values. The lack of flexibility to redefine values through softforks is identified as the biggest problem with sum tree.One solution proposed is to explicitly commit to the weight and nSigOp, but this comes with an overhead of 16 bytes per transaction. Another suggestion is to transmit the weight and sigop count as variable length integers to reduce the overhead. Additionally, the two costs could be combined into a unified cost to make it easier for miners to decide what to charge. Other proposals include requiring pre-payment of the majority of the fee for spending an unspent transaction output, a new coinbase transaction format, and a merkle sum tree for fraud-proofing for fee and weight. BIP157 proposes "Client Side Block Filtering", which allows clients to filter blocks based on specific criteria to improve network scalability. A softfork is defined to enable this feature, and when activated, full nodes will not validate the sum tree. Finally, BIP157 proposes a smooth halving of the reward for the last 2016 blocks in a halving cycle, reducing it by 25% and contributing it to the first 2016 blocks of the new cycle.A new experimental network called Forcenet has been created to show how a new header format may be implemented. The current bitcoin header is limited and forces users to put additional data in the coinbase transaction, which is not ideal for light wallets. The header hash is calculated in a way that non-upgrading nodes would see it as a block with only the coinbase tx and zero output value. This technique allows rewriting most of the rules related to block and transaction validity. The header format is described in Luke's BIP, but has been amended by the creator of Forcenet. In addition to increasing the maximum block size, the calculation and validation of witness commitment may be changed with a new header.Overall, these discussions and experiments aim to improve the bitcoin header format and increase its efficiency while maintaining its security and validity. Links to relevant GitHub commits and PRs were provided throughout the discussions.
