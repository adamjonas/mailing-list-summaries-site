Summary:

Bitcoin's issue of requiring a new address for every transaction makes it difficult to make recurring payments. A proposed scheme follows the structure of BIP44 and generates multiple derived addresses from a contact address, simplifying the process and allowing for multiple recurring payments using a single offline interaction. The proposal also gives Bob more privacy by enabling him to send payments to Carol using multiple outputs. However, this only works in one direction and adjustments may need to be made for different address types on Bitcoin.In an email thread on the bitcoin-dev mailing list, El_Hoy proposed the scheme, which Ruben Somsen noted as an improvement over the status quo. Ruben raised concerns about defining the "contact" level deterministically and potential exponential blowup in gap limits if each xpub has its gap limit. He suggests defining a low default gap limit for these xpubs and being explicit about how they are used to derive addresses for others.Eloy proposes a BIP to make it easy for developers to implement and users to use. The proposed scheme requires only one-time interaction and should be easier to implement on both parties involved. Overall, the scheme is a positive solution to the issue of using a new address for every transaction, allowing for privacy and easy labeling of payments received, with the added advantage of choosing to send payments using multiple outputs.
