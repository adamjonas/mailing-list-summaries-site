Summary:

The Bitcoin developer community has proposed a new set of mempool and transaction relay policies to aid Layer 2/contract protocols. The proposal aims to improve the security model of L2 contracts and enhance the composability of smart contracting. It includes revisions to Replace-by-Fee (RBF) that would allow a child transaction to pay for a fee-bump of its parent transaction. This change may simplify the process for L2 protocol developers who can create presigned transactions paying 0 fees and one output for attaching a CPFP at broadcast time.The proposed modifications also incorporate additional rules for V3 transactions, such as only allowing one descendant per transaction to prevent pinning attacks. A V3 transaction can be replaced even if it does not signal BIP125 replaceability, and any descendant of an unconfirmed V3 transaction must also be V3. An unconfirmed V3 transaction cannot have more than one descendant, and a V3 transaction with an unconfirmed V3 ancestor cannot be larger than 1000 virtual bytes. All package transactions with mempool conflicts must be V3.The proposal aims to solve Rule 3 pinning and potentially allow for the removal of CPFP carve-out, with feedback from users being encouraged. However, there are concerns about compatibility with miner incentives and potential UTXO set bloat issues. The proposal has generated positive feedback, and it is believed that it would be valuable to add this feature.The proposal also suggests modifications to the existing policy rules, such as increasing the upper bound of the V3 transaction size, allowing OP_TRUE as a standard script type, and adding a 0-value output to make anchors easier to design. However, there are concerns about 0-value outputs in the UTXO set and privacy issues surrounding unilateral closures.The proposal has been implemented and documented for Bitcoin Core, and feedback from users is welcome. The Bitcoin development team has answered some expected questions about this policy, including whether it fixes Rule 3 Pinning and whether it is a privacy issue. There are concerns that LN transactions could be fingerprinted based on nVersion. The proposed policies aim to prevent pinning attacks and improve the security model for layer-two contracts, but some issues still need to be addressed.
