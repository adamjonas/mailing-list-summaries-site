Summary:

Bitcoin developers have proposed a new set of mempool/transaction relay policies to enhance the security model for Layer 2/contract protocols. The proposal includes modifications to the existing policy rules, such as increasing the upper bound of the V3 transaction size, allowing OP_TRUE as a standard script type, and adding a 0-value output to make anchors easier to design. Additionally, modifications to package RBF rules are also included, allowing presigned transactions to replace each other.The proposal aims to solve Rule 3 pinning and may allow for the removal of CPFP carve-out. V3 transactions are replaceable even if they don't signal BIP125 replaceability, and V2 transactions can replace V3 transactions and vice versa. Commitment transactions for the Lightning Network should be V3 and have one anchor output, with a child at most 1000vB to limit the number of inputs used to fund the fee bump.Changes have been made to the proposal, such as restricting the size of the whole V3 package via a specific value in the taproot annex. However, this may make relay policy stricter and would need to be done separately. Some concerns have been raised about utxo set bloat with 0-value outputs in the utxo set, but it is argued that it would be better than the current situation.Feedback and review on the proposed update is encouraged. The proposed policies aim to improve the security model for L2 contracts by making transactions more incentive-compatible to mine and reducing costs for fee-bumping transactions.Gloria Zhao, a Bitcoin Core developer, has proposed a set of mempool and transaction relay policies to aid Layer 2/contract protocols. These involve additional rules for V3 transactions, such as only allowing one descendant per transaction to prevent pinning attacks. There are concerns that LN transactions could be fingerprinted based on nVersion.Finally, Bitcoin developers are discussing proposals to improve v3 transactions, including limiting the number of OP_TRUE outputs per transaction, adding an "ephemeral output" to pay CPFP fees without making the parent tx have fees itself, and allowing outputs that are below dust. They also discussed the need for new logic to handle package RBF, which is ParentTx*s*(plural) and ChildTx(singular).
