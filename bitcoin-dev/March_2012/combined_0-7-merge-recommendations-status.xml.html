Summary:

Bitcoin Core developer Pieter Wuille brought up the issue of nodes accidentally connecting to themselves in an email thread on March 31, 2012. This could lead to DoS or other vulnerabilities. To address this, Pieter suggests making each publicly accessible method of BlockChain grab either a reader lock (if it's a const function) or an upgradable lock, and take a writer lock in each method that actually performs changes for thread-safety.Michael Grønager has suggested pushing libcoin to bitcoin as a solution to some of the problems listed. Libcoin is designed with no global thread mutexes and there is no blocking of the main thread due to rpc methods except for a sendto. The libcoin rpc supports keep_alive and pipelining, runs in its own thread but can also run in the same thread as the node, and uses async operation. Ipv6 is easy to implement in libcoin as the CAddress/Endpoint class is implemented as a subclass of boost::endpoint.Pieter prefers more frequent releases and encourages decent encapsulation of wallet and blockchain data structures that allow making their mutexes private. Additionally, Pieter's IPv6 support is near completion, but there are still some issues to address such as how relaying will work and potential risks for DoS or other vulnerabilities. Supporting IPv6 would allow Bitcoin to function as a Tor or I2P hidden service.Luke-Jr proposes multithreaded JSON-RPC with keep-alive support that doubles performance of JSON-RPC. He also calls for better encapsulation of wallet and blockchain data structures to allow for safe multithreaded RPC. Luke-Jr suggests that there are enough changes waiting to start the RC process for 0.7, but the first merge will need rebasing. Some changes are already acknowledged for 0.7 like Hearn's "pong" message and Wladimir's Visual C++ 2010 fixes.Scott has a pull request for Bitcoin-Qt to behave more like other close-to-systray applications by toggling the hide/show action. Coderrr has rebased his Coin Control features to the latest version and it should be merged soon. Getmemorypool BIP standardization is also acknowledged, but it might be better to wait until later in the merge window. Joel has optimized the ToHex function.In addition, Pieter Wuille remarks on the status of IPv6 support, including the technical part of supporting IPv6, removing IsIPv4 checks, and adding listen/connect code that is IPv6-compatible. Luke-Jr thinks it will take extensive testing before it's ready for 0.7.Lastly, Pieter would like to reverse the order of the last 12 bytes in the address db since they are opposite to boost.
