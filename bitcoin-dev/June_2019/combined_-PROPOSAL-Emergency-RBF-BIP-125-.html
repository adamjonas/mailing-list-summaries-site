Summary:

Bitcoin developers are discussing proposals to modify rules for replacing unconfirmed transactions with fee-bumped replacements. One proposal involves storing a relay bandwidth used (RBU) value with each transaction in the mempool, while another proposes an "emergency RBF" rule which allows users to replace an unconfirmed transaction with a new one that has a higher fee and is more likely to be included in a block.However, concerns have been raised about the potential for this rule to be used for spamming. Matt Corallo has also expressed concerns over implementation of the proposal related to child-pays-for-parent (CPFP) for replacing-by-fee (RBF). Rusty Russell has suggested amending the proposal to mitigate these issues, including removing the requirement to increase the absolute fee as long as the fee rate is increased. These proposed changes introduce added complexity and require more client-side knowledge, possibly resulting in extremely high feerates.On the Bitcoin development mailing list, Rusty Russell has proposed a modification to rules 3, 4, and 5 of BIP 125. The new "emergency RBF" rule allows for Replace-By-Fee (RBF) in adversarial conditions such as lightning unilateral closes where an adversary has another valid transaction and can use it to block yours. This proposal makes RBF miner-incentive compatible and opens no significant new ability for RBF spam.However, concerns have been raised about potential mempool spamming and griefing other nodes. One possible solution suggested was to change the proposal so that if the replacement transaction is in the first 2,000,000 weight units of the fee-ordered mempool, the rule applies. Additionally, there are concerns about the constraints of the proposal producing high feerates, although it could be beneficial for time-critical transactions.Another discussion on the mailing list involves proposed changes to the relay fee system, with developer Matt Corallo expressing concerns about a worst-case scenario, but Rusty Russell suggesting it would reduce costs for attackers by 90% compared to the existing transaction pinning issue.Rusty Russell, a Bitcoin developer, has proposed an "emergency RBF" rule for Bitcoin transactions. The new rule would allow users to replace an unconfirmed transaction with a new one that has a higher fee and is more likely to be included in a block. However, there are concerns about the potential for spamming the mempool by repeatedly using this feature to push transactions down the queue and grief other nodes.Matt Corallo has raised concerns over the implementation of the proposal related to child-pays-for-parent (CPFP) for replacing-by-fee (RBF), stating that it needs improvement in motivation and description. Rusty Russell clarified that if miners have a conflicting tx in the top 4MSipa, there would be no problem.
