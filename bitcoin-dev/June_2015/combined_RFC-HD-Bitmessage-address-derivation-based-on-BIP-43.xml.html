Summary:

In June and September 2015, discussions were held on the bitcoin-dev mailing list regarding Bitcoin Improvement Proposals (BIPs). Justus Ranvier expressed concern about a proposal to modify BIP 43, which suggests referring non-Bitcoin purpose codes to the SLIP repository. Luke Dashjr made the proposal, suggesting that delegating the BIP-43 namespace management to a company-independent repository would be better than SatoshiLabs. Justus argued that SLIPs should not be biased towards their company and that moving purpose codes to a separate namespace would add complexity without any discernible benefit. Furthermore, polluting the BIP repository with non-Bitcoin related specifications would not be helpful. The Monetas company has developed a Bitmessage address derivation method from an HD seed based on BIP-43. This method allows Bitmessage users to generate their addresses from a seed, providing eternal key backups. Furthermore, this method enables future use cases where senders calculate Bitmessage addresses based on a recipient's extended public key and some other index value. Monetas proposed this as a BIP based on the recommendation in BIP-43 to reserve a purpose code. The same seed can also be used for a Bitcoin wallet and a Bitmessage client. This development could have significant implications for the Bitmessage community and may improve overall security and functionality. The current situation with centralized registries was also discussed, and it was suggested to use a code deterministically generated from the chain ID instead of having competing registries maintained by different companies or P2P chains. In September 2015, there was a discussion on the Bitcoin-dev mailing list about maintaining competing registries for different companies and P2P chains. It was noted that the current situation with centralized registries was not an accident and that the BIP repository had made a decision to move part of the standard to a repository willing to handle it. Overall, these discussions centered around proposals to modify BIP 43 and the development of a Bitmessage address derivation method based on BIP-43. The potential implications for the Bitmessage community were also explored, along with discussions about the current state of centralized registries and possible alternatives.
