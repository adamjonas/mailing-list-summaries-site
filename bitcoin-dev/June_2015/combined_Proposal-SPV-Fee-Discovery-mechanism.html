Summary:

The email thread discussing transaction fees and dropped transactions proposes various solutions to address these issues. One proposal is to update the protocol to allow for fee and other statistics to be encoded in op_return output within the coinbase transaction, which could later be enforced with a soft fork. However, until then, SPV clients should not rely on this encoding as it can be fabricated by individual miners. The proposed feature would allow wallets to rely on an API for timely stats verified through SPV verification of fee stats for confirmed blocks.There is also a need for ways to increase fees on transactions after initial broadcast. This includes replace-by-fee and child-pays-for-parent, as transactions with unconfirmed inputs should be among the first to get dropped. The discussion highlights the importance of predictability for users when it comes to transaction fees and dropped transactions.Another proposal is to put fee and other statistics in block headers to allow Simplified Payment Verification (SPV) clients to rely on data with the same PoW-backed assurances as all other header data. This proposal is seen as valuable regardless of the outcome of the blocksize debate and could also be applied to other stats such as UTXO count. However, until a softfork is activated, SPV clients should not rely on this encoding as the results can be fabricated by individual miners.The conversation also discusses a possible solution for SPV wallets to learn about fees, which requires a new transaction version that embeds output values into the signed data and an upgrade to the P2P protocol to send UTXO data. This method has the advantage of not having to trust miners to publish data accurately, but it requires significant protocol upgrades and new code in SPV wallets.The issue of dropped transactions is debated, with one person arguing that transactions are dropped in a completely predictable way - lowest fee/KB first. However, another person acknowledges this but points out that from the user's perspective, the dropping of transactions can still be unpredictable if they don't know how to pick the appropriate fees. One possible solution could be to work directly with the largest miners and pools to get transactions into blocks, guaranteeing that they would be included in the accepted order.In conclusion, the discussion focuses on various proposals to address the issues related to transaction fees and dropped transactions. While some proposals involve protocol upgrades and code changes, others require direct collaboration with miners and pools. The debate highlights the need for predictability and reliability in the Bitcoin network to ensure smooth transactions for users.
