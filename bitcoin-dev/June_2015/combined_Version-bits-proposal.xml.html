Summary:

Pieter Wuille, Peter Todd, and Greg Maxwell have proposed a method for coordinating future soft-forking consensus changes in Bitcoin. The proposal allows for multiple parallel changes and rejected changes that will not obstruct the rollout of others. Wuille suggested using the mailing list for commenting as the gist does not support notifying participants of new comments. In an email discussion, Tier Nolan suggested that deadlines for Bitcoin Improvement Proposals (BIPs) should be set as block counts rather than using the median time mechanism.However, block counts are inconvenient for planning since there is no guarantee that they will happen in any particular timeframe. Peter Todd responded that assuming no large reorganizations, a list of block heights could be used instead of a table, even if the median time mechanism is used. The email conversation is a discussion on the use of the median time and nVersion soft-fork mechanisms for Bitcoin Improvement Proposals (BIPs).The median time mechanism allows hashing power to show what time they think it is, while the nVersion soft-fork mechanism enables hashing power to indicate which features they want to support. Increasing the number of blocks used in the median could reduce "noise". Each bit is assigned to a particular BIP for a specific range of times or blocks, and if block numbers were used for the deadline, one only needs to check the block index for the deadline block.In a discussion on the Bitcoin-development mailing list, Chris expressed his dislike of retroactively changing semantics and proposed that the version field should only be used for its intended purpose. Additionally, there is a discussion on changing the 95% rule, where a minimum of 750 out of the last 1000 blocks are needed to activate the rule, and then at least 95% compliance is required to start rejecting blocks that do not apply it.The context also discusses the efficiency of building merkle tree roots using micro-controllers, with Sergio Lerner suggesting leaving at least 16 bits of the version fixed as an extra-nonce to avoid miners using them as a nonce and interfering with the soft-fork voting system. In a discussion about manipulating the merkle tree in Bitcoin, Christian Decker and Patrick Strateman agree that there is no need to misuse the version field. They suggest that the version field should only be used for its intended purpose, while Strateman argues that any reasonable micro-controller can build merkle tree roots significantly faster than necessary.Luke Dashjr has suggested adding a section describing how the changes interact with and change GetBlockTemplate (GBT), proposing that GBT clients indicate a list of softforks by identifier keyword, and the GBT server responds with a template indicating an object of softfork keywords to bit values that the server will accept. Additionally, the server will indicate the preferred softfork flags via the version number as presently conveyed.Finally, the email thread discusses a proposal to set deadlines for Bitcoin Improvement Proposals (BIPs) as block counts, eliminating the need for the median time mechanism, while keeping multiple users of the bit separate. Several suggestions have been made to improve the readability of BIPs and clarify the 95% rule. Overall, the discussion focuses on maintaining the integrity of the soft-fork voting system while implementing new features in Bitcoin.
