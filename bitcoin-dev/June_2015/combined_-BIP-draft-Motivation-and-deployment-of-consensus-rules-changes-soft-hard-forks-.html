Summary:

There have been ongoing discussions on the bitcoin-dev mailing list regarding hardforks and softforks, with a focus on fixes for the timewarp exploit and proposals for deploying uncontroversial hardforks. One proposal suggests that all blocks should be within one day of the median of the previous 11 blocks, while another proposal suggests forcing two blocks to have the same timestamp. It is suggested that fixing the issue "right" would require a hard fork.Jorge Timón requested a BIP number from Greg Maxwell to concentrate discussions about deploying uncontroversial hardforks into a single place. The preferred choice for non-emergency uncontroversial hardforks is defining a starting block height after which miners upgrade. Once the 95% threshold is reached, the hardfork takes effect. The check can be replaced by redefining the threshold height rather than the height when miners started voting after the first block is buried.The author of a discussion thread on GitHub is attempting to create an informational BIP about why and how hardforks can be deployed, with a focus on non-blocksize-related examples. They are seeking feedback to streamline the structure and wording of the draft BIP and removing some history lessons mentioned in the proposal. Discussions are ongoing regarding getting miner confirmation for uncontroversial hardforks and whether to use nHeight, nMedianTime or just use nTime. Thomas Kerin expressed his belief that a document should exist prior to assigning a BIP number. The proposal has been labeled as "informational | process" but may be changed depending on its appropriateness. The contained code in the proposal is an example of an uncontroversial hard fork to create a precedent. A proposal for a "patch into core as a module that hard forks can use in the future" requires elaboration.It is noted that spell checking has not been followed for any BIP. A separate discussion related to this BIP regarding activation thresholds is taking place in a separate mailing list. The code for the proposal is available on Github. Overall, there are ongoing discussions and proposals regarding hardforks and softforks, including fixes for the timewarp exploit and the deployment of uncontroversial hardforks.
