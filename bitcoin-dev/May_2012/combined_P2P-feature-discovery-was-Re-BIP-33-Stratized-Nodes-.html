Summary:

Amir Taaki proposed BIP 33 to use the existing Bitcoin protocol for providing new functionality while maintaining compatibility. This would allow a parallel system to operate alongside the main Bitcoin network without affecting current Bitcoin clients. Jeff Garzik suggested improving the discovery process by creating a new command "get-features" that returns a list of key/value pairs, with the key being a string and the value type being determined by the key.In an email exchange, Luke-Jr stated that service bits are propagated as part of the address, making it easier to identify which nodes one wants to connect to for special services. However, passing long lists might be impractical, although it could work for protocol changes that don't add new services. Jeff Garzik cautioned against using bitcoin's P2P network for unrelated tasks, stating that if the peer list becomes too stratified, it is a sign that clients should use another network entirely.In a discussion on May 16, 2012, Jeff Garzik and Luke-Jr debated the idea of passing a huge list of nodes along as part of the address. Jeff Garzik proposed making feature discovery an easy, flexible, and extensible process by creating a new command called "get-features" that returns a list of key/value pairs. The key is a string, and the value type is determined by the key. However, this solution assumes that one already has a connection to the peer in question. Overall, the debate centers around improving the discovery process of Bitcoin's protocol to make it easier to identify and connect to nodes for specific services without overloading the P2P network with unrelated tasks.
