Summary:

Mike Hearn and Amir Taaki discussed a proposal for stratized nodes in an email thread. The design involved using a new protocol command to set a Bloom filter on a connection, allowing clients to filter data server-side rather than downloading the entire chain and dropping irrelevant transactions. This would enable users to choose their preferred point on the privacy/bandwidth-usage spectrum by strengthening or weakening the filters, without having specific knowledge of addresses. The filters would be applied to each data block in each script, allowing selection for things like multisig outputs or outputs which don't use addresses/pubkeys to authenticate.Hearn suggested that he could write a BIP for this alternative protocol if someone else wants to implement it, as simple optimizations to BitcoinJ could keep its performance good enough for the short term. However, concerns have been raised regarding security and use cases beyond a sample conversation. While the concept of stratized nodes has been deemed cool and useful, there are no economic motivations for running a stratized server, as is the case with BitTorrent super-nodes, Tor relays, or email servers.The discovery process for stratized nodes occurs in the same way as normal nodes, with service nodes being explicitly chosen similar to IRC servers for clients. It is suggested that keeping each request minimal would prevent resource starvation and simplify the protocol, while allowing history to be resolved with multiple services during data download and sorting.
