Summary:

In recent discussions on the Bitcoin-dev mailing list, various proposals have been made to improve the lightweight Bitcoin client model while maintaining security and privacy. One suggestion is to create a Bloom Filter Digest of every block, which could potentially improve communication bandwidth, but concerns have been raised about its efficiency. A more efficient data structure has been proposed that uses more bits and only one hash function, resulting in mostly zero bits, and entropy coding is used to compress the data. There has also been discussion surrounding the importance of unconfirmed transactions, their impact on real-world use cases, and how to handle them from a privacy perspective.While some argue that displaying unconfirmed transactions isn't worth the privacy trade-off until they are in a block, others stress their importance for instant transactions, citing examples such as merchants accepting credit card payments. The BFD concept has potential for deployment in handling mempool filtering, but user experience is an important consideration. Additionally, there has been debate around the security and privacy of BIP37, with suggestions for co-using wallet services or connecting to the mempool of each miner to ensure transaction validity. Ultimately, the success of any proposed system will depend on its ability to balance varying profiles for time sensitivity and fraud risk effectively.The Bitcoin community is discussing the importance of unconfirmed transactions and how to handle them from a privacy perspective. The committed bloom filter idea is supported for better privacy over BIP37, but downloading blocks from multiple peers with new tor circuits is still necessary for good privacy when using Bitcoin frequently. Unconfirmed transactions are vital for real-world use cases, especially for merchants who accept credit card payments and ship goods. However, relying on Simplified Payment Verification (SPV) 0-conf against random peers is fundamentally insecure and could lead to a large-scale fiasco that hurts Bitcoin's reputation and trust from merchants.Solutions include eliminating merchants' use of zero-confirmation SPV transactions from random peers through education and offering different solutions, such as running a full-node or using a wallet service with centralized verification and two-factor authentication. Sacrificing the "incoming funds" feature in SPV wallets for stability and risk-scaling and using privacy-preserving Blind Federated Drops instead may also be recommended. Additionally, the BFD concept has potential for deployment in handling 0-conf/mempool filtering.The Bitcoin community has introduced a new concept called Bloom Filter Digest (BFD) that can be used instead of BIP37 to improve the lightweight client model in a secure, efficient and privacy compatible manner. The BFD is created for every block by encompassing the inputs and outputs of the containing transactions and allows the filter to be cached between clients without needing to be recomputed. Additionally, it enables normal pruned nodes to do re-scans locally of their wallet without having to have the block data available or read the entire block chain from disk.Overall, Bitcoin serves a wide variety of use cases with different profiles for time sensitivity and fraud risk. Unconfirmed transactions are incredibly important but not over SPV against random peers. The debate around 0-conf transactions should focus on hiding or marking them appropriately, rather than debating their validity. Wallet developers would request action from a list of tools if they don't exist. It is useful to accept payments via 0-conf transactions in situations where waiting for full confirmation isn't feasible. However, no one suggests that all transaction validation should be performed by querying miners mempools. Determining validity goes back to the usual way once a TX gets its first confirmation.For merchants who cannot run their own full-node, using a wallet-service with centralized verification in watch-only mode is recommended. While this may increase the security of zero confirmations, relying on a few web services for payment validity determination is a potential Bitcoin security catastrophe. The purpose of bitcoin protocol development is to build base functionality for companies and individuals to provide usability to the end-user.
