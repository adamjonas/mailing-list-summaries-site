Summary:

The discussion revolves around the use of SHA256 to generate entropy and the length of the seed used. It is noted that SHA256 cannot generate more entropy than what is put in, it only stretches it. If the seed used is only 32 bits long, hashing it wouldn't save one from having every possible private key easily calculated in advance. The length of the seed is discussed as well, with BIP32 allowing for a range of entropy sizes, but they picked 256 bits instead of 128 bits. It is questioned whether longer sizes have a cost in terms of making the seeds harder to write down on paper. Additionally, greenaddress.it was noted to be creating seeds that have 24 words instead of 12, which raises compatibility concerns. Another idea was suggested to get rid of the topology and simply count the number of unused addresses since the beginning of the sequence, instead of using a "gap limit".
