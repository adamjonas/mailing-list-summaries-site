Summary:

The security of Bitcoin transactions has been a topic of multiple discussions and email exchanges. One major concern is the potential vulnerability to side-channel attacks in OpenSSL's implementation of ECDSA, specifically secp256k1. A new technique was discovered that could recover secp256k1 private keys by observing L3 CPU cache timings after OpenSSL calculates as little as 200 signatures. To address this issue, experts recommend using n-of-m multisig instead of single-factor addresses and requiring authorization from both online servers and a second "hardened" server.To reduce the attack surface, a constant-time implementation of secp256k1 with branchless code is also recommended for signing. Additionally, it is suggested to avoid address reuse and use Tor + hidden services for connecting to the machine. There are also discussions on TRESOR encryption, avoiding data-dependent loads to hide memory timing side-channels, and building open-source hardware for secure storage of private keys.Furthermore, businesses are advised to limit losses by flagging unusual events and employing multi-factor authentication schemes. Peter Todd recommends n-of-m multisig and not reusing addresses, while Eric Lombrozo suggests implementing a constant-time constant-cache-access-pattern secp256k1 based on branchless implementations of field and group operations. James Hartig argues that there is no sufficient way to prevent someone from stealing all the money once they have access to the server.Overall, the discussions highlight the importance of continued efforts to address vulnerabilities in cryptographic protocols and the need for precautions to avoid possible attacks, including using multisig transactions, implementing a constant-time constant-cache-access-pattern secp256k1, and avoiding address reuse. A blinding approach may also be necessary to protect against side-channels.
