Summary:

In 2013, discussions about the Bitcoin Improvement Proposal (BIP) process took place on the Bitcoin Development mailing list. Jeff Garzik volunteered to be the BIPS editor and suggested having backups with commit access to bips.git in case the BIPS editor is not available due to any reason. The group discussed the need for a formalized protocol specification for Bitcoin, with some suggesting that the current wiki pages be taken down or updated from the git repo. Suggestions were made to create a more comprehensive protocol specification, potentially maintained alongside the Satoshi client to keep it up to date. Alternative implementations are likely inevitable, and accurately describing the rules is essential to trusting the system. There were numerous discussions about the allocation of numbers for draft proposals, with some suggesting that drafts should be labeled as such if they do not pass and that work should not be assigned a number until it is complete. The Internet Engineering Task Force (IETF) distinguishes between individual proposals and accepted documents by working groups, making clear distinctions to prevent confusion and misunderstandings.As software for Bitcoin Improvement Proposals (BIPs) is developed, tested, published, and publicly discussed in an open-source manner, the BIPS process has become disorganized. In response, a new process has been proposed, consisting of four steps: submitting a draft, developing and discussing the software, requesting a BIP number, and adopting the BIP if there is general consensus. The proposed process is modeled after IETF drafts and aims to be more organized than the current one. The use of a Github repository for collaboration has also been suggested, but some have pointed out that requiring a git pull request may discourage contributors.Overall, discussions have highlighted the importance of open collaboration and transparency in the development and implementation of Bitcoin Improvement Proposals. The Bitcoin protocol is in need of a formalized specification to reduce bugs and encourage alternate implementations. While the reference implementation serves as the specification, the information on the wiki is best thought of as a set of Coles Notes on the real specification. Consistency is more important than correctness, and effort should be put into accurately describing the rules with a strong notice indicating that it is descriptive rather than normative. Extensive documentation of the source code may be a middle ground to avoid misunderstandings regarding the reference implementation source code. The BIP process is a way for anyone to propose changes or improvements to the Bitcoin protocol. BIP drafts can be submitted via git pull request or by emailing an attachment to bips.editor at bitcoin.org. Once there is general consensus that the BIP should be adopted, the BIP status is changed to "accepted." The current BIP-number-assigner-in-chief is gmaxwell who is also the BIP editor.
