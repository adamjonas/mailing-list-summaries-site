Summary:

In October and November of 2013, developers had several discussions about the development of Trezor, a hardware wallet for Bitcoin users. One concern was the difficulty of generating mnemonics on microcontrollers, which would prevent embedded devices from using them. Thomas Voegtlin proposed a method to prove that the seed generated by Trezor was created using a combination of computer-provided entropy and device-provided entropy without leaking private information, but there were still concerns about verifying backups and ensuring that the mnemonic corresponds to the secret in Trezor.Developers also discussed the compatibility of BIP0039 with Electrum and the lack of version number information in the seed encoding. They suggested allocating a few bits of the mnemonic to encode a "version number" along with checksum bits to specify which branches of the HD tree should be used for which purpose. Additionally, Pieter Wuille suggested that if there is a standard for 'brainwallets', it should have some strengthening built-in to decrease the impact of worst-case situations.A new method was developed to prove that the seed generated by Trezor is created using a combination of computer-provided entropy and device-provided entropy, making it blackbox-testable and fully deterministic. Seed generation is currently the only operation that uses any source of RNG. There were concerns raised about the verification of pubkeys derived through bip32's private derivations being derived from its own entropy. Trezor could not use straight bip32 out of the box, according to Timo Hanke. The chain code would have to be SHA(R), and the seed that gets translated to mnemonic would be r itself, making it 256 bit instead of only 128 bit.The article highlights the importance of unifying different approaches and creating compatible applications for the use of BIP32.
