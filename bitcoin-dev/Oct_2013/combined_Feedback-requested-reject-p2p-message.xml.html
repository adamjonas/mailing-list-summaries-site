Summary:

The bitcoinj library receives bug reports about transactions not propagating, as it currently sends transactions to only one peer and watches it propagate. If the selected peer refuses the transaction, it never gets propagated. The failure to reject a transaction does not necessarily mean that it will be forwarded. Nodes may use priority queued rate limiting for relaying, which can cause a transaction to fall out of memory pool before it is sent to other peers. The absence of rejection is not a reliable indicator of forwarding. Categories related to protocol, core principles, and policy rules are discussed, including failed deserialization, script evaluation fail, outputs larger than inputs, outputs not found or already spent, and non-standard policies. Gavin Andresen proposes adding a one-byte machine-readable HTTP-like status code, but a specific list of reject categories needs to be decided upon.
