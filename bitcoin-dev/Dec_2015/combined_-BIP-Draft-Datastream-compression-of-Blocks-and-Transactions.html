Summary:

There is potential for a custom, Bitcoin-specific compressor to learn and replace sequences with shorter numbers, leading to higher compression ratios. However, there are concerns about maintainability and testing, as well as potential difficulty in changing the wire format later on. Despite these challenges, building a custom compressor through a programming challenge/contest was suggested as one of the best ways to find the best possible, Bitcoin-specific compressor and bring new programmers into the ecosystem.Bitcoin developers have experimented with various compression libraries to reduce network bandwidth and improve performance, and a BIP document has been drafted for compressing blocks and transactions before sending them to reduce daily bandwidth used, speed up data transmission, encourage longer node uptime, and allow for more peer connections. Compression can be turned on/off using a service bit, and compression library LZO1x will be used. Blocks and transactions will be concatenated together when possible to increase compressibility, and transactions below 500 bytes will be sent uncompressed unless they can be concatenated. Test results show up to a 20% compression using LZO1x-1 and up to 27% when using LZO1x-999, with a marked performance improvement when there is latency on the network. Most transactions happen to be in the range of 1KB-10KB where they compress well but don't compress well below 500 bytes.In a discussion among bitcoin developers, Matt Corallo expressed concerns about adding compression due to additional complexity and attack surface, while Peter Tschipper argued that it would not be difficult to add compression since Bitcoin could easily support multiple compression libraries. Emin Gün Sirer brought up the point that LZ compressors assign short unique numbers to every new subsequence they encounter, which works well for English text but not for mostly binary data like Bitcoin. He suggested that a custom Bitcoin-aware compressor could potentially achieve much higher compression ratios, but there are concerns about maintainability and testing. A programming challenge/contest was suggested as a way to find the best possible, Bitcoin-specific compressor.In response to concerns about potential vulnerabilities in compression libraries, Pavel Janik suggested selecting a preferable compression library and checking for CVEs. Peter suggested isolating the decompression phase by saving incoming compressed blocks to a quarantine folder, and having an external process/daemon to decompress and verify the block's hash. Overall, while LZ compressors have been used for Bitcoin wire protocol, there is potential for better compression through a custom, Bitcoin-aware compressor. However, there are challenges in building and testing such a compressor, and concerns about maintainability and changing the wire format later on.The Bitcoin Dev Community has been exploring ways to compress blocks and transactions to reduce network bandwidth and improve performance. They have drafted a BIP document for compressing data before sending it, which would reduce daily bandwidth usage, speed up data transmission throughout the network, encourage users to keep their nodes running longer, and allow for more peer connections. The community continues to explore different options and approaches to find the best solution.A custom, Bitcoin-specific compressor could potentially achieve much higher compression ratios than the Lempel-Ziv family of compressors. However, there are concerns about maintainability and testing. A programming challenge/contest to find the best possible, Bitcoin-specific compressor could be an effective way to discover the limits of compressibility for Bitcoin bits on a wire. The Lempel-Ziv family of compressors has been used for Bitcoin wire protocol, but it may not be as compact as it could possibly be for mostly binary data and there are opportunities for better compression.
