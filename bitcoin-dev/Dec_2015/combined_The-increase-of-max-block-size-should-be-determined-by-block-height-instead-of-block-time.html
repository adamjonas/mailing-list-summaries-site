Summary:

The Bitcoin community has been debating the best approach for determining the maximum block size in Bitcoin. Some members suggest using height activation, while others argue for median time from the previous block. Jeff Garzik prefers height activation with one step per block, while Jorge Timón suggests using the median time of the previous block. Developers suggest using bip9 for 95% miner upgrade confirmation and always picking a difficulty retarget block. The sign bit can be used as the "hardfork bit" that gets activated for the next diff interval after 95% is reached when a hard fork becomes active. However, there are concerns about the use of block time to determine the maximum block size due to issues with ntime roll, which make it difficult for mining pools to issue a job with the maximum block size unknown.In an email thread on December 18, 2015, Peter Todd discussed the use of median time in block creation. He recommended that a decision be made on whether to use median time or height approach and always use it for both hardforks and softforks as part of bip9. Todd suggested that using median time of the previous block would achieve the same goal of not giving the creator of the current block direct control over the activation process. He also noted that an initial time threshold is not necessary for uncontroversial softforks but can simplify bip9's implementation. Todd urged for a final decision to be made, updating bip9 and bip99, and stopping the practice of doing something different on every hardfork patch.Finally, Jeff Garzik proposed a scheme for Bitcoin block size increase that uses height activation and one step per block, decreasing every 210,000 blocks at the time of reward halvings. This proposal is seen as a more efficient alternative to time-based activation. In conclusion, while there are differing opinions on the best approach, discussions continue to inform the development of Bitcoin and its future evolution.
