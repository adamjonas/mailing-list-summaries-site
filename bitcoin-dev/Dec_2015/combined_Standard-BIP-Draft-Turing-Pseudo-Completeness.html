Summary:

The Bitcoin developers mailing list had a conversation about using Bitcoin's scripting language for voting on proposals. Luke Durback proposed a design that would allow users to "mark" their BTC as being "For Proposition X," but struggled with finding a design that would allow this to work in the context of transactions. Jorge Timón suggested time locking the votes and using covenants instead of recursion or Turing completeness.A member of the mailing list proposed adding opcodes to make Script Turing Pseudo-Complete, as suggested by Wright. The proposal includes adding a return stack alongside the existing control stack and several new opcodes. Additionally, the proposal suggests adding OP_NAMESPACE to allow a function to pay its creator, with the ability to create transactions arbitrarily. Jeff Garzik suggested using a normal C/C++/etc. computer language, with preferred algorithm libraries and development tools, rather than adding opcodes.In the discussion, it was noted that each scriptSig is only executed once with its corresponding scriptPubKey, so changing that would be necessary. The conversation also touched on charging fees for complicated systems of scripts, such as a decentralized exchange between colored coins. However, the need for turing completeness was questioned, and high-level examples of use cases were requested to better understand the function of paying a creator.Later in the discussion, there was some confusion over the meaning of "a function" in the context of paying its creator. The conversation centered around specific use cases and the technical requirements needed to support them, including the ability to make new transactions arbitrarily. Overall, the discussion focused on the potential uses of Bitcoin's scripting language and the technical details required to implement them.
