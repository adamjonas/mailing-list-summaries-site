Summary:

The email exchanges that have been discussed revolve around the metrics used to determine the number of incoming connections needed for secure Bitcoin transactions. While one email argues that bandwidth is a better metric than connection count, another email debates the relevance of the number of connections and its impact on network traffic and security. The harm caused by "connect to everyone" nodes is highlighted in the third email, which emphasizes the need to drop such connections over time as they are not functioning nodes.The fourth email proposes using alerts instead of MSG_DOUBLESPEND, which adds more network traffic. The fifth email suggests a mining pool offering a real-time feed of accepted transactions. Finally, the discussion concludes that the network as a whole makes the eventual decision about which is the block-chain-to-rule-them-all. Chain forks and reorgs are less disruptive when each leg of a double spend isn't on each potential chain. Overall, the emails provide insight into the technicalities surrounding secure Bitcoin transactions.In a separate conversation, the topic of UPnP support and the IGD protocol on NAT routers is discussed, with one individual noting that it used to be enabled by default on most routers but is now typically disabled due to concerns over security vulnerabilities.The conversation primarily revolves around the issue of double-spending in Bitcoin transactions. Some suggest that payment processors can alert if a transaction is bad, while others argue that well-connected nodes can monitor miners/network for double-spends and alert relevant parties. It is noted that not everyone has the resources to implement this, but many storefronts do not even need such speed increases.The discussion then moves on to the proposal suggested by Andy Parkins, where a node drops a transaction and broadcasts it with a new "inv" type, "MSG_DOUBLESPEND," instead of "MSG_TX." This way, vendors can wait for confirmations and miners can lower the priority of flagged transactions. Matt adds that payment processors could handle double-spend checks and offer insurance against unconfirmed transactions being invalidated. Overall, there is no need to add extra network complexity for double-spending prevention as there are already effective solutions available.
