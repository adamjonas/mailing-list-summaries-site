Summary:

In 2011, the Bitcoin development mailing list saw various discussions on different features that could be added to the Bitcoin network. One developer, Luke-Jr, extracted and created proper git branches for each logical part of the proposed patches, including hub mode, threaded RPC, signal blk notify, bugfix_CreateThread_leak, getwork_dedupe, coinbaser, gitignore, minfee_modes, eligius_relay, and bitcoinuri. While some members supported multithreading the RPC to speed up services and adding Keep-Alive, others were hesitant about merging any new feature until it was thoroughly tested. There were also suggestions to fix resource leaks and update gitignore to ignore some common miscellaneous files.Jeff Garzik shared a link to http://davids.webmaster.com/~davids/bitcoin-3diff.txt on IRC, which included several features that big pools were asking for, including HTTP/1.1 keep-alive support. On July 10, Luke-Jr mentioned a new version at http://davids.webmaster.com/~davids/bitcoin-4diff.txt and extracted proper git branches for each logical part of the "3diff" version. Matt Corallo provided feedback on each of these branches, expressing his concerns about hub_mode and suggesting more generalized fee control.There were objections to some of these branches, such as long polling support, which would benefit very few people, and lower fees only eligible for Eligius, which could lead to DDoS attacks. On the other hand, there was agreement on the benefits of multithreading RPC and Keep-Alive. The solution requires careful thought and implementation rather than just lowering fees.Luke-Jr discussed different patches that need to be implemented soon, including some form of patch that implements threaded_rpc and rpc_keepalive, which is necessary due to the current implementation using a lot of socket resources. He also mentioned a bugfix_CreateThread_leak patch and a getwork_dedupe pull request, which should be merged. However, he was against adding signal_blk_notify and eligius_relay/sendfee, which he believes are features for small minorities. Additionally, Luke-Jr suggested that Bitcoin needs something more generalized to allow more custom settings for minfee_modes and not just blind acceptance if the fee is x per kb.In a conversation between Matt Corallo and Luke-Jr, they discuss the issue of controlling user fees in Bitcoin without causing too many side effects. Developers are writing software to get in the way of users and pools willing to do business despite accepting lower fees. The conversation also touches on various features such as bugfixes, gitignore, and native long polling support.Lastly, Jeff Garzik mentions he is moving his home computer lab setup to a new house and may be unavailable until August 11.
