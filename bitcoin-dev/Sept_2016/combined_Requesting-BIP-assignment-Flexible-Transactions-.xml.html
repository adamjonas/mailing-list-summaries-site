Summary:

Recently, the Bitcoin-dev community has been discussing several issues related to transactions. One of the issues concerns changes made in version 2 of the script that simplifies the data signed to be equivalent to the transaction-id. However, this causes some functionality loss. Andreas Schildbach suggested using Protocol Buffers (protobuf) for the bitcoin protocol, but another user recommended using Compact Message Format (CMF) instead due to its advantages over protobuf. There were also concerns raised about the compatibility of Flexible Transactions with Lightning Network.In various email exchanges, there was discussion regarding the serialization order of tokens in transactions and whether BIPs should be self-contained or rely on previous BIPs. Additionally, Gregory Maxwell reviewed a proposal called FT spec and found that it lacked specificity and motivation. The use of Protobuf as a serialization format has also been suggested due to its compact and binary nature. Compatibility with Lightning Network has been raised as a concern for the proposed serialization format for OP_CLTV and OP_CSV.Christian Decker argues that creating a new document and re-signing it does not constitute transaction malleability. The writer poses questions regarding the dependencies and order of inputs and outputs in tagged systems such as BIP. Despite these issues, the proposal for a new format using Compact Message Format (CMF) has been submitted as a BIP and has a reference implementation in Bitcoin Classic beta releases.On September 22, 2016, a discussion was initiated on the bitcoin-dev mailing list with Jonas Schnelli responding to Tom's misunderstanding of tagged systems. They compared Tom's understanding to thinking that HTML can only use a bold tag once in a document. Jonas posed a question regarding the "additional" segment and whether it contains the same amount of nSequence-equivalent tokens as the number of inputs in the "inputs" segment. There was also a discussion about spending limitations being assigned to an output rather than an input, which should be specified in the BIP.In another email thread on the Bitcoin-dev mailing list, there was a conversation about using 'Coin Movement Formats' for transactions in Bitcoin Classic. It was suggested that BIPs should be self-contained, and referencing external formatting documents should be avoided. A few minor issues were pointed out with the table and the definition of 'third-party malleability' was clarified. A debate about allowing miners to reject transactions with unknown fields potentially making OP_NOPs unusable was discussed, which could result in a hard-fork.Christian Decker questioned whether the comparison to XML and HTML holds up in CSV, given that the use of segments gives at most two levels of nesting, which could result in ambiguities about whether a tag refers to a single input or the transaction as a whole. Christian also disagreed with the idea that creating a new document and re-signing it constitutes transaction malleability. He argued that this is simply re-creating and something that the coin owner has every right to do. There was a discussion about dependencies in current transactions and the possibility of adding new fields that non-upgraded nodes may not recognize. If new fields are added and a non-upgraded node rejects transactions containing them, there may be a hard-fork. To address this issue, it is proposed that transactions with unknown fields should not be rejected if they are included in a block.Finally, a Bitcoin Improvement Proposal was submitted to formally request the assignment of a BIP number for FT spec, which aims to make Bitcoin's transaction more flexible and easier to extend while fixing all known cases of malleability and resolving significant amounts of technical debt.
