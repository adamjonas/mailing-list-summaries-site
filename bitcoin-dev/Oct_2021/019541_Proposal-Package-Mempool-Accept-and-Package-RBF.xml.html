Summary:

The email discusses decisions that require input from developers on whether to start with multiple-parent-1-child or 1-parent-1-child and if it is acceptable for the child not to have conflicts with mempool transactions. The writer of the email points out that package relay can be useful for better user experience of CPFP, specifically in a scenario where a wallet manages coins that can only be spent using pre-signed transactions. The pre-signed transactions are pre-signed with a fixed feerate, which may be below the mempool minimum fee at the time of broadcast. This is where multiple-parents-1-child packages come into play as they allow for batching of child-fee-paying transactions. Finally, the writer did not come up with a reason for dropping the rule regarding conflicts with mempool transactions, but individual submission can be used to replace the child and the child itself does not conflict anymore if the parent needs to be replaced.
