Summary:

The Bitcoin community has been discussing the idea of eliminating the mempool, which stores unconfirmed transactions. The proposal is that users should submit their transactions directly to mining pools via an anonymous communication network such as Tor. While some believe this would greatly reduce the bandwidth requirement for running a node and keep transaction intentionality private until confirmed, others have raised concerns about centralization pressure and higher barriers to entry.Proponents argue that direct communication between block template construction venues and transaction proposers could provide a mechanism for enforcing minimum security budgets. Critics suggest that communicating two numbers can easily be done over HTTP and already exists. Nonetheless, proponents argue that this would make transaction confirmation timelines less variable and provide block producers with a mechanism for enforcing their own minimum security budget.However, critics suggest that eliminating the mempool may not be feasible without compromising security and anonymity, which are important for maintaining the fee market and protecting against potential attacks with sufficient hashpower. Removing the mempool would complicate solo mining and make BetterHash proposals more difficult, while also providing a target for DoS attacks. Additionally, there are issues with pre-signed transactions under the mempool min fee, and dynamic lower bounds must exist to prevent spam. Despite these concerns, there are benefits to eliminating the mempool, including reducing the bandwidth requirement for running a node and keeping the intentionality of transactions private until confirmed. The community also emphasizes the importance of maintaining a decentralized and trust-minimized system while avoiding creating artificial barriers to entry or centralization vectors. There is a suggestion for an opt-in publicly accessible "overlay" network to relay miner-incentive compatible transactions with higher DOS limits, but issues like proving the validity of transactions and getting actual transaction data need to be addressed. Nonetheless, most agree that the mempool is still necessary for fee estimation and other benefits.In conclusion, while removing the mempool would resolve current issues inherent in package relay and RBF rules, it would also introduce new challenges. The proposal to eliminate the mempool is still under discussion and requires further consideration. The conversation highlights the need to improve the current design of Bitcoin's transaction pool while balancing the trade-offs between security, anonymity, and efficiency.
