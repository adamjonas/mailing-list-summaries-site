Summary:

The email exchange on the bitcoin-dev mailing list discusses a proposal called UTWFOTIB (Use Transaction Weight For Ordering Transactions In Blocks) put forward by Damian Williamson. The proposal suggests using the next block size to order transactions in blocks, but there are concerns about its feasibility as nodes may have different views of the transaction pool.ZmnSCPxj argues that consensus rules should restrict themselves to the characteristics of the block and its transactions, rather than depending on the transaction pool, which is not safe for any consensus rules. Nonetheless, both parties agree that the proposal is worth considering.The proposal aims to address the issue of transaction bandwidth limit by assigning each transaction a transaction weight based on the fee paid and the time waiting in the transaction pool out to n days. This way, it maximizes transaction reliability, total fees paid per block, and provides additional block entropy and greater security. However, further programming is required before it can be implemented.Moreover, there are questions about enforcement and implementation, as the transactions to be included in a block are entirely up to the miner of that block. Jim Renkel raises this concern in an email exchange with Damian Williamson, who suggests that block sizes based on transaction weight could be implemented if the transactions conform to a probability distribution curve or fee distribution curve. The proposed solution could provide additional block entropy and greater security. Nonetheless, further programming is required before it can be implemented.There is also another concern raised by ZmnSCPxj that the transaction pool is not part of consensus currently, and putting it into consensus would effectively lift the block size to include the entire transaction pool in each block. Additionally, full nodes using -blocksonly mode have no transaction pool and cannot verify incoming blocks for these new rules. The term "weight" has already been used in Bitcoin's SegWit weight equation, measured in virtual units called "sipa," which could cause confusion if reused for a different purpose.Despite these limitations, the UTWFOTIB proposal aims to maximize transaction reliability and total fees paid per block without reducing reliability, potentially leading to uncapped block sizes. Williamson and ZmnSCPxj agree that the proposal needs more work and explanation before it can be posted back to the list. The potential cons are yet to be identified, and the proposal must first be programmed. The email exchange between Williamson and ZmnSCPxj also discusses the issue of SPV confirmation faced by a node that is temporarily stopped and restarted after some time.
