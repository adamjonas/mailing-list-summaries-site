Summary:

The Bitcoin-dev mailing list has been discussing various issues related to Bitcoin's storage and consensus mechanisms. One critical point of discussion has been the UTXO storage and its implementation, which plays a vital role in ensuring consensus protocol success. Some have suggested having two storage options, while others warn against it due to the risk of consensus failure. There have been debates about outsourcing data management to existing databases, but it has been explained that Bitcoin's core consensus engine requires a highly optimized data structure for coin existence lookups that is not useful for other applications.The possibility of network splits caused by bugs in different implementations using different storage has also been raised. In addition, there is discussion about isolating storage from consensus code, with some arguing that doing so would be technically desirable, while others warn against the risk of network splits due to incompatible implementations. The goal remains to find a maintained and reliable alternative to leveldb, which is deemed unmaintained. Bitcoin developers are researching alternatives to the leveldb database and have initiated an implementation to replace it with sqlite. Facebook's fork of LevelDB called RocksDB has also been suggested as a replacement, but there are concerns regarding its optimization for big databases running on multiple cores and flash. The hope is to achieve implementation diversity without compromising reliability and security.There have been discussions about the transparency of ODBC/JDBC/JConnect or an equivalent driver for the consensus code and the technical desirability of isolating storage from the rest of consensus code. Overall, there is a push for diversity in implementation without compromising reliability and security.One proposed solution is to insert a column with a hash of what is stored in each row and another column with an incremental row by row hash composed of the hash of each row and the previous column one. Another proposal suggests having a publishing daemon that would allow companies and their software packages to write things on the Bitcoin blockchain. The daemon would check for changes every X minutes on that specific address in the db and, if funded, would publish "send" the transaction through the Bitcoin client. The aim is to make Bitcoin information available to thousands of software packages and programmers who work with RDBMS, not just "RocksDB" or some obscure database system, by outsourcing data management to the existing database world.In terms of UTXO storage, some have suggested allowing users with large databases to operate them to keep up with the network, but this has been met with resistance due to the need for consistency between all nodes. There is also concern about Type 2 consensus failures, which are more severe but less likely than Type 1 failures, such as exceeding the block size limit, which is considered invalid.Another point of discussion has been the issue of using multiple database technologies in Bitcoin. It was noted that isolating storage from consensus code is desirable, but implementing it using different storage can lead to network splits. The choice of database could affect the chances of consensus protocol failure. Furthermore, there is a push for diversity in implementation without compromising reliability and security.The conversation between Peter R. and Gregory Maxwell highlighted the criticality of UTXO storage in Bitcoin and the importance of maintaining consistency between all nodes. Gregory explained that the storage engine for the UTXO database in Bitcoin Core is fully internally abstracted, making it relatively straightforward for someone to drop something else in to experiment with. However, the application and needs of the system are very specialized, and off-the-shelf key value stores may not be suitable in all cases.Peter R. clarified the differences between Type 1 and Type 2 consensus failures in the Bitcoin system, citing examples such as the 92 million bitcoin bug from August 2010. In discussions about libconsensus, Peter argued that the network of miners and node operators being net-econo-rational is a core axiom for Bitcoin to be viable.Finally, the email conversations among developers highlighted the need to find a reliable and actively maintained alternative to leveldb, which is deemed unmaintained. The Bitcoin project is currently researching alternatives, with an implementation to replace leveldb with SQLite being introduced on Github.
