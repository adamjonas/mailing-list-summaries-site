Summary:

The context discusses various proposals and ideas related to upgrading versionbits and ensuring network adoption of new changes. The conversation highlights the challenges in enforcing new rules and the need for a middle ground solution that allows thin clients to treat blocks with increased skepticism while avoiding warnings on older clients. The difficulty of enforcing new rules without relying on someone mining a block that breaks the rule is discussed. The solution proposed is to have miners keep setting a bit for another 2016 blocks after activation and, if needed, make it a consensus rule. However, requiring the bit to be set during the soft fork activation period was deemed problematic as it would trigger warnings on older clients.A BIP deployment mechanism is suggested for fairly uncontroversial changes, and an awareness campaign can help address concerns about older clients. Rusty Russell proposes keeping the bit set until activation so network adoption is visible, and having a consensus rule that rejects blocks without the bit to force upgrades on the last miners. Soft fork features are carved out from non-standard behavior to prevent non-upgraded miners from mining invalid transactions.The Version Bits mechanism is not a voting system but rather a deployment mechanism for measuring adoption of uncontroversial changes before activating them. If a particular BIP stands little chance of hitting the 95% mark relatively quickly, it's probably better not to deploy it. The current controversies around features like CLTV, CSV, etc. do not seem to revolve around the features themselves, but much more likely to be around deployment strategies.While getting explicit acknowledgment from miners that a new rule is in effect would be ideal, there still lacks a means to determine whether or not miners are actually enforcing these rules unless someone happens to mine a block that breaks the new rule. A middle ground could be required where miners set the bit for a period of time after rule enforcing begins, but don't enforce the bit, just enforce validity of the block under new rules. Overall, the focus seems to be on ensuring a smooth transition towards new features while minimizing risks and disruption.
