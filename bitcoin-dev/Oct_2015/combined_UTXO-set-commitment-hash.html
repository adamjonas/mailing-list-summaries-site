Summary:

Bitcoin transactions can be committed in blocks by maintaining two running hashes of unspent and spent transaction outputs. Miners can place these hashes into a block's Merkle tree through a soft fork, which allows for an implementation independent definition of consensus to be defined by computing (hTXO, hSXTO) at a low cost. Leveldb is critical for consensus as it stores the state of transactions, but there's only one correct answer to what the set of unspent transaction outputs (UTXOs) is. To prevent consensus failures caused by bad database behavior, (hTXO, hSXTO) can be computed twice - once without the database while processing a new block, and once by requesting the same data from the database. By creating and publishing (hTXO, hSXTO), miners can share their view of the transaction state, thereby enabling any implementation to be validated against it.
