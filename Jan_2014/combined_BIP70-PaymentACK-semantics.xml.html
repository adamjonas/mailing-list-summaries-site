    Summary:

    The discussion is about the payment protocol being bidirectional, which allows merchants to use features such as refunds and memos. However, if the merchant's receive server is down at the time of transaction, the transfer of funds may occur without knowing what to do with it, making these fields useless. To prevent this issue, Pieter suggests that if a payment_uri is specified, the client must attempt to send the payment there. If a transaction is broadcast, a client should make a reasonable attempt at delivering the payment. If a paymentACK has been received, the client is no longer responsible for broadcasting the transaction but still may. The primary meaning of receiving an ACK is that the receiver takes responsibility for getting the transaction confirmed. This means the client does not need to stay online anymore, which removes the requirement for the P2P network to function as a reliable sender->receiver communication channel and reduces it to a broadcast medium to get transactions to miners.
