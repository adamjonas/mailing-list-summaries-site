    Summary:

    The email conversation between Mike Hearn, Chuck and Pieter discusses the implementation of the PaymentRequest/Payment/PaymentACK protocol in Bitcoin. Mike Hearn suggests that the transaction is only committed to the wallet if the server accepts the Payment message and ACKs it, which would mean that if there's a failure submitting or some kind of internal server error, the tx would not be retried if there's a failure submitting or some kind of internal server error. Pieter states that this is one right way to do it but not what is suggested or required by the specification and not what bitcoin core master currently implements. Chuck raises the issue of failure situations and how to address them with minor changes. Pieter agrees that the current implementation has a hole in the delivery of the payment message, which could cause a merchant to argue that the transactions seen on the network were insufficient. Pieter suggests that the specification should prevent transactions confirming without the payment message ever being delivered by storing them in the sender's client, retrying when necessary, or using exponential backoff.
