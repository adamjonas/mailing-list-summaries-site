    Summary:

    The email conversation discusses the implementation of multiple signature escrowed transactions and the challenges in getting it added to the client. The IsStandard part of the patch is seen as uncontroversial and could go in quickly, while the full escrow feature set would require more testing and discussion before being added. However, it won't be deployable until an updated IsStandard is widely used in the network. The patch exposes an issue with multisig validation due to redundancy in the script length coding opcodes, which can cause the signature validation to fail when something used OP_PUSHDATA when a direct length could have been used. Pull 349 contains one candidate fix for this: excluding the length opcodes from the comparison. However, this carries a risk of creating differences in how nodes validate transactions leading to lasting forks. Another way of fixing it would be to just define that OP_PUSHDATA* cannot be used to push the smaller lengths which could be more efficiently encoded with the direct length opcodes. Changing out the templating code is also suggested but deemed tricky to implement and validate.
