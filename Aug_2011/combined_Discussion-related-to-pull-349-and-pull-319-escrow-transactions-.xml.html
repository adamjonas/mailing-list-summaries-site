Summary:

In an email conversation from August 3, 2011, the topic of implementing multiple signature escrowed transactions was discussed. Pull 349 on Github contained a candidate fix for a problem with multisig validation that exposed an issue with redundancy in script length coding opcodes. The fix suggested excluding the length opcodes from the comparison but carried a risk of creating differences in how nodes validate transactions leading to lasting forks. Another solution proposed was defining that OP_PUSHDATA* cannot be used to push the smaller lengths which could be more efficiently encoded with the direct length opcodes. A third way was to change the templating code to make sure it codes the template the same way the original was coded. The discussion highlighted the need for testing and further discussion before implementation, especially since the new transaction type required an updated isStandard version. One suggestion was to split pull 319 into two pull requests - one for the IsStandard change and another for the full escrow feature set. This would ensure that when the escrow patch entered the mainline client, it would meet up with a network that could handle its transactions. The IsStandard part was deemed uncontroversial and could be added quickly.
