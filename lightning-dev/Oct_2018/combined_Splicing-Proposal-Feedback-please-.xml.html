Summary:

The Lightning Network community has been discussing the addition of a splice feature to the protocol, which would allow nodes to add or remove liquidity from channels without closing them. The proposed design includes a distinct message that references an active channel ID and a splice channel ID for the pending splice, allowing remote nodes to locate spliced channels. The developers have also discussed the need for both parties to send channel updates with the option_channel_moving bit set and continue to keep the channel in their routing table. Additionally, both sides can modify parameters such as CSV param, reserve, max accepted HTLCs, max HTLC size, etc., as the size of the channel is either expanding or contracting.There are two different methods for splicing being debated, and failure modes during splicing must be designed most precisely. The safety of Lightning protocol transactions, which are not on any block and use 0-conf transactions, is also discussed. While 0-conf transactions are generally known to be unsafe due to the possibility of transaction replacement, Lightning uses a 2-of-2 federation as signatories for commitment transactions, which makes it safe by preventing the replacement of a 0-conf transaction without the receiver allowing it.Discussions on the Lightning-dev mailing list have been taking place regarding proposals to splice funds in and out of Lightning Network channels. The proposed solution involves not enabling any splicing mechanism that allows off-chain use of spliced-in funds before the new funding output confirms. The negotiation process for splicing is discussed, as well as the limitations of the descriptor language, nested P2SH inputs, and allowing sides to set a sig script as well as witness during the signing phase. Maintaining a 1-week CSV when the channel size has dipped from 1 BTC to 100k satoshis is also discussed.The conversation between Laolu and ZmnSCPxj revolves around the Lightning Network and its protocol, with ZmnSCPxj expressing concerns about doing away with the initiator distinction in certain actions. C-lightning users are complaining that they can't open multiple channels, therefore, a rough proposal has been introduced to allow splicing of existing channels. Rene proposes a new method for splicing in the Lightning Network, involving creating an additional output spent by a special commitment transaction, which simplifies the process and avoids the need to send around all messages twice. The proposal also allows for multiple splicing operations in parallel.The Lightning Network enables instant and high-volume Bitcoin transactions as an overlay network to the Bitcoin Blockchain. The email thread discusses two methods of splicing channels in the Lightning Network. The first method involves merging channels once enough confirmations are received, while the second method involves creating a spent of the funding transaction which has two outputs. Rusty Russell's proposal involves creating parallel funding transactions that can be merged at an agreed depth. The architecture of c-lightning assigns a separate process to each peer, which handles only a single channel. Changing this architecture would be difficult, but there is no fundamental reason why nodes cannot create multiple channels. The proposal suggests splicing in one on-chain transaction to allow for future iterations, rather than using multiple funding outputs which adds more complexity. ZmnSCPxj suggests pre-seating inputs to generate addresses that can be spliced directly into the channel. This can be done by each channel having two public-key-derivation paths to create on-chain addresses, with the client detecting incoming funds and initiating a splice-in automatically from the UTXO paying to the address into the channel.Finally, the article discusses the benefits of creating multiple channels, including distinct acceptance policies and indirectly routed payments. The limitations of c-lightning in creating multiple channels, as well as the implementation of splicing in c-lightning, are also discussed. Rusty suggests an alternative approach to the current proposal which is based on inline splice-in and would not require the protocol at all.
