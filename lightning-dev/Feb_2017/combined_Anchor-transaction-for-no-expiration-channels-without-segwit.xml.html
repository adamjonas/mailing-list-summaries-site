Summary:

In the given context, a step-by-step protocol is described for setting up a channel between Alice and Bob. The protocol involves two transactions, TX1 and TX2, and includes safe aborts built into the system. A proposed solution to establish a symmetrically funded channel between Alice and Bob without third-party monitoring is also discussed. This solution involves using nTimelock after the bounty's expiration and creates four 1BTC outputs, essentially forming two single-funded channels. Locktime is suggested as almost useless but can solve the problem at almost no cost. However, it sets a minimum delay for a forced channel close. The proposal aims to create a secure and efficient time-locked transaction system. Additionally, the discussion mentions an anchor transaction for indefinite channel lifetimes. A proposed solution for the Bitcoin network involves confirmation of two transactions with indefinite channel lifetime or one transaction with a definite channel lifetime. It is uncertain which option would be best from a user perspective, as channel renewal can be done in the background.Two-layer HTLCs require fixes to avoid excessive timeouts. Symmetrically funded channels require four 1BTC outputs and are slightly smaller than two single funded channels. The proposal includes TX1 for initialization of the channel and TX2 to be nTimelocked after bounty's expiration. Both parties should sign both transactions before broadcasting TX1, and if TX1 is mutated, both parties should abort and spend their timeouts to recover their funds.
