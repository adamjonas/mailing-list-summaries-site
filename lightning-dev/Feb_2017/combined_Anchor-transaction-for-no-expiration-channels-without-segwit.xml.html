Summary:

The Lightning Network is a protocol built on top of the Bitcoin blockchain that aims to enable faster and cheaper transactions. One of the challenges with the Lightning Network is opening channels, which currently requires Segwit. However, there are various methods to open a channel without Segwit, including solutions involving nLocktime and CSV.To ensure the safety of setting up a channel using two transactions, TX1 and TX2, a proposed step-by-step protocol has been created. This protocol includes checks at each step to ensure that it is safe to proceed. If one party refuses to complete any step, then a safe abort is performed. However, there is a potential race condition where Alice broadcasts two transactions while Bob is broadcasting his transaction to claim TX1/0. To avoid this, an extra timeout is added with later expiration to TX1/0 so that Alice cannot broadcast TX2 until Bob has a chance to claim his output.The Lightning-dev mailing list discussed a proposal for creating a symmetrically funded channel in which Alice would take the bounty and Bob would take the escrow. If Alice is responsive, then Bob waits to receive the bounty before using the channel. TX2, the transaction involving the bounty, should be nTimelocked after the expiration of the bounty. This solution eliminates the need for a second timeout in the transaction.Once an unchanged version of TX1 is in the blockchain, all further updates of the channel can exclude the nLocktime. The locktime in TX2 is basically useless and has no effect on the overall transaction. AliceSecret2 is provided by Alice as a courtesy to allow Bob to reclaim the bounty with some confirmation, and Bob should not start using the channel before he could reclaim the bounty.
