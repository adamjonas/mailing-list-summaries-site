Summary:

The Lightning Network proposal suggests implementing a payment channel protocol with technical details for handling various scenarios. The proposal includes suggestions such as using "opener" and "accepter" instead of "funder" and "fundee" for channel establishment version 2, relaxing limits when `option_will_fund_for_food` is used, and adding a 128-bit seed to the `open_channel2` function for improved security.A new dual funding flow for Lightning developers is proposed, involving creating a new channel open protocol set (v2) with three new message types - `funding_puts2`, `commitment_signed2`, and `funding_signed2`, plus two for negotiating RBF, `init_rbf` and `accept_rbf`. Exchanging witness data allows either side to broadcast the funding transaction while maintaining information symmetry between the nodes. The proposal includes details about the various messages involved such as `accept_channel2`, `funding_puts2`, and `commitment_signed2`, and the requirements for sending and receiving nodes. The context also describes the requirements and rationale for RBF (Replace-By-Fee) for Channel Establishment v2.Once an `init_rbf` has been accepted by the dual-funding node, the message flow returns to `commitment_signed2`. A suggestion has been made on the Lightning-dev mailing list to add a 128-bit seed in `open_channel2` with sorting by SHA(seed | input | ) and SHA(seed | ). This would add further entropy to the already present temporary channel ID and various basepoints.Lisa Neigut has asked for clarification on the script mentioned in the variables scriptlen, script, max_extra_witness_len, wscriptlen, and wscript. She wants to know if the script is the scriptPubKey required for hashPrevouts in BIP143. Rusty speculates that wscript could be the BIP141 witnessScript, but this can complicate P2WPKH where an empty witnessScript means P2WPKH. Rusty confirms that non-SegWit inputs are not disallowed due to malleability problems.
