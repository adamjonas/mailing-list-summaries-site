Summary:

In a discussion between CJP and Rusty Russell about proposal on merchant forward service using rendezvous routing, CJP raised concern that Bob might be excluded from the side of the network with the more harsh regulatory conditions if it is widely known that he provides this forwarding service. Rusty agreed that this is a concern, as it allows somebody to notice such use of rendezvous routing. Rusty also mentioned that while this is allowed within the protocol and existing node software do not particularly care, it is important to check if forwarding nodes are currently cool with such weird forwarding requests. Rusty proposed a design for a "merchant forward" service using stuff they have today where Alice wants to remain anonymous even from the lightning network and Bob's node offers a forwarding service. Alice pays Bob (base + percentage?), gives a path Bob->Alice, and Bob gives Alice a short-channel-id (BobAliceSCID) and privkey to use (BobAliceSecretKey). Anything sent from Bob to this short channel id and pubkey is in fact forwarded via a new HTLC to Alice. The payer doesn't even know this arrangement exists: it looks exactly like Alice has a private channel with Bob. The minor downside is that because they conflate invoice keys (Alice needs) and onion keys (Bob needs), Bob can now issue invoices as Alice which is not very useful, but it is an argument for a separate invoice key in future.
