Summary:

The discussion on the bitcoin-dev mailing list about using SIGHASH_NOINPUT raises some questions about its potential use in Lightning Network. The concern is that malleability problems could occur with every commitment transaction when using HTLC transactions, which depend on it. It's suggested that SIGHASH_NOINPUT could potentially fix this issue, but it would require separate keys on every output to ensure transactions can't be connected to the wrong outputs. It has been argued that SIGHASH_NOINPUT is important to prevent excessive recreation and routing of punishment transaction to 3rd party monitoring services. Rusty Russell believes that SIGHASH_NOINPUT is a generally nice thing to have, but extremely dangerous if you reuse keys at all. If you reuse addresses, a signed SIGHASH_NOINPUT input can be attached to either one, which may or may not be a problem depending on exact usage.
