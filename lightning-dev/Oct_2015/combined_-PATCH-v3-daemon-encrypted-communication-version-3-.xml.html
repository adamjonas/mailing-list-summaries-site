Summary:

The Lightning Network protocol uses even and odd numbered protobuf fields to handle unknown fields in the protocol. According to Rusty Russell, odd numbered fields are optional and can be ignored by existing implementations, while even numbered fields are required and will cause an abort if received. This numbering system can also be used to deliberately break backward compatibility in future versions of the protocol. The author discusses the use of ECDH to derive a shared secret for encryption and raises concerns about the 'totlen' field, which represents the size of the unencrypted serialized protobuf message instead of the total data transmitted. Mats Jerratsch suggests that replay protection is already provided by AES-CTR and that 'totlen' may not be necessary. However, Rusty notes that some level of replay is still required for robustness when nodes crash. Additionally, Rusty discusses the process of introducing a field with the intention of making it optional or eventually compulsory, suggesting adding the field as two optional fields, labeled 11 and 12, to make the transition easier.
