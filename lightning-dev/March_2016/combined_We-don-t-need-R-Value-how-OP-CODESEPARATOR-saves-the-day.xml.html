Summary:

The Lightning-dev mailing list contains various email conversations between individuals discussing the use of signatures as a method of revocation for commitments in Bitcoin transactions. The main focus is on finding efficient ways to handle Bitcoin transactions and revocations. One such conversation involves Nicolas Dorier, who has adapted the payment channel script to use less space by modifying the escape redeem function. This resulted in 100 bytes being saved between the anchor transaction and commitment broadcasted. Links to check the scripts for Alice closing the channel and Bob using revocation are provided. Another conversation between Nicolas Dorier and Rusty discusses the usefulness of OP_CODESEPARATOR in removing duplication of bytes in a script. They propose modifying the escape redeem function from using OP_HASH160 OP_EQUAL OP_IF to using OP_CHECKSIG OP_IF, with an output of {} for SIGA-Revocation and 0 {} for the second escape where script equals to OP_ENDIF OP_CHECKSIG. The commitment revocation process involves revealing a specific key signature computed with ScriptCode. A tool called http://n.bitcoin.ninja/checkscript can be used to test the script.Joseph Poon and Nicolas Dorier discuss the possibility of using private keys for HD and whether the index of the commitment would need to be stored in the transaction. Poon proposes disclosing temporary private keys instead of signatures for compact revocable examples. He provides an example of Alice broadcasting the correct commitment and Bob not having access to PrivkeyAlice, but Alice having to wait for the CSV delay. In case Alice does not broadcast the correct commitment, Bob can take the money immediately since she disclosed her private key when creating the new commitment transaction. They also discuss the possibility of using OP_CODESEPARATOR if there is SIGHASH_NOINPUT inside segregated witness in the future, which would allow one signature to apply towards multiple types of transactions.
