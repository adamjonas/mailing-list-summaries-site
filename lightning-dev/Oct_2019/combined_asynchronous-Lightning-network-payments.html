Summary:

Intelligent assistants are expected to be available online most of the time, and this availability can be used for communication between them using standard https. Trampoline implementations could potentially implement polling behavior without a `please_poll` flag, especially if given a substantial fee and time budget. This proposed solution would facilitate asynchronous payments and outsourcing of routing to S.However, there are potential issues that need to be addressed before implementing this solution. These include privacy concerns, locked up capital, communication channel security, and proof of payment. One proposed solution involves the use of eltoo, where A pre-signs a transaction and sends it to B in an end-to-end encrypted communication channel. B can sell the signature to S using pay-for-signature enabled by payment points and scalars. A and S sign the settlement transaction, increasing the balance between them, and exchange signatures. A then sends the update transaction to B using an encrypted asynchronous communication channel. When B comes online, they decrypt the update transaction and sell the signature to S for payment. Privacy concerns can be addressed by splitting the payment between S and a public routing node, while locked-up capital can be addressed by overwriting or rebalancing the payment multiple times.Another proposed solution involves B signing a new settlement transaction and sending it to A, who acts as a watchtower while B is offline. Overall, these solutions enable truly asynchronous Lightning network payments and provide another reason to move to payment points and scalars.To ensure the effectiveness and security of the solution, proper safeguards and protocols must be in place.
