Summary:

The Lightning Network team has announced changes to the splicing protocol, allowing users to add or remove funds from a channel without closing it first. The new protocol will simplify the process by using active commitments and sending one commitment_signed message per active commitment in the order that funding transactions were created. Funding amounts exchanged in splice_init and splice_ack will be relative amounts based on each peer's current channel balance. Although implementing 0-conf splicing remains problematic due to the risks of vulnerabilities, the Lightning Network developers are interested in discussing this further. ZmnSCPxj suggests a proposal for cases where an LSP is doing splice-in while the other side is a client of that LSP. This protocol can be applied to splice-in and channel factory construction with the promise that the LSP will do its best to get the transaction confirmed before a future blockheight so that the client can rely on it being confirmed later. However, this proposal is risky for the LSP because the LSP is held liable if the TXID never confirms, leading to lost funds and a destroyed reputation.
