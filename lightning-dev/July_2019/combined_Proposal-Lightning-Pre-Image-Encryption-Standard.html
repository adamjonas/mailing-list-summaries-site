Summary:

In various conversations on the Lightning-dev mailing list, members discuss proposals and strategies to improve the security and efficiency of lightning payments for REST APIs. One proposal requires customers to trust merchants, which raises concerns about potential risks associated with using lightning payments for REST APIs, such as denial-of-service (DoS) attacks. To address these risks, members suggest different strategies, such as splitting large data into chunks and applying Forward Error Correction (FEC)-encoding to each chunk with a separate MAC to recover data in case of errors during transmission.In response to Nadav's proposal, Stepan suggests improvements concerning encrypted data and payment protocols, recommending the use of authenticated encryption with HMAC-SHA256 and proposing a modified payment protocol where only the buyer can decrypt the data. Another proposal is made for an industry standard that would allow the creation of uniform libraries for standard ln-encryption and ln-decryption using AES encryption in CFB mode with no padding to plaintext.ZmnSCPxj proposed creating guidelines for different lightning use-cases and raised concerns over proving that data provided by a data provider is bogus. The conversation also revolves around the implementation of monetized REST APIs for lightning payments, and they discuss accounting questions and privacy implications of such a scheme. A Payment-Atomic Information Decryption (PAID) API has been documented and is currently functional on testnet, but concerns have been raised about the potential for clients to perform denial-of-service attacks by repeatedly requesting data without paying.The group discusses simplifying the process of decrypting stream data, including adding a fixed sign to a generated point to simplify the decryption process and creating a Lightning Application Protocol Proposals (LAPP) repository for LAPP developers. They consider adding extra protection to a payment protocol to make sure only the buyer can decrypt the data and suggest encrypt-then-mac on chunks of data for users to authenticate that they received the intended data. Overall, the discussions aim to address potential risks and limitations while highlighting ways to improve the security and efficiency of lightning payments for REST APIs.
