Summary:

Antoine Riard, a developer on the Lightning Network, has identified two safety issues related to Lightning in an email thread. The first issue is the pre-signed feerate problem with future broadcasted time-sensitive transactions, which can result in a loss of funds. The second issue is malicious transaction pinnings, which are attacks against LN nodes. Two potential solutions to the pre-signed feerate issue for Lightning are discussed: package-relay or SIGHASH_ANYPREVOUT. While SIGHASH_ANYPREVOUT is a better long-term solution, package-relay could provide a fallback plan. However, there are concerns about attackers using pinning to force honest users to pay another anyprevout bounty to miners.The email thread also discusses the eltoo proposal, where signing the update transaction with SIGHGASH_ANYPREVOUT lets you attach non-interactively a single-party controlled input at broadcast-time. This should allow the transaction to propagate across network mempools and solve the pre-signed feerate problem as defined in the post. Package-relay is useful for CPFP fee bumping very low feerate transactions received from an external party, but it poses security concerns and requires careful consideration of upgradeability.The proposal recommends the implementation of "package-relay" in Bitcoin Core to allow for more efficient routing of Lightning Network transactions. Anti-DoS measures for package-relay would also need to be considered. The deployment timeline includes early 0.24 or 0.25 for the LN/L2 ecosystem to provide feedback on package API, followed by mempool hardening in early 0.26 or 0.27 for the whole Bitcoin ecosystem to adapt their clients.The conversation encourages open sharing and modification of ideas within the hacker ethos of the community. Dave Harding suggests that more detailed write-ups on these ideas should be published before they are promoted further. It is important to gather diverse in-field authorities' opinions or reason from scratch based on raw, public facts.
