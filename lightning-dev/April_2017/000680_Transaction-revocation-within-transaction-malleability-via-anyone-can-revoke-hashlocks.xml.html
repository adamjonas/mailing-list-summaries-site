Summary:

The need for bidirectional channels requires a primitive called revocable transactions. In order to generate a revocable output, the output is encumbered such that either the receiver can receive the output after some time or the output may be signed by both sender and receiver. However, in the presence of malleability, a fraudulent receiver may malleate the transaction producing the output, invalidating the existing revocation transaction. Another issue is that it is possible for the receiver to defraud the sender via a two-prong attack. To fix these issues, the receiver generates a secure random number, the revocation key, and publishes it to the sender and any interested participants in the Lightning Network. If the output has not been revoked, the receiver simply needs to wait 2 days to spend. If the output has been revoked, the sender can construct a revocation transaction by itself by signing the transaction and showing the published revocation key. Anyone who has heard and kept the revocation key may claim the receiver's coins after 1 day of the CSV+1 period. The receiver needs to prevent everyone to whom the revocation key has been sent from the Bitcoin network during the 1 day between the CSV+1 and CSV+2 periods to use a revoked output.
