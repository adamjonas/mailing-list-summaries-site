Summary:

The Lightning Network development team is exploring different ways to improve its performance and security. One proposed solution is the use of fast-forwards or explicit 2-of-2 for signing commitment transactions. Another proposed solution is implementing pointlocked timelocked contracts (PTLCs) to switch to payment point+scalar. The team discussed using scriptless and revocable PTLCs, with revocable outputs being necessary as per the Poon-Dryja mechanism. The team is also considering upgrading from Hash Time-Locked Contracts (HTLCs) to PTLCs, with Decker-Russell-Osuntokun being a better option than Poon-Dryja if all required features are enabled simultaneously. The implementation of Schnorr and Taproot at the base layer of Bitcoin will enable the use of payment point+scalar at the Lightning layer, making Bitcoin more efficient, secure, and private.In discussions regarding improving distinguishability of transactions, Lloyd Fournier suggested setting relative time locks on normal payments, while Dave proposed suggesting to the Bitcoin Core project to set nSequence to the block age of the UTXO being spent. ZmnSCPxj proposed a "maximally private" channel that uses only absolute locktimes, but it was deemed insecure by other members. Instead, they suggested using relative-locktime requirements for revocable outputs.The Lightning Network community is considering the implementation of PTLCs to replace HTLCs, which require all nodes along the payment path to support them. A potential issue with using MuSig is the increased number of communication rounds needed to generate signatures, but an alternative script path could reduce this. After agreeing on the specifications of the PTLC, one member creates a transaction that pays to a MuSig of two members and shares the txid and output number with the relevant MuSig output. The approach could allow existing channels to support PTLCs.Overall, the Lightning Network is exploring different options to optimize its performance while maintaining security and privacy. The community is discussing various solutions such as relative lock times, scriptless lightning, taproot, and PTLCs. However, further discussion is needed before selecting specific options to implement and evaluate in alpha software.
