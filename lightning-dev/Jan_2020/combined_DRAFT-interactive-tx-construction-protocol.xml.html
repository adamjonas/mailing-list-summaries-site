Summary:

The Lightning-dev mailing list has been discussing proposals to improve the Lightning Network protocol. One proposal suggests implementing a dual-funding protocol update to simplify Lightning Network channel funding, splicing, and closing process by breaking it into smaller chunks. The update involves the move from a single peer constructing a transaction to two participants. Additionally, developers have suggested improvements to protocol messages used in transactions. The suggestion involves adding a serial_id to inputs and outputs which can be reused for deletions and used as an ordering heuristic for transaction inputs during construction, replacing current BIP69 usage.Another proposal aims to add a second commitment requirement called PoDLE to limit the validity of a commitment only between an initiator and a single peer and prevent malicious nodes from attacking the network. JoinMarket, a bitcoin mixing service, is also discussed, which uses PoDLE to prove that the prover knows the private key corresponding to a given public key. Initiators must use odd serial IDs, while non-initiators must use even ones, and collisions between peers are considered a protocol error. Finally, the email suggests introducing block headers to address potential block height disagreements and limit gross differences in block height, with the acceptor challenging the initiator to provide proof of the block height.JoinMarket, a privacy-focused Bitcoin mixing protocol, proposes to add a second commitment requirement for Proof of Discrete Logarithm Equality (PoDLE) to limit the use of a commitment's validity between an initiator and a single peer. The proposal can be implemented by each initiator providing two commitments: one to the shared/global J point and one to a point found from the hash of the non-initiating node's node_id. A new transfer opening method was also proposed using the equation s = k + H(kG || kJ || P || P2 || utxo || receiving-node) x to prevent any node that is not the intended recipient from verifying the signature.The Lightning Network protocol outlines various rules for opening, splicing, and closing payment channels between two peers. Additionally, RBF (replace-by-fee) allows either party to initiate another round of transaction composition when any input is flagged as RBF'able. There are concerns about the possibility of funding transactions being dropped from the mempool if nLocktime is set to the tip, and suggestions for implementing anti-fee sniping and mutual closing transactions. The proposal also includes cases such as Simple Open, Splice, and Close.The email exchange between Max and Antoine focuses on the transaction format used in Bitcoin's Lightning Network, with Antoine suggesting obfuscating or using standard values to break chain analysis heuristics. Lisa Neigut discusses the suitability of PSBT for wallet interoperability. The conversation addresses nLocktime, multi-party tx construction, and transactions format. The protocol allows for the removal of inputs and outputs, with duplicate inputs or outputs considered an error. Feerate is set by the initiator or negotiated before transaction construction.The Lightning-dev mailing list has proposed a new collaborative RBF option for fee bumping in Lightning Network transactions. The proposal suggests that each "fee_step" should add 1/4 to the initial transaction feerate, which must exceed the minimum transaction relay setting as per Rule 4 of BIP125. To ensure that the sender meets certain minimum requirements, they must set "fee_step" greater than zero and greater than any prior "fee_step." Additionally, the recipient must check that the new fee does not exceed the sender's current balance minus reserve after being applied to the splice transaction, and an error must be thrown if it exceeds. The proposal also requires at least one identical, replaceable input as the original transaction to make it easier for users to change fees on their transactions.
