Summary:

The Lightning Network protocol is a topic of discussion regarding potential vulnerabilities in its signature algorithms and message transport encryption. A new message protocol has been proposed for re-anchoring transactions, which includes a new "signing algo" concept and a new "reopen_channel" to move a channel from one signing algorithm to another. However, there are concerns about the vulnerability of Scriptless Script used for payment forwarding, as an attacker could print money freely without detection, compromising the fundamental security of the blockchain.ZmnSCPxj expresses interest in using the SS (scriptless script) and HD (hierarchical deterministic) features for AMP (atomic multipath payments). Two proposed solutions for the Lightning Network include Lindell's technique and a new cryptographic construction using only 2-party ECDSA signatures. Lindell's technique can replace regular 2-of-2 multi-sig with a single p2wkh, reducing fees for opening/closing channels and making htlc timeout+success transactions smaller. However, additional cryptographic overhead would be added, and the process of adding and settling an HTLC would require more interaction. Another solution is a new cryptographic construction using only 2-party ECDSA signatures, which offers improved privacy, security, and scalability.ZmnSCPxj expresses interest in a recent paper on Schnorr signature aggregation and hash-based signatures, but notes that supporting Schnorr signatures would require a significant effort and may complicate routing implementations. Finally, Olaoluwa Osuntokun discusses the potential of using Lindell's technique to integrate randomized payment identifiers across the route, requiring both parties to complete an initial setup and verification phase before the signing process can take place.The Lightning Network team has proposed a new cryptographic scheme to improve the contract protocol, which includes a scriptless version of adaptor signatures and contracts using only two-party ECDSA signatures. The proposed scheme would require modifications to the channel opening process, adding and settling HTLCs, and the onion payload. Additionally, a new global feature bit would be required, partitioning the network into paths that understand this new feature.The concept of scriptless scripts is vulnerable to Schor's algorithm and currently has no alternative implementation. However, there is hope that scriptless scripts may be possible under lattice-based cryptography. Mord suggests that additional protocol changes relying on the DL assumption shouldn't be a problem as long as there is some backup plan, like lattice cryptography, or other signature algorithms for lightning to move to.In the context of quantum computing, insecure channels could be closed if an attacker with a quantum computer tried to steal all live payments routed through them from their senders before they get to the recipient. Although the functional concept of scriptless scripts is impressive, there needs to be some cryptographic conservatism before building on it to avoid creating a house of cards.The email exchange also discusses the potential vulnerabilities of cryptographic algorithms in the face of quantum computing, highlighting the importance of planning for their eventual failure. The need for redundancy and the salience of certain primitives' finite lifespan is emphasized. Ultimately, the global economy deserves careful consideration when it comes to cryptography, and backups should be identified at least at a conceptual level before use.
