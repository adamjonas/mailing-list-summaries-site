Summary:

The context covers various discussions and proposals related to Lightning technology, Bitcoin script, and Numerifides transactions. One proposal suggests using a BOLT to allow domain operators to create SRV records for their nodes, which would improve user experience and increase decentralization of the network. However, concerns were raised about selective attachment to a merchant's node and creating a network of large hubs that are only weakly interconnected. Tyler and Robert Olsson have proposed an additional BOLT to allow operators of domain names to create SRV records for their nodes, which would make it easier for end-users to connect to nodes with enough liquidity to pay every merchant. Additionally, there are conversations discussing the appropriate opcode to be used for a relative time lock in Bitcoin script and the use of blockchain for verification and gossip overlay for censorship resistance. They discuss the benefits of reducing the blockchain footprint and ways to communicate bindings and avoid being censored.In an email conversation, ZmnSCPxj clarified that the proposed BOLT does not replace bootstrapping seed functionality. They explained the risk associated with public and private nodes owned by the same entity and suggested using a public node directly instead of running a public node and one or more private nodes as it reduces the risk. The email also discussed the use of external parties for Bitcoin transactions while maintaining privacy.Tyler is working on finding a secure way to advertise human-understandable node names that retains the open and decentralized properties of Lightning technology and blockchain. He suggests two favorite approaches, including Namecoin-style registration and Web of Trust. A proposal called Numerifides has been developed, which is committed directly to the Bitcoin blockchain and solves Zooko's triangle. Numerifides can use already existing SPV infrastructure for lookups and doesn't have the squatting problems of Namecoin. In a related thread, two favorite approaches to the problem of securely advertising human-understandable node names were discussed: Namecoin-style registration and Web of Trust.The context of the conversation involves a method that brings efficiency without bloating the blockchain with non-financial data. The user acknowledges the value of having data live in the base chain for accessibility and censorship resistance, but recognizes that constraints may necessitate other avenues if the number of users filling up blocks increases. The design requires a separate method of communicating bindings and not being censored. The author suggests a similar scheme to pay-to-contract instead of putting non-financial data on-chain which will make Bitcoin-dev colleagues cry. Proof-of-work can be embedded in the "command" rather than on the Bitcoin blockchain if proof-of-work is insisted upon.
