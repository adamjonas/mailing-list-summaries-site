Summary:

The Lightning-dev mailing list has a thread discussing the eltoo protocol, which is a layer 2 scaling solution for Bitcoin's blockchain. The conversation focuses on the use of NOINPUT signatures in eltoo and how it could reintroduce third-party malleability to transactions. This concern could cause problems for individuals accepting bitcoins resulting from a non-cooperative close of an eltoo channel.Participants in the conversation evaluate the use of tags in output transactions and different scripts to enforce certain conditions in the update and settlement transactions. They also consider the storage and communication requirements needed to implement these changes. One participant suggests that output tagging does not provide a workable defense against third-party malleability via deeper-than-the-CSV-delay reorg and that requiring a non-NOINPUT signature may be necessary.Another participant proposes a script that enforces the relative timelock in the settlement branch by refusing to sign a settlement tx that doesn't have the timelock set, thereby avoiding the need for an OP_CSV. The discussion also considers the use of MuSig for Lightning Network channels and proposes a new approach that would reduce the number of required signatures, making it cheaper.The author notes that maximizing privacy may require more signatures but can also keep more information off-chain for better privacy. Overall, the goal is to create a safer and more efficient version of eltoo that will improve scalability and security on the Bitcoin network.The context also discusses the use of NOINPUT signatures in the eltoo update mechanism for Lightning Network channels and suggests that taproot should make it possible to hide this information and make cooperative closes appear like ordinary 2-of-2 spends. It also touches upon the need for NOINPUT signatures when claiming funds with in-flight HTLCs, as well as watchtower usage and different signature styles for multiple parties.
