Summary:

In various email threads on Bitcoin-dev and Lightning-dev mailing lists, members discussed different methods for enforcing timelocks in Bitcoin transactions. Options include signing with NOINPUT, NOSCRIPT, and codeseparatorpos=1; signing with codeseparatorpos=-1 and NOINPUT but committing to the script code and nSequence (for the CSV delay); or using OP_CHECKLOCKTIMEVERIFY and OP_DROP, as well as OP_CHECKDLSVERIFY and OP_CHECKDLS. Discussions also centered around the use of NOINPUT signatures in eltoo channels and the potential risks and benefits associated with their implementation.Solutions proposed include requiring NOINPUT signatures to always be accompanied by a non-NOINPUT signature, using output tagging to indicate NOINPUT capability, and dropping NOINPUT from normal taproot key path spending. Members also discussed the concept of offchain transactions and how they are affected by settlement transactions.The discussion on the bitcoin-dev mailing list regarding the eltoo paper suggests that BIP68 relative-time is sufficient for update and settlement purposes instead of using OP_CSV (BIP112). For update purposes, A and B will co-sign a muSig with nLockTime = t without committing to the scriptCode and BIP68 lock time. However, for settlement purposes, they will co-sign with an agreed BIP68 locktime and commit to the scriptCode. The update sig can be bound to any previous update tx immediately, while the settlement sig would only bind to a specific update tx after the relative locktime has passed.The eltoo paper suggests using OP_CSV which is not necessary if one party could single-handedly decide the relative locktime, which is not the case with a muSig. Finally, even a script using OP_CHECKLOCKTIMEVERIFY OP_DROP OP_CHECKSIGVERIFY OP_CHECKSIG does not require OP_CSV because the two signatures must use the same relative locktime, or the transaction is invalid. In order to bind to only that update tx, NOINPUT but committing to the script code and nSequence (for the CSV delay) requires two pubkeys, two sigs, and the taproot point reveal. The discussion also touched on privacy concerns and the workarounds available to limit the amount of historical data revealed about on-chain coin used in Lightning.
