Summary:

Ariel Lorenzo-Luaces expresses his liking for a proposal that can reduce payment failures and increase privacy through mitigating probing based network analysis. However, he is concerned that this proposal could introduce a vulnerability to a spoofed-payment attack where an adversary could create a malicious payment that's guaranteed to fail for free. This can cause intermediary nodes on the path of the payment before it fails to lose fees due to rebalancing if the rebalancing path's success is not dependent on the original payment's success. ZmnSCPxj suggests considering the rule of when rebalancing would be beneficial to the node. The node is offered a fee amount for forwarding but knows that under current channel states, it will definitely have to fail and earn zero fees. If it engages in JIT-routing, it must pay some fee for the rebalancing operation. The probability of success (success_rate), a value between 0 to 1.0, is the estimated probability that we will succeed the route after we forward it. JIT-routing should only be engaged if offered_fee_amount multiplied by success_rate minus rebalancing_fee_amount is greater than zero. The success rate can be computed statically from some node data, or the node can start with precomputed static information and augment this over time with its own experienced success_rate for all forwards.
