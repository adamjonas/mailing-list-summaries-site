Summary:

The discussion on the Lightning-dev mailing list revolves around finding a secure protocol for the atomic swapping of data and lightning payments. DLAS (Data Lightning Atomic Swap) proposes a solution that uses preimage embedment to ensure that sending and receiving data requires lightning payment at the same time. The DLAS protocol has two patterns: DLAS-down and DLAS-up, with some limitations such as restricted data size and possible multiple iterations for transfer. However, DLAS has various use cases, including hosting incentives and preventing email spam and DDoS attacks.To ensure the correctness of transferring files using lightning payments, ZK proofs are suggested as a possible method. However, generating a zk-proof of some property of the entire file may not be possible, and it might be necessary to prove that a thumbnail of a video file is extracted from a frame of the video. Moreover, ZmnSCPxj suggests using protocols that involve sending an encrypted version of the entire file instead of considering blocks of a file, which eliminates the need for dividing files into chunks and provides a single payment method. Matt Corallo warns against sending data in Lightning payments unless necessary, as it can cause a DoS attack and is rude to peers. Instead, he recommends using Zero Knowledge Contingent Payments (ZKCP) to lock funds to a hash preimage, which can detect if data is correct or not. However, protocols like TumbleBit and Storm that have proposed Atomic Swaps between data and cryptocurrencies lack proper academic papers on ZKCP.Overall, DLAS is a promising protocol for secure data and lightning payment exchange, although there are some limitations that need to be addressed for wider adoption. The academic papers around Bitcoin often lack relevant context, and ideas in this space are often undocumented. The email conversation between Takaya Imai and ZmnSCPxj pertains to suggestions for optimizing data transfer in various scenarios, including using OG AMP or payment points + scalars and payment decorrelation for upload cases. While ZK proofs may slow down the system, they are doable for relatively simple programs in a two-party system.
