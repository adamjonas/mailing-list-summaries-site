Summary:

The conversation is about the payment_hash txn field layout in the context of a donation example in the Lightning Network. The second line stanza differs from the others by going directly to the bes32 static int instead of using a 0x20 to indicate not a bc value or a reserved value to indicate no bc value in txn. The writer questions why this is so and suggests that it could simplify optimizing for any sort of deep inspection. The other lines use "bcvalue" to indicate a bitcoin value, but the second line, which is the donation example, uses a different format. The writer proposes a simplified format of "none_bc_value". The discussion is part of the rfc for using payment_hash fields in Lightning Network transactions.
