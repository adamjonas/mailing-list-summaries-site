Summary:

The Lightning Network's Eltoo update mechanism was discussed in several forums and mailing lists, with participants proposing ways to optimize it using taproot/tapscript. One suggestion was to use a shared private key to collapse 1-of-2 multisigs into a single-sig, while MuSig could be employed to reduce the need for `OP_CHECKMULTISIG`/`OP_CHECKSIGADD`. Additionally, there may not be a requirement for an explicit `OP_CHECKSEQUENCEVERIFY`, or separate keys for state and update paths. The chaperone signature was also debated, with suggestions made to modify the update transactions by using the proposed OP_IF/OP_ELSE structure. In this structure, Alice and Bob would exchange their anyprevoutanyscript signatures, and the chaperone signature could be provided by either party at transaction broadcast time. However, the chaperone idea might not play nice with taproot collaborative spends that require anyprevout/anyprevoutanyscript/noinput. ZmnSCPxj questioned the use of the "collaborative path" mentioned in a previous message, which refers to using the taproot-tweaked public key to sign without revealing any scripts. The bip-taproot proposal disallows all `SIGHASH` flags except for the current set of valid ones when using this path, which means it does not include `SIGHASH_NOINPUT`/`SIGHASH_ANYPREVOUT`. Participants suggested that the collaborative path is only useful unless parties are cooperatively closing anyway, since they don't need to require `SIGHASH_ANYPREVOUT` once they agree to spend the funding txo. Finally, Bastien Teinturier discussed the separation between anyprevout and anyprevoutanyscript, noting that it simplifies funding transactions and removes the need for a trigger transaction. However, integrating the chaperone signature is more complicated. Christian Decker proposed using a shared private key to collapse 1-of-2 multisigs into a single-sig, while also acknowledging that the chaperone idea might not be compatible with taproot collaborative spends that require anyprevout/anyprevoutanyscript/noinput. He concluded by noting the importance of ensuring Eltoo can support chaperone signatures, even though it is uncertain if they will eventually be needed.
