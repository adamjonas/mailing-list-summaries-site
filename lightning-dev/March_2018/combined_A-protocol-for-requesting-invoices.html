Summary:

On March 8, 2018, Corné Plooy proposed an idea for using Lightning for various types of payments on the Lightning-dev mailing list. The proposal involves having a direct line of communication between payer and payee instead of BOLT11-style interaction. Although the proposal is still in its early stages, Plooy registered Bolt #12 for his proposal. ZmnSCPxj suggested that the Web Payments Working Group of W3C may provide better perspective on various payment use cases, including Lightning invoice payments.In a conversation about privacy concerns surrounding Lightning Network transactions, it was noted that channels can be brought on-chain at any time, which potentially allows a payer to determine the neighborhood of the payee. Using non-public channels helps here, but there is unlikely a truly private channel. The conversation also touches on the idea of using TOR hidden services or other pseudonymous communication methods for added privacy.The BOLT12 protocol may not require returning to null state, but some suppliers may want to require it. The format of the "description" field is unspecified for now, and should be human-readable with an ambiguous MIME type. Different TOR addresses can be used for things you want to keep separated, and any pseudonymous communication method can be used. Depending on the transport layer used, the URL will be different. Partial onion routes can be used for payments in the opposite direction to keep the original payer anonymous.Corné Plooy's proposal aims to address some issues with Lightning Network for various types of payments, such as salary payments or direct pay-out of cryptocurrency bought on exchanges. The proposal specifies that payees give URLs to potential payers along with payment amount and timeout which allows the payer to know under what conditions the payee is likely to release the preimage. Additionally, the partial onion route can be used as a destination for payee to send invoices to the payer.The author is considering simplifying BOLT 12 by removing the possibility of refunds, which can be added back later as an optional feature. A discussion on refund payment onion routes follows, with concerns about capacity and the use of partial onion routes. Another section discusses a URL that contains an optional invoice ID for payees to request payments for specific transactions, with the option to hand over the URL without an invoice ID for one or more payers to use. The issue of how the payer derives the payment hash is also addressed.The use of a TOR hidden service and partially payee-determined routes in the Lightning Network can potentially compromise user privacy. If a user does not use a payee-determined route, they may need to reveal their node ID to the payer, which increases the risk of identifying information being leaked over time. Overall, the use of TOR and partially payee-determined routes are necessary to protect against IP addresses and node IDs being linked to users.
