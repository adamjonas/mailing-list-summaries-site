Summary:

Thomas Voegtlin has proposed a change to the BOLT-11 invoice protocol that would allow lightning service providers to bundle prepayment and main payment in the same invoice. This would involve adding two preimages and two amounts to the invoice: one for prepayment and one for the main payment. The receiver of the payment would need to wait for both payments' HTLCs to arrive before fulfilling the prepayment HTLC. If the main payment does not arrive, the prepayment would be failed with a MPP timeout. Once both payments' HTLCs have arrived, the receiver can fulfill the prepayment HTLC and broadcast their on-chain transaction.Thomas believes that implementing this change in BOLT-11 rather than using BOLT-12 or onion messages would be more feasible and efficient. However, Matt Corallo expresses skepticism about introducing new changes to BOLT-11 and suggests working towards specifying a "standard" way of doing swaps, potentially built on onion messages for more privacy. He also clarifies the distinction between splice-out and submarine swaps, with submarine swaps enabling the addition of receiving capacity to a channel.There are discussions about the feasibility of introducing these changes into BOLT-11, with suggestions for alternative solutions using BOLT-12 or onion messages. Thomas seeks clarification regarding the compatibility of bundled payments with the current BOLT-12 specification. Bastien TEINTURIER argues that extending BOLT-12 may make more sense since it requires new sender code anyway and all implementations are actively working on it.In conclusion, Thomas's proposal aims to address limitations faced by lightning service providers and improve competition by allowing invoices with bundled payments. This extension to BOLT-11 would provide a non-custodian solution and level the playing field for lightning service providers. However, there are discussions about the feasibility of introducing these changes into BOLT-11, with suggestions for alternative solutions using BOLT-12 or onion messages. Further discussions are ongoing within the Lightning community to determine the best way forward.
