Summary:

In an email exchange, Antoine Riard expresses gratitude to Matt for his work and confirms the release number v0.0.114 for the LDK "fake" lightning channels mitigations. Antoine mentions potential Denial of Service (DoS) issues in peer state machine handling and suggests using privilege watchtower and process separation as defense mechanisms. However, Antoine clarifies that these architectures are not yet implemented in the "vanilla" LDK library.Antoine provides some lessons learned regarding the mitigation of DoS issues. He emphasizes the importance of using watchtowers for revoked state punishment and "monitor replica" for second-stage HTLC. Additionally, he highlights the need for a Lightning node to be free of deadlock and other processing contaminating bugs in order to maintain liveliness even if the off-chain state machine coordinator experiences a DoS or crashes. The chain monitoring should still be able to detect revoked states and react accordingly.Antoine also shares his opinion on the prioritization of security and robustness in lightning implementations. He believes that the focus has been more on spec features parity to maintain market share, rather than considering the long-term network sustainability and safety of end-user funds. He suggests that more security auditing of the Lightning Network is needed to address these issues.The email includes a link to a GitHub issue discussing the DoS vector affecting all node implementations. This vector allows an attacker to open numerous fake channels without publishing them on-chain, causing the victim to consume excessive resources. To mitigate this risk, recent releases of various node software versions have included defenses against the DoS. The email provides a list of releases that include these defenses and urges users to update their node software to at least one of the specified versions to protect their funds. A link to more detailed information about the DoS vector is also provided.
