Summary:

The email conversation revolves around the topic of using runes/macaroons for validating and authenticating commands in a lightning node. The sender expresses skepticism about the effectiveness of this approach, stating that it does not provide protection if the machine issuing the RPCs is compromised. They argue that this method is more suitable for RPCs involving payments, such as channel open, channel close, and pay invoice, rather than "read" RPCs like listing channels.In another part of the email, the sender mentions their interest in the commando plugin and the need for additional security features. They point out that while commando uses a lightning connection to send commands to a lightning node, it lacks the handling of Bolt 8 by the Hardware Security Module (HSM) and the validation of commands using a trusted display. The sender believes that these additions would enhance security without compromising user experience and enable secure remote control from a mobile phone. They provide a link to the documentation for further information on the commando plugin.Furthermore, ZmnSCPxj contributes to the conversation by mentioning that CLN already has a similar feature in its standard distribution, which can be found at the provided link. However, it is specifically tied to the CLN command set, but shares a similar concept with the discussed topic.Bastien, on the other hand, submits a pull request to the bLIPs repository to document an idea he has been investigating. The idea involves creating a simple architecture to securely send administrative commands to a lightning node from an untrusted machine. This can be achieved by utilizing a hardware security module (HSM) acting as a whitelisted lightning peer. The proposed architecture uses a standard lightning connection as the communication channel and custom lightning messages to send commands. Notably, Bastien emphasizes that minimal changes are required on the lightning node itself, and suggests implementing the idea in a custom application running on a Ledger Nano S, which he has been exploring. Although the bLIP still requires work on the commands and encoding, Bastien considers the HSM app to be the most significant part. He offers to provide better specifications and review the implementation, and suggests that this work could potentially be done under an open-source grant.In summary, the email introduces the concept of using an HSM as a whitelisted lightning peer to securely send administrative commands to a lightning node. The sender emphasizes the minimal changes required on the lightning node and expresses their willingness to contribute to further development and implementation.
